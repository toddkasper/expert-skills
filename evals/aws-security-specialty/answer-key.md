1. **Competent:** The KMS key policy does not delegate to the account root or the Lambda role. For a customer-managed key, the key policy is the primary access control — if it does not contain `"Principal": {"AWS": "arn:aws:iam::<account-id>:root"}` (or name the role directly), IAM policies granting `kms:Decrypt` have zero effect. Fix: add a key policy statement that allows the Lambda role (or delegates to the account root so IAM policies can take effect). **Trap:** Blaming the IAM policy, the Lambda VPC config, or SCPs — the first thing to check for a KMS `AccessDenied` is always the key policy, not the IAM policy. **Verify:** In the KMS console, view the key policy and confirm a statement covers the Lambda role; re-test the Lambda invocation.

2. **Competent:** No — IAM Access Analyzer is region-scoped. An organization-level analyzer created in `us-east-1` only analyzes resources in `us-east-1` across all org accounts. Resources in the other five regions are not evaluated. You must create an analyzer in every region where resources exist, or automate creation via CloudFormation StackSets across all regions. **Trap:** Assuming an "organization-level" analyzer implies global (multi-region) coverage — "organization-level" refers to the principal scope (all accounts in the org), not geographic scope. **Verify:** Create an Access Analyzer in a second region (e.g. `eu-west-1`) and confirm it surfaces findings from that region's resources.

3. **Competent:** The engineer is wrong. SCPs are a ceiling, not a bug. An SCP `Deny` is unconditional — it overrides even `AdministratorAccess` for any principal in that OU/account. `AdministratorAccess` is an identity-policy grant; it cannot override an explicit Deny at the SCP layer. Explain the policy evaluation stack: explicit Deny (including SCPs) is evaluated before any grant. **Trap:** Agreeing that `AdministratorAccess` should bypass the SCP, or investigating the SCP as if it is malfunctioning. **Verify:** Use the IAM Policy Simulator with the SCP enabled; confirm `s3:DeleteBucket` is denied. Alternatively, run `aws organizations list-policies-for-target` and review the SCP document attached to the Prod OU.

4. **Competent:** Use a **Resource Control Policy (RCP)** applied to the organization root or the relevant OU. The RCP must include a `Deny` statement with condition `"aws:SecureTransport": "false"` on the `s3:*` (or specific S3 data-plane) actions. RCPs apply to all principals — including cross-account and external principals — acting on resources in the OU/account, regardless of individual bucket policies or identity policies. An SCP alone does not cover external principals; a bucket-policy campaign is not scalable. **Trap:** Using an SCP (which constrains IAM principals in the account but does not cover external/cross-account callers), or relying on bucket policies deployed via Config remediation (reactive, not preventive, and has a remediation lag). **Verify:** Attempt an HTTP (non-TLS) `PutObject` to any member-account bucket with a principal that has no conflicting bucket policy; confirm `AccessDenied`.

5. **Competent:** The engineer is likely wrong. Cross-account `AssumeRole` requires AND logic: both the trust policy on the target role (Account A) AND an identity-policy `sts:AssumeRole` allow in Account B must be present — and neither a permission boundary nor a session policy on Account B's role can be blocking it. Before blaming the SCP, verify the Account B role's effective permissions: check permission boundaries and session policies, which can cap below the identity-policy allow. Also confirm the trust policy uses the exact role ARN (not an account root). **Trap:** Jumping to SCP investigation before verifying the Account B role's full effective permissions (identity policy minus permission boundary minus session policy). **Verify:** Use the IAM Policy Simulator on Account B's role for `sts:AssumeRole` on the Account A role ARN; check `GetCallerIdentity` to confirm the session identity; review CloudTrail for the specific `AssumeRole` error code and message.

6. **Competent:** EC2 instance credentials are delivered via the Instance Metadata Service to the instance profile role — these are short-lived role session credentials, not a static access key attached to an IAM user. "Deactivating the access key" does nothing for a role session. The correct immediate containment action is to attach an explicit-deny IAM policy to the role (e.g. a deny-all inline policy) to revoke all active sessions using those credentials, effective within seconds. Deactivating a user access key is the right action only for IAM user key exposure, not EC2 instance role compromise. **Trap:** Treating a role session the same as an IAM user access key and declaring the deactivation sufficient. **Verify:** Check CloudTrail for continued API calls using the role session's `sts:AssumeRole` ARN after the purported deactivation; confirm they are blocked only after the deny policy is attached.

7. **Competent:** Auto-isolating on Low-severity GuardDuty findings is incorrect. Low-severity findings have a high false-positive rate and include informational detections; auto-isolation of instances based on them will cause unnecessary operational disruption. The correct threshold for automated containment is **Medium and High severity** at minimum; Low findings should route to a notification queue for human review only. **Trap:** Treating all GuardDuty severities as equally urgent and applying the same remediation automation to avoid "missing anything." **Verify:** Review GuardDuty documentation for severity-level definitions; redesign the EventBridge rule to filter on `detail.severity >= 4` (Medium) or `>= 7` (High); route Low findings to SNS for human triage only.

8. **Competent:** A VPC endpoint policy was not attached (or was left at the default allow-all). A Gateway Endpoint for S3 should have an endpoint policy that restricts access to specific S3 buckets — for example, allowing only `arn:aws:s3:::your-org-bucket-prefix*` and denying all others. Without an endpoint policy, any bucket (including attacker-controlled ones) is reachable via the endpoint. Add an endpoint policy that whitelists only the organization's own buckets. **Trap:** Concluding that the endpoint itself is insecure and should be removed, rather than adding a restrictive policy to it. **Verify:** Attempt `aws s3 ls s3://external-attacker-bucket` from the private subnet after applying the endpoint policy; confirm `AccessDenied`. Confirm access to allowed buckets still works.

9. **Competent:** The application pods are caching the old secret value. Secrets Manager rotation creates a new version (`AWSPENDING` → `AWSCURRENT`) but applications that cache the secret in memory or at startup will continue using the `AWSPREVIOUS` value until they refresh. The 30-second self-recovery is the cache TTL expiring and the app re-fetching the now-current secret. Fix: configure the application to handle the `AWSPREVIOUS` → `AWSCURRENT` transition gracefully — either by implementing retry-with-refresh logic on auth failure, or by reducing cache TTL to a value shorter than the rotation window. **Trap:** Blaming the rotation Lambda or concluding that Secrets Manager rotation is unreliable because the Lambda logs show no error. The Lambda succeeded; the problem is client-side caching behavior. **Verify:** Increase load during rotation and observe error rate; instrument the app to log which secret version it fetched; confirm errors correlate with cache TTL, not with Lambda execution time.

10. **Competent:** Use **AWS Config rules** (managed rules `cloud-trail-enabled` and `config-enabled-in-region`) combined with **SSM Automation remediation documents** set to auto-remediate on non-compliant findings. For CloudTrail, an organization trail created from the management account cannot be disabled by member-account principals even with `AdministratorAccess` if the trail is set to be an organization trail — that is a stronger preventive control. For Config, a Config conformance pack deployed via StackSets across all accounts with auto-remediation enabled will re-enable the service within the remediation interval. The enforcement is detective (Config detects the change) + automated remediation (SSM Automation re-enables it). **Trap:** Relying solely on an SCP to deny `cloudtrail:StopLogging` or `config:StopConfigurationRecorder` — while valid as a preventive layer, SCPs require explicit deny crafting and do not automatically remediate if the deny is bypassed or missing. The Config + SSM auto-remediation loop is the recovery mechanism. **Verify:** In a test account, disable Config; confirm the Config rule triggers within minutes; confirm SSM Automation re-enables it; check the CloudTrail event for the remediation action.

11. **Competent:** Configure S3 Object Lock in **Compliance mode** with a retention period of seven years. In Compliance mode, no user — including the account root user — can shorten the retention period or delete the object before the period expires. This is categorically different from **Governance mode**, which allows users with the `s3:BypassGovernanceRetention` permission (and account root) to override the lock. For a seven-year immutable audit log requirement, Compliance mode is the only S3-native option that satisfies "even root cannot delete." Also ensure versioning is enabled on the bucket, as Object Lock requires it. **Trap:** Configuring Object Lock in Governance mode and assuming it is irrevocable, or thinking S3 Versioning alone prevents deletion. **Verify:** Attempt a `DeleteObject` and `DeleteObjectVersion` on a locked object as root; confirm both are rejected with a `403` Object Lock error.

12. **Competent:** Each EC2 instance needs: (a) the SSM Agent installed and running, and (b) an instance profile (IAM role) with the `AmazonSSMManagedInstanceCore` managed policy attached. At the VPC level, since there is no internet gateway, you need a **VPC Interface Endpoint for AWS Systems Manager** — specifically endpoints for `ssm`, `ssmmessages`, and `ec2messages` — so the SSM Agent can communicate with the SSM service without internet access. All session activity through Session Manager is automatically logged to CloudTrail; optionally, stream session data to S3 or CloudWatch Logs for full content logging. No bastion host or open inbound ports are required. **Trap:** Attempting to add a bastion host or NAT Gateway to provide internet access for SSM (which undermines the private-subnet security posture), or forgetting that SSM requires three separate interface endpoints (not just one `ssm` endpoint) in a fully private VPC. **Verify:** Start a Session Manager session from the AWS console or CLI on one instance; confirm session initiation appears in CloudTrail under `ssm:StartSession`; confirm no inbound port 22 or 3389 rules exist on the instance security groups.

16. **Competent:** The colleague is wrong about the prerequisite. 17. **Competent:** The compliance officer is correct that bucket-level BPA prevents ACLs from making objects public — bucket-level BPA blocks ACLs from granting public access regardless of what individual object ACLs say. However, the additional control needed is **account-level Block Public Access**, which overrides bucket-level BPA settings and cannot be disabled by bucket owners in member accounts without account-level permissions. Without account-level BPA, a bucket owner with sufficient IAM permissions could disable bucket-level BPA and then use an ACL to make objects public. Account-level BPA is the control that removes the bucket owner's ability to undo BPA. **Trap:** Claiming that bucket-level BPA "can be overridden by object ACLs" — this is incorrect; bucket-level BPA specifically blocks ACL-based public access grants. The real gap is that bucket-level BPA can be toggled off by the bucket owner, not that ACLs bypass it. **Verify:** Attempt to add a public-access ACL to an object in a bucket with bucket-level BPA enabled — confirm it is rejected. Confirm account-level BPA is enabled via `aws s3control get-public-access-block --account-id <account>`.

18. **Competent:** The most likely explanation is that S3 object-level operations (`PutObject`, `GetObject`, etc.) are **data events**, which are **disabled by default** in CloudTrail. A trail with only management events enabled captures control-plane API calls (bucket creation, policy changes, etc.) but not data-plane operations (object reads and writes). The fix is to enable S3 data event logging on the trail: configure the trail to log `s3:PutObject` (and other needed data-plane actions) for the specific bucket ARN or all S3 buckets. Note: enabling data events for all S3 buckets generates high log volume — scope to specific buckets where possible. **Trap:** Concluding the trail is misconfigured or that the application is bypassing CloudTrail — management events on the trail do not imply data events are captured. **Verify:** Check the trail configuration: `aws cloudtrail get-event-selectors --trail-name <trail>` — if `DataResources` is empty or does not include the S3 bucket, data events are not enabled; add the data event selector and confirm `PutObject` events appear in subsequent writes.
