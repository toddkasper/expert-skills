# Answer key — salesforce-business-analyst (grading rubric)

> _Held-out eval content — original, not exam material (no real exam questions; see POLICY.md). Do not paste into a skill body._

PASS = competent move identified AND trap avoided. Partial = right instinct, misses the key rule or trap.

---

1. **Competent:** Before writing any new requirement, verify the org reality: inspect the existing NPSP Relationships object, its active Role picklist values, and whether the concept of "primary contact per cohort" is already representable with the existing data model (e.g., a Relationship record with Role = "Volunteer Lead" or a dedicated junction object). Don't write a requirement for a field or object the org already has a structural answer for. **Trap:** jumping straight to "add a Primary_Contact__c lookup field" on a new custom object — which would duplicate NPSP's existing Contact-to-Contact relationship model and create a parallel, diverging data structure. **Verify:** In Setup, open the npe5__Affiliation__c and npe4__Relationship__c objects, review their fields and active picklist values, and check whether an existing Relationship Role covers the need before scoping any new configuration.

2. **Competent:** The RACI has a defect — exactly one person must be Accountable per decision, and two Accountable owners for the same reporting-visibility decision is a red flag. The BA must escalate: log the conflict in the RAID log as an Issue, convene a decision meeting with both the VP of Programs and CFO, present the tradeoffs neutrally (access too broad vs. too narrow), and drive to a single Accountable decision that is recorded in the decision log with rationale and owner. The BA does not choose the "better" requirement. **Trap:** picking the CFO's position because they are more senior, or splitting the difference without escalation — either silently transfers the conflict risk and exposes the BA to a "we never agreed to that" dispute at go-live. **Verify:** The decision log has one entry for this reporting-visibility decision, with one named decision-maker, a date, and the rationale.

3. **Competent:** Translate the vague goal into a measurable success criterion first ("reduce data-entry time per record from ~12 minutes to under 3 minutes"), then write a properly formed story: "As a data-entry volunteer, I want a pre-populated Quick Action that captures the minimum required fields in a single screen so that I can complete an application record in under 3 minutes." Include Given/When/Then acceptance criteria that are independently testable (e.g., Given a logged-in data-entry volunteer, When they open the Quick Action on a new Application, Then all required fields are visible on one screen without scrolling and the record saves in under 3 minutes on standard hardware). The story must include the "so that" value clause to distinguish it from a task. **Trap:** writing "As a data-entry volunteer, I want a faster form" — no measurable success criterion, no testable acceptance criterion, and no "so that" clause; the team cannot estimate or validate it. **Verify:** The acceptance criteria can be phrased as a pass/fail observation (timed with a stopwatch against a representative record); if they can't, rewrite before grooming.

4. **Competent:** Flag the prioritization as arithmetically impossible and non-functional. Calculate: 110 points of Must stories across 3 sprints at 25 points/sprint = 75 points of capacity — a 35-point shortfall before any Shoulds or Coulds are considered. Present this calculation to the sponsor and facilitate a forced-ranking exercise: ask them to identify which Must stories become Shoulds if the team hits the 75-point wall. This converts an abstract "everything is critical" stance into a real, sponsor-owned deferral list before sprint planning begins. **Trap:** carrying 31 Must stories into sprint planning without escalation — when the team can't finish them all, the BA has no agreed deferral order, and the team (not the business) ends up making scope decisions in the final week. **Verify:** Sum the velocity-adjusted story points for all Must items; if they exceed available sprint capacity before go-live, document the arithmetic and present it to the sponsor before the first sprint begins.

5. **Competent:** The story and its acceptance criteria never specified API behavior or restricted-picklist enforcement — this is a **new requirement** (or an undocumented non-functional constraint), not a defect. However, the BA must also check whether the picklist field is set to "Restricted" in Salesforce: a restricted picklist silently drops non-API-list values on upsert with no error. If restricted picklist behavior is the root cause, the BA classifies the gap as a non-functional requirement that was never captured ("the picklist must accept programmatic upserts with the values listed in the form") and opens a change-request story. It does not block go/no-go unless the API path is a Must-have integration. **Trap:** logging it as a blocker defect against the existing story — that artificially inflates the defect count and misrepresents the build's quality against its actual acceptance criteria. **Verify:** In Setup, open the picklist field definition and check whether "Restrict picklist to values defined in the value set" is enabled; confirm with the integration team whether the API upsert path is in scope for the current release.

6. **Competent:** Before writing any story or requirement about automated email on Closed Won, audit all active automations on the Opportunity object in the org: open Flow Builder filtered by object = Opportunity and trigger type = Record-Triggered; check Process Builder (even if retired, it may still be active); check Workflow Rules. An existing automation may already fire an email alert on Closed Won — writing a new story without this audit risks building a duplicate that fires two emails to the same recipient. Also confirm the org's email deliverability settings and whether an Email Alert already exists in the process automation library. **Trap:** taking the requirement at face value and sizing it as a new story without the automation audit — the developer discovers a live Process Builder rule that already fires an email alert, and the team has built a duplicate. **Verify:** In Setup, navigate to Process Automation → Workflow Rules → filter by Opportunity; open Flow Builder → filter by Opportunity; open Email Alerts and confirm whether one is already configured for Closed Won.

7. **Competent:** Rewrite to the business need: "When a new Contact is created, if a Lead with the same email address already exists in Salesforce, that Lead should be converted and linked to the new Contact." Then evaluate the build path using the cost/risk ladder: can this be accomplished with a point-and-click Flow (Record-Triggered, after-insert on Contact, find Lead by email, invoke the Lead Convert action)? If the Lead Convert standard action is available in Flow in the org's release, that is a config-level solution requiring no Apex. Document the business rule; do not commit to Apex in the requirement. **Trap:** accepting the stakeholder's Apex-trigger requirement verbatim and sizing it as a development story — this locks in high-cost, governor-limit-sensitive Apex for something potentially solvable by Flow, adding test-coverage debt and deployment risk. **Verify:** In the org's Flow Builder, confirm whether a "Convert Lead" action is available as an invocable action in a Record-Triggered Flow for the current API version; check whether a standard Lead matching rule already handles this scenario.

8. **Competent:** Switch from interview/facilitated discussion to **observation (process shadowing)**. The non-verbal cues from the data-entry volunteers signal that the SOP the manager is describing diverges from what volunteers actually do at the "review handoff" step — people describe the ideal process, not their real behavior. After the session, arrange to shadow one or two volunteers during a live handoff to see what actually happens. The BA can also use a brief round-robin or write-then-share exercise to give volunteers a safe way to surface the gap without contradicting their manager in public. **Trap:** continuing to document the SOP as described by the manager and treating the volunteers' glances as irrelevant — the as-is map will reflect policy, not reality, and the to-be design will try to automate a process that doesn't actually exist. **Verify:** After shadowing, compare the observed handoff steps to the SOP; document every deviation as an as-is finding before drawing any to-be swimlane.

9. **Competent:** A wide estimation spread (1 vs. 13 points) signals that the story is not **Estimable** — it violates the E in INVEST. The ambiguity is usually because the story is too large (an epic masquerading as a story), contains unresolved technical uncertainty, or requires a spike. The BA's next action is to facilitate a story-splitting session: decompose by workflow step (list view vs. drill-down vs. filter), by user role, or by data scope, and confirm that each split story has its own "so that" value clause and testable acceptance criteria. If the team can't split without a technical spike, timebox the spike as a separate backlog item. **Trap:** averaging the estimates (e.g., 7 points) and moving on — this papers over the ambiguity, guarantees the estimate will be wrong, and leaves unresolved scope hidden in the sprint. **Verify:** After splitting, each resulting story gets its own independent estimate with team consensus (range ≤ 2 points apart); if consensus still can't be reached, schedule a spike.

10. **Competent:** Issue a **No-go**. The go/no-go rule is: any open blocker or critical defect against a Must-have scenario is an automatic no-go, regardless of sponsor pressure. The BA documents the open defect, its severity (silent data loss on a core workflow = blocker/critical), the specific Must-have scenario it fails, and the business risk of going live (submitted applications that lose attachments will require manual remediation and erode user trust). The BA presents this to the sponsor with a revised go-live timeline that includes the fix and a regression pass. **Trap:** accepting the sponsor's "fix it in week two" framing and issuing a conditional go — a Must-have scenario that silently drops data is not a cosmetic issue, and launching with a known silent data-loss defect on the primary user workflow is not a risk-based decision, it is an unacknowledged risk. **Verify:** Rerun the failed scenario in sandbox after the fix is applied; confirm the attachment persists on the created record by querying the ContentDocumentLink object, not just trusting the UI.

11. **Competent:** Updating only the requirements document breaks **traceability** — the chain from requirement → user story → test case → defect. If the user story's acceptance criteria and the UAT test case are not updated to reflect the new field list, a tester may pass a scenario that now tests against an out-of-date expected result, or a defect logged against the new fields will have no traceable acceptance criterion to reference. The BA must update all downstream artifacts: the user story (add the new fields to the acceptance criteria), the UAT test case (update the expected field list), and any existing defects linked to the story. **Trap:** treating the requirements document as the only artifact that needs updating — this creates a documentation split where the requirement says one thing and the test says another, making it impossible to audit whether the field is verified or validated. **Verify:** Walk the traceability chain: requirement ID → story ID → test case ID; confirm each artifact reflects the same field list before the next UAT cycle begins.

12. **Competent:** First, collect post-launch friction: conduct brief structured interviews or a short survey with non-logging volunteers to identify barriers (forgotten credentials, unclear login URL, confusion about which fields are required, device/browser issues). Second, convert the findings into change-request stories (e.g., a job-aid quick-reference card, a simplified login bookmark, or a mandatory required-field walkthrough). On the metric side, establish a baseline adoption metric upfront: target login rate (e.g., 90% of credentialed volunteers log in at least once per week within 30 days of go-live) and a record-creation rate (applications created in Salesforce vs. via the old paper process). Training too far in advance of go-live (two weeks is outside the recommended window) is a likely contributing factor — remedial micro-training or a drop-in session should be scheduled within one week of launch. **Trap:** concluding that UAT passing means the system is fine and the volunteers simply need more time — adoption failure after a passed UAT is almost always a change management or training-timing issue, not a system quality issue. **Verify:** Run a Salesforce login history report for the volunteer user group; compare login counts in the first 30 days to the credentialed-volunteer headcount; use the ratio as the baseline for the next 30-day target.
