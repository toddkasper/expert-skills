1. **Competent:** The colleague is correct on one point: SCPs **do not apply to the management (master) account** — ever. Attaching to the organization root would enforce the Deny on all member accounts but leave the management account unaffected (not a risk from over-enforcement there, but a governance gap). The right attachment target is the **root OU or a dedicated "Security" OU containing all member accounts**, with clear documentation that the management account is outside SCP scope. However, operations in the management account should be separately protected (e.g., by limiting who has management-account access). **Trap:** assuming root attachment also constrains the management account, or conversely assuming it breaks management-account CloudTrail operations. **Verify:** `aws organizations describe-effective-policy --policy-type SERVICE_CONTROL_POLICY --target-id <member-account-id>` confirms the Deny is in effect on a member account; verify the management account is exempt by design.

2. **Competent:** A full VPC mesh using peering requires **n(n-1)/2** connections — for 12 VPCs that is **66 peering connections**, each with its own route table entries, creating significant operational overhead and hitting practical manageability limits. The deeper architectural flaw is that VPC Peering is **non-transitive**: if a 13th VPC is added it needs 12 new connections. The correct replacement is **AWS Transit Gateway**, which provides a single hub for transitive routing across all VPCs and accounts, with one attachment per VPC and centralized route tables. **Trap:** thinking non-overlapping CIDRs make the peering mesh fine, or proposing a "hub VPC" with peering (hub VPC cannot route transitively between spokes). **Verify:** `aws ec2 describe-transit-gateway-attachments` to confirm all VPCs are attached and route propagation is enabled in the TGW route table.

3. **Competent:** The feature is **Route 53 Resolver** (formerly Conditional Forwarder). You configure an **inbound endpoint** — ENIs in the VPC that on-premises resolvers can forward queries to over Direct Connect (or VPN). You also configure an **outbound endpoint** with forwarding rules so VPC resources can forward queries for on-premises domains to on-premises DNS servers. Skipping the outbound direction means VPC workloads cannot resolve on-premises hostnames — fine if only one-way resolution is needed, but most hybrid architectures require bidirectional DNS. **Trap:** thinking a single inbound endpoint covers both directions, or that Route 53 Private Hosted Zones alone solve on-premises DNS forwarding. **Verify:** `aws route53resolver list-resolver-endpoints` to confirm both endpoint types and their ENI associations; test resolution from on-premises with `nslookup db.internal.corp <inbound-endpoint-IP>`.

4. **Competent:** The proposal is technically functional but **dramatically over-engineered for the stated requirements**. RTO = 8 hours and RPO = 1 hour map cleanly to a **warm standby** pattern: a scaled-down replica environment in a secondary region with cross-region RDS read replicas (promotable at failover) and Route 53 failover routing. Multi-region active/active — with its write conflict resolution, global latency routing complexity, and DynamoDB Global Tables replication cost — is the correct pattern only when RTO approaches near-zero and the business cannot tolerate any region-level failure. The CTO's stated tolerances leave roughly 8 hours for recovery, which warm standby handles at a fraction of the cost. **Trap:** equating "business-critical" with "must be active/active," or recommending "pilot light" when RPO = 1 hour requires near-continuous data replication. **Verify:** calculate warm standby failover time (promote replica, update DNS TTL, scale up standby environment) against the 8-hour RTO; confirm it clears with margin.

5. **Competent:** The real bottleneck is **Aurora's maximum database connection limit**, not Lambda throughput. When Lambda concurrency spikes, each function instance opens its own connection to Aurora; at 2,000 concurrent executions the connection pool exhausts Aurora's max_connections (typically a few hundred to low thousands depending on instance class), causing timeouts and `too many connections` errors. Increasing reserved concurrency makes this worse. The correct fix is to introduce **Amazon RDS Proxy** between Lambda and Aurora: RDS Proxy pools and multiplexes database connections so that thousands of concurrent Lambda invocations share a much smaller pool of actual database connections. **Trap:** increasing Lambda concurrency, upsizing the Aurora instance class, or switching to Aurora Serverless without RDS Proxy (Serverless still has a connection limit). **Verify:** monitor the CloudWatch metric `DatabaseConnections` on the Aurora cluster; after adding RDS Proxy, monitor `ProxiedQueryRequests` and confirm connection count at the database stays bounded under load.

6. **Competent:** The QA engineer is correct. Blue/green with CodeDeploy terminates the blue (old) fleet once traffic shifts to green; any session state stored **in local instance memory** on the blue instances is lost for active users mid-session, causing immediate session drops. Blue/green is safe only when session state is **externalized** — moved to a shared, persistent store accessible by both fleets. The fix before running blue/green is to migrate session storage to **Amazon ElastiCache (Redis)** or another centralized session store, and update the application to read/write sessions there. **Trap:** thinking a connection-drain timeout solves the problem (drain waits for in-flight HTTP requests, not in-memory session state), or using rolling deployment as a substitute (rolling still replaces instances, still loses local session data). **Verify:** load test with active sessions across a blue/green deploy in a pre-production environment; confirm sessions persist through the traffic cutover.

7. **Competent:** Auto-remediation at scale across 300 buckets without safeguards risks silently making intentionally public buckets (e.g., static website hosting, public data sets) private and breaking production workloads. The blast-radius control is **AWS Config remediation with an execution rate limit / concurrency cap** and, more importantly, wrapping the SSM Automation document in a **Change Manager approval workflow** or at minimum enabling "Auto Remediation" with a **retry count of 1 and a maximum execution rate** to slow rollout. For destructive or broad remediations, the pattern should be: auto-create a finding → require manual or time-windowed approval → execute with a cap on simultaneous executions. **Trap:** enabling fully automated remediation with no rate cap or approval gate, treating all 300 buckets as equivalent violations. **Verify:** after remediation, confirm the specific buckets that were public intentionally still serve their content; audit SSM Automation execution history for failures.

8. **Competent:** No — permission boundaries do not work here. **IAM permission boundaries** restrict the maximum permissions an IAM principal can have within a single account; they cannot prevent account administrators from creating new IAM roles without a boundary or from creating resources directly via the console/API. Restricting EC2 instance launches to specific regions requires a **Service Control Policy (SCP)** with a `Deny` on `ec2:RunInstances` when `aws:RequestedRegion` is not in `[us-east-1, eu-west-1]`, attached to the **OU containing the workload accounts** in AWS Organizations. **Trap:** using IAM permission boundaries (can be bypassed by administrators in the account), or using IAM policies in individual accounts (same problem — a sufficiently privileged user can modify them). **Verify:** attempt to launch an EC2 instance in `ap-southeast-1` from a workload account and confirm the SCP blocks it; `aws organizations describe-effective-policy` confirms the SCP is in force on the target account.

9. **Competent:** The cause is that traffic between the EMR cluster (in a private subnet) and S3 is routing through a **NAT Gateway** rather than an S3 VPC Gateway Endpoint, and AWS charges for NAT Gateway data processing ($0.045/GB) plus potentially inter-AZ data transfer. The zero-cost fix is to create an **S3 VPC Gateway Endpoint** in the VPC and add a route to the private subnet's route table pointing the S3 prefix list to the endpoint. Gateway endpoints for S3 and DynamoDB are free (no hourly or per-GB charge) and keep traffic off the NAT Gateway entirely. **Trap:** assuming the traffic charge is S3 egress (S3-to-EC2/EMR in the same region via a Gateway endpoint is free); proposing an Interface Endpoint for S3 instead (Interface endpoints cost per AZ per hour plus per-GB — more expensive than the NAT Gateway for high-volume use). **Verify:** after creating the Gateway endpoint and updating route tables, re-run a representative EMR job and check the NAT Gateway `BytesProcessed` CloudWatch metric — it should drop to near zero for S3 traffic.

10. **Competent:** Snowball Edge is **not the right choice** for 12 TB over a reliable 100 Mbps link. At 100 Mbps sustained, 12 TB transfers in roughly 27 hours — far faster than the 1–2 week Snowball Edge roundtrip (order, ship, load, return, import). AWS DataSync or even a direct DMS full-load over the internet is faster and cheaper at this data volume and bandwidth. The DMS ongoing-replication piece is sound for maintaining CDC until cutover, but the **critical missing step** is running the **AWS Schema Conversion Tool (SCT)** first: Oracle-to-Aurora PostgreSQL is a heterogeneous migration, and DMS cannot handle schema/stored-procedure conversion — SCT must convert the schema and flag incompatible objects before DMS begins. **Trap:** defaulting to Snow family whenever a migration involves "large data" without checking the bandwidth math; starting DMS on a heterogeneous migration without SCT. **Verify:** SCT generates a migration assessment report flagging manual-conversion items; DataSync or DMS full-load progress is visible in Migration Hub.

11. **Competent:** Before downsizing, verify **P99 (99th percentile) CPU utilization and memory**, not just average CPU. Compute Optimizer and Trusted Advisor both surface average-based signals; a latency-sensitive application (e.g., a synchronous API or a real-time pricing engine) may spike to 90% CPU for short bursts even when the average sits at 8%, and downsizing based on average would cause latency SLA violations under those spikes. Also check **CloudWatch Container Insights or detailed monitoring** at 1-minute granularity to catch burst patterns that 5-minute averages smooth away. Only downsize after confirming peak CPU and memory remain safely below the new instance's capacity under worst-case load. **Trap:** acting directly on Compute Optimizer recommendations without reviewing peak/p99 metrics; checking only average CPU at 5-minute resolution. **Verify:** after rightsizing in a staging environment, run a load test reproducing peak traffic and confirm p99 latency stays within SLA.

12. **Competent:** The bottleneck is **RDS PostgreSQL's maximum connection limit**. Kinesis Data Streams triggers Lambda with one invocation per shard; if the stream has many shards (or Lambda concurrency is high), each Lambda invocation opens its own connection to RDS. PostgreSQL's `max_connections` is constrained by instance memory (e.g., a db.t3.medium supports ~170 connections); when all slots are consumed, new connection attempts time out. Upsizing the RDS instance class raises the limit modestly but does not solve the root architectural problem. The correct fix is **Amazon RDS Proxy**: it pools connections so that many concurrent Lambda invocations share a small, bounded set of actual database connections, eliminating connection exhaustion without requiring an oversized RDS instance. **Trap:** scaling up RDS instance class (treats a connection-pooling problem as a compute problem); increasing Lambda concurrency (makes the exhaustion worse); switching to Aurora Serverless without RDS Proxy. **Verify:** add CloudWatch alarm on RDS `DatabaseConnections` approaching `max_connections`; after adding RDS Proxy, confirm `DatabaseConnections` stays bounded and `ProxiedQueryRequests` increases proportionally under load.
