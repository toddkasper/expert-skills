# Answer key — salesforce-technical-architect (held-out set, 2026-06-07)

> _Held-out eval content — original, not exam material (no real exam questions; see POLICY.md). Do not paste into a skill body._

PASS = competent move identified AND trap avoided. Partial = right instinct, misses the rule/trap.

1. **Competent:** The deploy "succeeded" but it included `<fieldPermissions>` for `ApplicationType__c`, which is now a required field in the org. Salesforce does not reject the deploy outright in all cases — the required-field check behavior can vary by API version and context — but more critically, the permset XML that attempted to explicitly grant FLS on a required field may have silently corrupted or cleared the field's visibility for that permset, since required fields must be omitted from `<fieldPermissions>`. The correct response is to remove the `<fieldPermissions>` entry for `ApplicationType__c` from the permset XML (required fields are always visible/editable and must not appear there), redeploy the permset, and verify by running a SOQL query against `ApplicationType__c` as an affected user. **Trap:** Assuming the deploy is authoritative and the field just needs to be re-added to a page layout — the issue is in the permset metadata, not the layout. Also wrong: re-deploying the field itself. **Verify:** Run `SELECT ApplicationType__c FROM Grant__c LIMIT 1` as an affected user; if it succeeds, FLS is restored. Inspect the live permset via `SetupEntityAccess` to confirm no residual broken entry.

2. **Competent:** The proposal is wrong for the stated volume. Row-by-row REST upserts for 40,000 records per night will consume API call allowances at a 1:1 ratio and is an anti-pattern at this scale — the correct tool is **Bulk API 2.0**, which batches server-side, consumes a fraction of the daily API limit, and is purpose-built for multi-thousand-row loads. For near-real-time receipt confirmations, the correct pattern is **fire-and-forget async**: the webhook receiver accepts the payload, writes it to durable storage, enqueues a Queueable or publishes a Platform Event, and immediately returns a 202 to the payment processor. The Queueable/consumer then performs a Bulk-style upsert in the background. This satisfies the "near-real-time" receipt SLA without the anti-pattern of synchronous REST at scale. **Trap:** Accepting "near-real-time = synchronous REST per record" — these are independent requirements; real-time confirmation can be decoupled from the SF write mechanism. **Verify:** After implementation, confirm daily API consumption stays within license allowance; confirm exactly one SF record per submission after a simulated retry (idempotency check).

3. **Competent:** Two refresh-specific causes: (1) **The Connected App / External Client App was not re-authorized for the sandbox org.** Sandbox refresh creates a new org instance; the Consumer Key remains the same (it is a copy of production's metadata) but any prior OAuth grants are wiped. The integration user must re-authorize the app in the new sandbox — for JWT Bearer this means the integration user must exist in the sandbox, the ECA must have its Policies set (ECA → Policies → App Policies → Select Permission Sets), and the integration user's permset must be assigned. (2) **The sandbox's Certificate/Key pair was replaced.** If the sandbox was refreshed after a cert rotation in production, the sandbox may have the old cert or none; the JWT signature will not validate. Verification step for (1): query `SELECT Id FROM User WHERE Username = '<integration_user>'` in the sandbox and confirm the ECA permission assignment via `SetupEntityAccess`. Verification for (2): in the ECA or Connected App detail, confirm the certificate thumbprint matches the public key paired with the private key the Node.js job is using to sign assertions. **Trap:** Assuming the error means the Consumer Key changed — sandbox refresh preserves the app metadata including the Consumer Key; the failure is authorization scope or cert mismatch, not a key change. **Verify:** Run the full JWT smoke chain (sign assertion → POST to token endpoint → confirm 200) after correcting each item before marking resolved.

4. **Competent:** The root cause is a duplicate `relationshipName`. Salesforce requires that every lookup to the same parent object on a given child object has a unique `relationshipName`. Both lookups declare `relationshipName = "Applications"` pointing at `Contact`, so the deploy fails with a "Duplicate relationship name" error. The fix is to assign distinct, role-specific `relationshipName` values — for example, `PrimaryApplications` for the primary-applicant lookup and `CoApplications` for the co-applicant lookup. The SOQL parent traversal syntax (`Contact__r.Name`) uses the field name, not the `relationshipName`, so renaming the relationship does not break existing queries. **Trap:** Renaming both lookups' API names instead of their `relationshipName` values — the API name is distinct from the relationship name; changing the API name does not resolve the duplicate-relationship-name error. **Verify:** After updating both `relationshipName` values in the field-meta XML, redeploy and confirm success; then run a SOQL query using each relationship name to confirm child-to-parent traversal works for both.

5. **Competent:** The competent answer is to defend the **single `Application__c` object with a `Type__c` picklist discriminator**, paired with Record Types for per-type page layouts if field sets differ significantly. The architectural reason: one object keeps the Apex mapper, the integration layer, the schema-sync script, and reporting in a single namespace. Reporting, roll-ups, and approval routing all work across types without UNION queries or cross-object complexity. What you give up: a wide object (all type-specific fields exist on all records; unused fields are null) and the cognitive overhead of validating that type-specific required fields are enforced via validation rules or record type restrictions rather than schema-level required flags. The separate-object proposal trades mapper complexity for "clean schemas" — a poor trade when the application variants share 80%+ of their fields. **Trap:** Proposing three objects without naming what is lost (cross-object reporting complexity, duplicated automation, three mappers to maintain) — the board scores this as missing the trade-off. **Verify:** Before committing, enumerate all type-specific fields and the shared fields; if type-specific fields are fewer than ~20% of total fields, the single-object design wins on simplicity. Document the decision and the threshold in an ADR so it is not relitigated.

6. **Competent:** The file is stored in a single `ContentDocumentLink` pointing only to the `Application__c` record. The `ContentDocument` / `ContentVersion` model allows one file to be associated with multiple records, but each association requires its own `ContentDocumentLink` row. The fix is to insert a second `ContentDocumentLink` record with the same `ContentDocumentId` pointing to the `Contact.Id` as the `LinkedEntityId`. No new `ContentVersion` is needed — the file bytes are shared; only the junction record is added. **Trap:** Uploading a second copy of the file and creating a new `ContentVersion` — this creates duplicate storage and breaks the "single source of truth" for the document. Or embedding the file reference as a URL in a text field on the Contact. **Verify:** After inserting the second `ContentDocumentLink`, query `SELECT LinkedEntityId FROM ContentDocumentLink WHERE ContentDocumentId = '<id>'` and confirm two rows exist — one pointing to the Application, one to the Contact. Then confirm the file appears in the Contact's Files related list in the UI.

7. **Competent:** Even though the flow contains no SOQL, a Record-Triggered Flow on Contact fires synchronously within the triggering transaction. A data-loader bulk update sends records in batches of 200; each batch is a transaction. The flow fires once per record in the batch, and Salesforce accumulates all flow-interview context within the transaction. At high volume, the likely limit is **CPU time** (10,000 ms synchronous) or **heap size** (6 MB synchronous) — both accumulate across all flow interviews in a transaction. The correct fix is to switch the flow to **run in Bulk** mode if the logic is record-independent, or to convert the logic to a **batch Apex job** that processes the 8,000 records in configurable chunks outside the triggering transaction. If the field update is truly simple, a before-save flow (which runs in the same transaction but with lower overhead than after-save) may reduce limit consumption enough for moderate volumes. **Trap:** Concluding there is no problem because "the flow has no SOQL" — governor limits are transaction-wide and accumulate across all automation (triggers, flows, processes) in the transaction, regardless of which automation component does the work. **Verify:** Enable Apex debug logging for the data-loader user, retry a 200-record batch, and inspect the debug log for the `LIMIT_USAGE_FOR_NS` section to identify which limit is being exhausted.

8. **Competent:** Without the External ID flag, `ExternalSubmissionId__c` is a plain text field — Salesforce's upsert operation keyed on an External ID requires the field to have the External ID attribute set, which enables the database index that makes upsert deduplication work. Without it, every upsert call becomes an insert, creating duplicates on retry. To restore idempotency: (1) enable the External ID flag on `ExternalSubmissionId__c` (this is a metadata change, not a data change — no data is lost); (2) deduplicate existing records by identifying all `ExternalSubmissionId__c` values with more than one record and merging or deleting the extras (preserve the oldest, merge data as needed); (3) verify by re-running a test upsert with a known external ID and confirming exactly one record exists after multiple retries. **Trap:** Deleting all duplicates without first checking which record has the canonical data (e.g., the one with the approval status set) — the merge must preserve the most complete record. **Verify:** After enabling the External ID flag, SOQL `SELECT ExternalSubmissionId__c, COUNT(Id) FROM Application__c GROUP BY ExternalSubmissionId__c HAVING COUNT(Id) > 1` to enumerate remaining duplicates; confirm zero rows after cleanup. Then re-run the integration job twice with the same payload and confirm the count stays at one.

9. **Competent:** The Web Server (auth code) flow is designed for scenarios with a human user who can interactively consent. For a headless server with no user interaction, storing a refresh token is the wrong model: refresh tokens can expire, be revoked, and represent a long-lived credential that should not live in an environment variable. The correct replacement is **JWT Bearer flow**: the server signs a JWT assertion with a private RSA key, exchanges it for an access token at the token endpoint, and receives a short-lived access token — no refresh token, no interactive login, no persistent credential in an env var. The private key should live in a secrets store (e.g. AWS SSM SecureString or HashiCorp Vault), not in an environment variable, and be rotated annually. **Trap:** Proposing the Client Credentials flow — Client Credentials is valid for a machine client with its own Salesforce identity, but JWT Bearer is the standard pattern for a headless service acting as (or on behalf of) a named Salesforce user, which is the more common nonprofit integration requirement. Confirm which identity model the org requires before choosing between them. **Verify:** After switching, run the JWT smoke chain (sign → POST → confirm 200 access token → describe a protected resource → cleanup) and confirm no refresh token is issued or stored anywhere in the system.

10. **Competent:** The root cause is the NPSP managed workflow rule in the `npe01` namespace. Because NPSP's managed automation fires on Contact insert and copies `Phone` to `MobilePhone` whenever `PreferredPhone__c = "Telephone"`, any API write that sets `PreferredPhone__c` to that value (or leaves it at its managed default) will have `MobilePhone` overwritten after the insert, with no custom code involved. The fix is a **configuration change in NPSP Setup**, not code: locate and deactivate (or reconfigure) the managed workflow rule in Setup → Process Automation → Workflow Rules, filtering for `npe01` namespace. Alternatively, set the integration to explicitly write `npe01__PreferredPhone__c = "Mobile"` on each Contact insert so the managed rule copies `MobilePhone` (the correct field) instead. Never attempt to edit the managed Apex or workflow — it is locked. **Trap:** Writing a custom trigger or Flow to "fix" `MobilePhone` after the managed rule fires — this creates an automation conflict (two automations fighting over the same field) and is fragile against future NPSP upgrades. The fix is always a config change in the managed package. **Verify:** After the config change, create a test Contact via the API with a landline `Phone` value and a mobile `MobilePhone` value; confirm `MobilePhone` retains the mobile value after insert. Check the field via SOQL, not the UI, to catch any async automation.

11. **Competent:** The problem is not the metadata — it is the working directory. All `sf project deploy` commands must be run from the **SFDX project root** (the directory containing `sfdx-project.json`), not the monorepo root or any parent directory. `InvalidProjectWorkspaceError` means the CLI cannot find `sfdx-project.json` in the current directory or any ancestor. The fix is to `cd` into the SFDX project root (e.g. `force-app/` or whatever the repo's SFDX root is named) and rerun the command. No field XML editing is needed or appropriate; the metadata is fine. **Trap:** Editing field XML in response to a workspace error — the error is a CLI/path issue, not a metadata issue. Making XML edits and rerunning from the wrong directory will produce the same error. A subtler trap: running the command with a `--source-dir` flag from the monorepo root, which may appear to work but can produce unexpected partial deploys if the path is wrong. **Verify:** Confirm `sfdx-project.json` exists in the directory before running the deploy command (`ls sfdx-project.json`). After a successful deploy, run a post-deploy SOQL query to confirm the field exists and is queryable by the integration user — a successful deploy does not guarantee FLS was included.

12. **Competent:** The architect made the classic audience-mismatch communication error: explaining the solution in technical implementation terms (JWT expiry, API limits) to a non-technical audience whose question was about organizational risk and cost. The correct framing translates every architecture decision into the dimensions the board cares about — cost, risk, and time-to-recover. For this audience: "The integration is designed so that no single volunteer holds a password or a secret key in their head. Credentials are stored in a managed secrets system; rotating them when a volunteer leaves takes one staff hour, not a rebuild. The batch-load design means we can process a year's worth of donations in minutes if we ever fall behind — it scales with our data, not our headcount." **Trap:** Going deeper into technical detail to "clarify" — more technical depth makes the communication worse, not better. The architect should read the room and switch registers immediately when the audience signals confusion. **Verify:** After delivering the re-framed answer, check for comprehension signals (nodding, follow-up questions about cost/timeline vs. about the technology). In a written architecture document, maintain two sections: an executive summary in cost/risk/time terms, and a technical appendix for engineers.

13. **Competent:** The disabled "New Connected App" button is not a permissions problem — the developer has the correct permission. Since Spring '26 (approximately February 2026), Salesforce blocks new Connected App creation by default on all orgs, via both the UI and the Metadata API. The correct path forward is to create an **External Client App (ECA)** instead: in Setup → search "External Client Apps" → New. The ECA is the current-generation replacement for Connected Apps for new integrations. The developer should create the ECA, upload the public certificate, configure OAuth scopes and the integration user policy (ECA → Policies tab → App Policies → select the integration user's permission set), and capture the Consumer Key now (it is available only in the UI, not via any API). If an existing Connected App is genuinely required (e.g., a managed package dependency), the org admin must file a Salesforce Support request to unblock Connected App creation. **Trap:** Assuming the greyed-out button is a permissions or profile issue and escalating to an admin to grant more permissions — the root cause is an org-level platform policy, not a user permission gap. Also wrong: attempting to create a Connected App via the Metadata API (also blocked). **Verify:** After creating the ECA, confirm the Consumer Key is captured, run the JWT token exchange, and confirm `SetupEntityAccess` returns the expected row for the permission set assignment.

14. **Competent:** For license type selection, the decision rule is to match the license to the actual access need. Field technicians who need only custom field service app access (work orders, custom objects, mobile app) do not need a full Salesforce license — a **Platform license** covers access to custom objects and apps without paying for standard CRM objects (Leads, Opportunities, Cases) the technicians do not use. If the technicians also need standard Service Cloud objects (Cases, Entitlements), then a full Salesforce or Service Cloud license is appropriate. If they are external contractors, an **Experience Cloud (Partner Community or Customer Community Plus)** license may apply — but the key constraint is whether they need role hierarchy and Apex-driven sharing, which Customer Community lacks. For mobile delivery, the three decision criteria are: use the **Salesforce Mobile App** when the technicians need standard objects and navigation with no custom branding or offline requirements; use **Mobile SDK** (iOS/Android native) when offline sync, push notifications, device-hardware integration (camera, barcode scanning, biometrics), or a fully branded app are required; use **LWC OSS + PWA** when a lightweight mobile-web experience is sufficient and avoiding app-store distribution is a priority. The wrong answer in a board session is to leave mobile as an afterthought — mobile surfaces enforce the same FLS, sharing model, and governor limits as desktop, so mobile access patterns must be designed, not assumed. **Trap:** Defaulting all field technicians to full Salesforce licenses because "that's what we use for everyone" — license cost compounds with headcount; naming the wrong license type in a board session signals weak commercial awareness. Also wrong: proposing Mobile SDK for a simple work-order checklist app that the standard Salesforce Mobile App handles without code. **Verify:** Before finalizing the license choice, enumerate the specific objects and features the technicians need and cross-reference the current Salesforce license comparison matrix (volatile — verify live). For mobile, build a prototype in the target delivery mode and confirm FLS and sharing rules behave as expected on a mobile device before committing to the architecture.
