# Answer key — salesforce-nonprofit-cloud-consultant (grading rubric)

PASS = competent move identified AND trap avoided. Partial = right instinct, misses the key rule or the trap. FAIL = wrong action or falls for the trap.

---

1. **Competent:** Run `list_objects` before creating any records and inspect the namespaces. "Gift Commitment" is the Nonprofit Cloud / Industries term (`GiftCommitment`); the NPSP equivalent is `npe03__Recurring_Donation__c`. If `list_objects` returns `npe03__Recurring_Donation__c` and no `GiftCommitment` object, this is an NPSP org — configure an NPSP Enhanced Recurring Donation, not a Gift Commitment. Inform the client of the distinction so scope documentation stays accurate. **Trap:** accepting the client's framing and searching for a `GiftCommitment` record, which either errors or produces an unmanaged object. **Verify:** `list_objects` returns `npe03__Recurring_Donation__c`; `GiftCommitment` is absent; `describe_object("npe03__Recurring_Donation__c")` shows `RecurringType__c` confirming ERD is active.

2. **Competent:** CRLP rollup fields are not recalculated retroactively for records loaded while TDTM handlers were active but without triggering the rollup computation path — and bulk imports loaded with handlers active at 120k volume almost certainly hit governor limits mid-transaction, leaving rollups stale or partially computed. The correct fix is to run **NPSP Settings → Bulk Data Processes → Recalculate Donor Statistics / Recalculate Rollups** after the load. (If handlers were inadvertently left active during the load itself, also verify no partial-failure orphan records exist.) **Trap:** concluding the data itself is wrong and re-importing, or waiting for "the next nightly job" to self-correct — CRLP does not auto-backfill stale rollups triggered by bulk imports. **Verify:** spot-check a handful of donor records: `soql_query("SELECT npo02__TotalOppAmount__c FROM Contact WHERE Id = '…'")` should match the sum of that Contact's primary-role Opportunity amounts after the recalculate job finishes.

3. **Competent:** Use the NPSP-native **LYBUNT** (Last Year But Unfortunately Not This year) report or the time-bound rollup fields (`npo02__OppAmountLastYear__c` > 0 and `npo02__OppAmountThisYear__c` = 0). CRLP computes these fields natively; an ad-hoc SOQL join across Opportunities is slower, harder to maintain, and misses the fiscal-year logic embedded in NPSP's rollup configuration. **Trap:** building a raw SOQL or report-type join that counts Opportunity close dates manually — this duplicates logic that CRLP already maintains and is brittle against fiscal-year settings. **Verify:** `soql_query("SELECT Id, npo02__OppAmountLastYear__c, npo02__OppAmountThisYear__c FROM Contact WHERE npo02__OppAmountLastYear__c > 0 AND npo02__OppAmountThisYear__c = 0 LIMIT 20")` returns the expected LYBUNT cohort.

4. **Competent:** NPSP's Address Management TDTM handler (`ADDR_Addresses_TDTM`) treats the `npsp__Address__c` record linked to the Household Account as the canonical source of truth and syncs it down to the Contact's Mailing fields, overwriting any direct Contact-field edit. The correct path is to edit the address on the `npsp__Address__c` record (or use Manage Household in the NPSP UI), not the raw Contact Mailing field. **Trap:** disabling Address Management to prevent future overwrites — this silently breaks Seasonal Addresses, Primary Address switching, and household-level inheritance for all Contacts in that household. **Verify:** `soql_query("SELECT npsp__MailingStreet__c, npsp__Primary__c FROM npsp__Address__c WHERE npsp__Household_Account__c = '[HH Account Id]'")` shows the canonical address; if the Contact's Mailing fields still don't match after editing the Address record, run Verify Addresses from NPSP Settings.

5. **Competent:** Use NPSP **Engagement Plan Templates**. An Engagement Plan Template defines a named sequence of tasks with day-offset and assignee configuration; applying the template to a Contact creates all the task records automatically. This is the NPSP-native equivalent of a multi-step stewardship sequence and requires no code. **Trap:** building Apex or a Flow to create the three Task records — this duplicates what Engagement Plan Templates already provide natively, adds a maintenance burden, and can conflict with NPSP's own task automation. **Verify:** `soql_query("SELECT Name, npsp__Description__c FROM npsp__Engagement_Plan_Template__c")` shows the template; verify the generated Tasks after applying the plan to a test Contact.

6. **Competent:** The report is double-counting. Hard credit (`npo02__TotalOppAmount__c`) represents the donor of record — the total dollars raised. Soft credit (`npo02__Soft_Credit_Total__c`) is a parallel recognition layer for contacts who influenced or are household members of the gift; it is not additional money. Summing both inflates the "total raised" figure by the amount of every gift that also has a soft-credit recipient, which in an NPSP Household model is nearly every gift. The correct "total raised" is the sum of hard credit only. **Trap:** accepting the report because the development officer is the business owner and "it's just a report" — incorrect totals propagate into board decks, grant reports, and major-gift segmentation. **Verify:** `soql_query("SELECT Role, IsPrimary, ContactId FROM OpportunityContactRole WHERE OpportunityId = '…'")` confirms which role carries hard credit (IsPrimary = true); cross-check one Contact's `npo02__TotalOppAmount__c` against the sum of that Contact's primary-OCR Opportunity amounts.

7. **Competent:** No, the sharing rule will not fix the problem. In Nonprofit Cloud / Industries, feature module access is gated by **Permission Set Licenses** — not OWD or sharing rules. The program officer needs the Nonprofit Cloud Program Management PSL assigned to her user account, and then the corresponding Permission Set. 8. **Competent:** NPSP's `REL_Relationships_TDTM` handler automatically creates the reciprocal direction whenever a Relationship record is saved — so creating a "Husband→Wife" record causes NPSP to generate the "Wife→Husband" reverse automatically. Hand-creating both directions results in duplicate reciprocal records (the manually created reverse plus the NPSP-auto-generated one), polluting the Relationships panel and potentially creating data-integrity issues on future edits. The second manual record should be deleted. **Trap:** leaving both records thinking "both directions are needed anyway" — the duplicates create visual noise, confuse end users, and can cause unexpected behavior if the TDTM handler runs again on the pair. **Verify:** `soql_query("SELECT npe4__Type__c, npe4__RelatedContact__c, npe4__Contact__c FROM npe4__Relationship__c WHERE npe4__Contact__c = '[Contact Id]'")` should show exactly one record per direction for a given Contact pair.

9. **Competent:** The matching strategy should use a stable, unique combination — the NPSP-recommended approach is **email + birthdate**, not name. In a Household Account model many household members share an address, and common surnames/first names create false-positive matches across unrelated records. Name alone (or name + address) collapses distinct individuals into one Contact and generates orphan duplicates for name variations (abbreviations, maiden names). Email + birthdate identifies a specific individual reliably. **Trap:** refining to name + address, reasoning that the full address makes it unique enough — Household members all share the same address, so this still collapses spouses and children. **Verify:** `soql_query("SELECT Id, Email, Birthdate, FirstName, LastName FROM Contact WHERE Email = '…' LIMIT 5")` to test match uniqueness for a representative sample before deploying the updated strategy.

10. **Competent:** OmniScript is designed for guided, branching, multi-step intake flows — not repeatable, structured measurement forms. The correct NPC-native mechanism is a **Dynamic Assessment** (structured form for repeatable measurement collection). Dynamic Assessments are purpose-built for monthly or periodic outcome check-ins and produce `Indicator Result` records tied to the client's Program Engagement. OmniScript adds licensing and maintenance overhead and should be reserved for complex guided intake with conditional branching, callouts, or FlexCard integration. **Trap:** approving the OmniScript proposal because "it's a form and OmniScript builds forms" — conflating guided intake flow with repeatable outcome measurement. **Verify:** confirm Dynamic Assessment object is available in the org (`describe_object("DynamicAssessment")` or equivalent); configure and test a sample assessment against a Program Engagement record.

11. **Competent:** CRLP rollup fields are **not recalculated automatically** after bulk imports — they update on transactional DML events in normal operation, but bulk-loaded Opportunities that were not processed through NPSP's rollup chain (whether due to disabled TDTM or batched direct API writes) leave the rollup fields stale. The team's assumption that "a nightly job will catch up" is incorrect; no automatic background job recalculates all rollups after a bulk import. The correct remediation is to manually run **NPSP Settings → Bulk Data Processes → Recalculate Donor Statistics / Recalculate Rollups**. **Trap:** waiting for the nightly scheduled job, or assuming the issue is a data loss / corruption problem requiring re-import. **Verify:** after running Recalculate Rollups, spot-check `npo02__TotalOppAmount__c` on a sample of affected Contacts against the raw sum of their primary-OCR Opportunities via SOQL.

12. **Competent:** The Data Import Wizard is the wrong tool for an NPSP org's gift-entry workflow for two reasons: (1) it has a 50,000-record cap (not the issue here at 200 records, but still wrong tool), and (2) it does not invoke NPSP's Household Account creation logic, GAU (General Accounting Unit) allocation logic, or matching rules. The correct tool is **NPSP Batch Gift Entry (GE)**, which processes gifts through `npsp__DataImportBatch__c` / `npsp__DataImport__c`, respects the NPSP data model, handles Household auto-creation, applies GAU splits, and can match to existing donors using configurable matching rules. **Trap:** using Data Loader instead — while Data Loader is appropriate for large migration loads, it also bypasses GAU allocation and NPSP Household creation logic and should be reserved for full migration loads run by a consultant, not day-to-day gift entry. **Verify:** after processing through GE, confirm the resulting Opportunity records have correct Household Account linkage and GAU Allocation records via `soql_query("SELECT npsp__GAU_Allocation__c, npsp__General_Accounting_Unit__c FROM npsp__Allocation__c WHERE npsp__Opportunity__c = '…'")`.
