1. **Competent:** `JSON.stringify` converts `Date` objects to ISO 8601 strings; it does **not** preserve the `Date` type. The downstream script receives a plain string and `getFullYear()` does not exist on strings. Fix: parse the string back with `new Date(field)` before calling date methods, or use a reviver function in `JSON.parse` to reconstruct `Date` objects: `JSON.parse(text, (k, v) => /date/i.test(k) ? new Date(v) : v)`. **Trap:** Blaming `JSON.parse` or the SOQL response format — the data loss happens silently at `JSON.stringify` time, not on read. **Verify:** `JSON.stringify(new Date()) === '"2026-06-07T..."'` — the result is a string, confirming the type is lost. Add a roundtrip unit test that asserts `typeof parsed.CreatedDate === 'object'` and `parsed.CreatedDate instanceof Date`.

2. **Competent:** `Set` is an object; mutating it in place (`.add()`) does not change the property reference on `this`. LWC's reactivity system detects only reference changes — the `Set` reference stays the same, so the template never re-renders. Fix: reassign a new `Set` after each mutation: `this.selectedIds = new Set([...this.selectedIds, id])`. **Trap:** Adding `@track` to the `Set` property expecting LWC to observe internal mutations of a `Set`. `@track` adds deep reactivity for plain objects and arrays, but its behavior with `Set`/`Map` is not guaranteed — the idiomatic, safe fix is always the new-reference reassignment. **Verify:** In an LWC Jest test, simulate a click that adds an id and assert the rendered count element updates. In a scratch org, add a `console.log(this.selectedIds)` in the getter and verify it re-evaluates after the fix.

3. **Competent:** Replace `fs.readFileSync` with the Node.js Streams API (e.g. `fs.createReadStream` piped through a CSV parser like `csv-parse` in streaming mode). Process each row as it arrives rather than buffering the entire file. This caps memory at one chunk at a time regardless of file size. **Trap:** Switching to the async `fs.promises.readFile` — it still loads the entire file into memory as a `Buffer`; async does not mean streaming. **Verify:** Measure Lambda memory consumption with `process.memoryUsage()` per batch at the top of each chunk handler. A 450 MB file should process with peak RSS well under the Lambda memory limit when streamed.

4. **Competent:** The reviewer is **incorrect** — `apiResponse == null` is the one idiomatic use of loose equality that is intentional and correct. The `==` operator special-cases `null`: `null == null` is `true` and `null == undefined` is `true`, so this single check guards against both. Using `===` would require `apiResponse === null || apiResponse === undefined`. This is the one exception the skill explicitly allows to the "always use `===`" rule. **Trap:** Replacing it with `if (!apiResponse)` — a falsy check — which also catches `0`, `""`, and `false`, producing incorrect behavior when those are valid non-null values. **Verify:** Assert the guard in unit tests with `null`, `undefined`, `0`, `""`, and `false` inputs to confirm only `null` and `undefined` short-circuit.

5. **Competent:** Two errors: (1) the event name `record-Updated` contains an uppercase letter — LWC custom event names must be **all lowercase, no camelCase** (hyphens are allowed as separators but the name is compared case-sensitively as lowercase). (2) The template binding `onrecord-Updated` must exactly match the event name — since the event name must be lowercase, the binding must be `onrecordupdated` or, if using a hyphenated name, `onrecord-updated`. Rename the event `recordupdated` (or `record-updated`) and update both the `CustomEvent` constructor and the template handler attribute. **Trap:** Setting `bubbles: true` and `composed: true` and assuming the event simply failed to propagate — propagation config is correct; the name mismatch is the actual failure. **Verify:** In a Jest test for the parent, dispatch the correctly-named event on the child element and assert the parent handler spy was called.

6. **Competent:** The test is missing an `await` (and is not declared `async`). The mock is set up **after** the element is appended, and there is no `await Promise.resolve()` (or equivalent flush) to let the component's async data-loading lifecycle complete before the assertion runs. Because the test is synchronous, the assertion fires before the mock resolves, and the DOM still has zero rows — yet the test passes because `querySelectorAll` returns an empty `NodeList` and... actually the test **should fail** unless the assertion is wrong too. The deeper issue: the test is not `async`, the mock is set up after mount (too late for `connectedCallback`), and there is no microtask flush (`await Promise.resolve()`) before asserting. Fix: make the test `async`, set up the mock before appending the element, and `await Promise.resolve()` after append to flush the microtask queue. **Trap:** Assuming the test is a valid signal because it is "green" — a false positive from a missing `await` means assertions run before the state is updated, so the test never actually validates the feature. **Verify:** After the fix, deliberately break the Apex mock (return `[]`) and confirm the test now fails.

7. **Competent:** Replace `Map` with `WeakMap`. A `Map` holds strong references to its keys — even after the component instance is unreachable by application code, the `Map` entry keeps it alive, preventing garbage collection. A `WeakMap` holds keys by weak reference: when the component instance has no other live references, the GC can collect it and the `WeakMap` entry is automatically removed. **Trap:** Periodically calling `map.delete(instance)` manually to "clean up" — this requires the service to know exactly when instances are done, which is error-prone; the entire point of `WeakMap` is that cleanup is automatic. Also: `WeakMap` cannot be iterated or measured (no `.size`, no `forEach`) — if iteration is needed, `Map` is required and a different lifecycle hook (explicit teardown) must manage cleanup. **Verify:** In a Node.js heap snapshot (using `--inspect` and Chrome DevTools), confirm that after dereferencing all component instances the metadata objects are no longer in the heap.

8. **Competent:** Spreading a `Set` into `{ ...setA, ...setB }` produces a plain object, not a `Set` — plain objects are not iterable by default (they have no `[Symbol.iterator]`). The `for…of` loop throws because it cannot find the iterator protocol. The correct deduplicated, iterable result is a new `Set` from the union: `const unique = new Set([...setA, ...setB])`, then `for (const id of unique) { … }`. **Trap:** Trying to fix it with `Object.keys(unique)` — that would return the numeric string indices of the spread object, not the original record IDs (since spread on a `Set` creates `{ '0': id0, '1': id1, … }`). **Verify:** Unit-test with two overlapping sets and assert the union size equals the count of distinct ids, then assert each id in the union came from one of the input sets.

9. **Competent:** Yes, the anonymous-arrow wrapper fixes the `this` problem — the arrow closes over the lexical `this` of the surrounding class context, so `this.recordId` is accessible. However, the downside is the same as the listener-cleanup scenario: the anonymous arrow has no stable reference. If the form library stores the callback for later removal (`form.offSubmit(...)`) the original reference is needed and the anonymous wrapper cannot be deregistered. The correct fix is an arrow class field (`save = () => { ... }`) — it fixes `this` and provides a stable reference for registration and removal. **Trap:** Recommending `.bind(this)` inline at the call site (`form.onSubmit(this.save.bind(this))`) — `.bind` also fixes `this` but returns a *new* function on each call, again breaking any later `offSubmit` removal unless the bound reference is stored. **Verify:** Store the callback reference and confirm `form.offSubmit(ref)` removes it; write a test that confirms the handler is not called after deregistration.

10. **Competent:** `npm install` resolves against the `package.json` version ranges and can install a newer (but range-compatible) version than what `package-lock.json` records, overwriting the lockfile. Different CI agents that run at slightly different times may install different sub-dependency versions, causing non-deterministic failures. Fix: replace `npm install` with `npm ci` in all CI steps. `npm ci` installs **exactly** the versions in `package-lock.json`, ignores range expressions, and fails if the lockfile is out of sync — guaranteeing a reproducible environment. **Trap:** Deleting `package-lock.json` and regenerating it — this would create a fresh resolution that all agents agree on temporarily, but the root cause (using `npm install` in CI) remains and drift recurs. **Verify:** Compare `package-lock.json` before and after `npm install` across two CI runs that show the failure; confirm `npm ci` produces identical node_modules trees on both agents using a checksum of the lockfile.

11. **Competent:** The bug is the classic `var`-in-closure loop-capture trap. `var i` is function-scoped (not block-scoped), so all closures share the same `i` variable. By the time any button is clicked, the loop has finished and `i` equals `buttons.length`. Every handler logs the same final value. The minimal fix is to replace `var` with `let` — `let` is block-scoped and creates a fresh binding per iteration, so each closure captures its own `i`. **Trap:** Wrapping the body in an immediately-invoked function expression (IIFE) to snapshot `i` — this was the pre-ES6 workaround and still works, but it is unnecessary and verbose; `let` is the correct modern solution and the reason `var` is banned in the skill. **Verify:** After the fix, attach `console.log` assertions in a Jest DOM test with `@testing-library/dom` buttons and confirm each button logs its own index. In the browser, click each button in order and verify ascending output.

12. **Competent:** Replace `Promise.all` with `Promise.allSettled`. `Promise.all` rejects immediately when any one Promise rejects, discarding the other results. `Promise.allSettled` always waits for all Promises and returns an array of `{ status: 'fulfilled', value }` or `{ status: 'rejected', reason }` results. The component can then process the fulfilled results normally and handle only the failed item's error separately — for example, showing partial data and a targeted error message for the permission-denied call. **Trap:** Wrapping each individual Apex call in its own `.catch(() => null)` to prevent rejection from propagating to `Promise.all` — this silences the error entirely and the component has no way to surface which call failed or why. **Verify:** In the LWC Jest test, mock the three Apex imports so one rejects and two resolve; assert that the component renders the two successful results and displays a specific error only for the failed one. Confirm no `UnhandledPromiseRejection` appears in the test output.

13. **Competent:** The colleague is describing Lightning Locker behavior, which is the legacy model. Orgs created in Winter '23 and later default to **Lightning Web Security (LWS)**, which became GA for all existing orgs in Summer '23. LWS uses JavaScript-level namespace isolation rather than DOM-level sandboxing; it is architecturally *less* restrictive than Locker — many Locker-era DOM restrictions (including constraints on canvas elements and cross-namespace DOM access) are relaxed under LWS. The component works because the org runs LWS, not because Locker's rules were bypassed. The correct framing: LWS is the current default security model; it is less DOM-restrictive than Locker; the active model is visible in Setup → Session Settings → Lightning Web Security. **Trap:** Treating "Lightning Locker" and "Lightning Web Security" as interchangeable names for the same sandbox, or assuming LWS enforces the same DOM-blocking rules as Locker. They are distinct architectures. **Verify:** In Setup → Session Settings, confirm "Use Lightning Web Security for Lightning web components and Aura components" is enabled; test the canvas manipulation in a scratch org with Locker forced on to observe the restriction that LWS relaxes.
