# Answer key — salesforce-advanced-administrator (grading rubric)

> _Held-out eval content — original, not exam material (no real exam questions; see POLICY.md). Do not paste into a skill body._

PASS = competent move identified AND trap avoided. Partial = right instinct, misses the key rule or trap.

---

1. **Competent:** Approval submission locks the record to non-approvers. An after-save record-triggered flow fires post-commit, at which point the record is already locked — any DML against it from a flow running in user context fails with `ENTITY_IS_LOCKED`. The fix is to move the field-stamp into a **before-save** record-triggered flow (keyed on the same trigger condition), which runs before the record commits and uses no DML — it modifies the in-flight record directly, so the lock is irrelevant. Alternatively, configure a final-approval action (a Flow or Field Update) inside the approval process itself, which runs in the approval process's own unlock context after the decision. **Trap:** Adding a Recall step before the flow fires — this adds operational friction and is not a root-cause fix. Also wrong: switching to an Apex trigger without proper lock-awareness (`Database.setSavepoint` does not bypass a lock). **Verify:** In a sandbox, submit a test Opportunity for approval and approve it. Confirm `Finance_Review_Date__c` is stamped and no `ENTITY_IS_LOCKED` event appears in the debug log.

2. **Competent:** In Enterprise Territory Management, territory assignment rules run against **Accounts only** — Opportunity territory assignment does **not** cascade automatically when accounts land in a territory. Opportunity territory assignment is a separate, explicit step: either run "Assign Territories to Opportunities" from the territory model, configure access rules, or trigger it via the API/scheduled job after account assignment completes. **Trap:** Assuming that because Accounts landed in the right territory, related Opportunities are automatically covered. They are not — the two assignments are independent. **Verify:** After running the Opportunity assignment step, query `UserTerritory2Association` and the Opportunity's `Territory2Id` to confirm Opportunities now carry the expected territory.

3. **Competent:** A standard permission set grants its permissions permanently for every session once assigned — there is no time-bound or session-bound scoping. The correct tool is a **session-based permission set**: assign it to the user, then configure it to activate only when the user has a verified/MFA session. The rep gets the export permission only during elevated-authentication sessions; the privilege expires when the session ends. **Trap:** Assigning a standard permission set "temporarily" and planning to revoke it manually — this is operationally unreliable and grants the permission even in low-assurance sessions. **Verify:** Log in as the rep without MFA, confirm the export is not available. Complete MFA verification, confirm the export is now available. Check `PermissionSetAssignment` with `IsActive` to confirm the session-based permset is active only during the verified session.

4. **Competent:** Field History Tracking is capped at **20 fields per object**. Attempting to enable a 21st field silently drops it rather than throwing an error — the UI accepts the save and discards the overflow fields. For the fields that fall outside the 20-field cap, the alternatives are: (a) **Event Monitoring** (add-on license) which captures field-level change events for auditing, or (b) a **before-save flow or Apex trigger** that copies old values to an audit child object on each change, effectively building a custom history table. **Trap:** Assuming all 20+ fields were tracked because the save appeared to succeed. Never rely on the UI confirmation — verify with `describe_object` on the `History` object or `soql_query SELECT Field FROM ObjectHistory` to enumerate which fields are actually tracked. **Verify:** `SELECT Field FROM AccountHistory LIMIT 200` (or the equivalent for the custom object) — only fields that appear in results are actually being tracked.

5. **Competent:** A field marked `<required>true</required>` in its metadata is always readable and editable for every user — Salesforce enforces this system-wide. Therefore, including a `<fieldPermissions>` entry for a required field in a permission set or profile XML causes a deploy failure: *"You cannot deploy to a required field."* The fix is to **omit the required field from `<fieldPermissions>` entirely** — it does not need an FLS grant because it is unconditionally accessible. **Trap:** Trying to work around the error by setting the field permission to `false` or removing just one of the two sub-elements (`readable`/`editable`). Either still fails — the element must be absent. **Verify:** Remove the `<fieldPermissions>` entry for `Contact.Notes__c` from the permset XML and redeploy; the deploy succeeds and the field remains accessible.

6. **Competent:** Cross-Filters (WITH/WITHOUT related records) filter report rows based on the existence of related records — they are not a trend/snapshot mechanism. The correct tool is a **Reporting Snapshot**: schedule a report to run on a recurring basis (e.g., end of each week) and persist its output as rows in a custom object. Those rows accumulate over time and can then be reported on to produce a historical trend chart. Key limits: Reporting Snapshots are scheduled, not real-time; the target custom object must be set up in advance; historical data only starts accumulating from when the snapshot is configured. **Trap:** Suggesting **Historical Trending** — it tracks up to 8 fields on a fixed set of standard objects (Opportunity, Forecasting, etc.) over a rolling 3-month window. It does not apply to Case trending over 6 months and isn't user-configurable for arbitrary objects. **Verify:** After configuring the Reporting Snapshot and letting it run for at least two cycles, query the target custom object and confirm rows are accumulating with the expected date and count values.

7. **Competent:** Salesforce requires **≥ 75% org-wide Apex code coverage** (not just the new class's coverage) AND **at least 1 line covered per trigger** for any Apex deploy to production. If the rest of the org's code base has low coverage — for example, legacy triggers or classes with no test coverage — the new deploy is rejected even if the new trigger is at 100%. Options to resolve without deleting code: (a) Write or improve test classes for the low-coverage legacy code and deploy those first to bring org-wide coverage above 75%. (b) Work with the development team to raise org-wide test coverage before deploying the new trigger. (c) Use a **Partial** sandbox to confirm org coverage totals before targeting production. **Trap:** Believing 100% coverage of the new trigger class is sufficient — the gate is **org-wide**, not per-class. Also wrong: using `--ignore-warnings` or `--test-level RunLocalTests` to bypass the check — the 75% gate cannot be bypassed for production deploys. **Verify:** Query `ApexCodeCoverageAggregate` via Tooling API or the Developer Console to compute org-wide coverage before attempting the production deploy.

8. **Competent:** Data Import Wizard does **not support Opportunities** as an importable object. The wizard is limited to Accounts, Contacts, Leads, Solutions, and custom objects. Attempting to import Opportunities with it is not possible — the object simply does not appear or the wizard blocks the attempt. The correct tool is **Data Loader**, which supports all objects including Opportunities and handles up to ~5M records via Bulk API 1.0 or up to ~150M via Bulk API 2.0. For an NPSP org with gift data, the NPSP Batch Data Importer (BDI) is the appropriate path because it respects Household matching logic. **Trap:** Suggesting the Data Import Wizard with a custom object workaround or a CSV mapping trick — no such workaround exists for Opportunities in the Wizard. **Verify:** Open Data Loader, confirm Opportunities appear in the object list, run the import with External ID upsert, and `soql_query COUNT() FROM Opportunity` to confirm the expected row increase.

9. **Competent:** Password policy changes are **metadata/setup changes**, not data-level events. They are recorded in the **Setup Audit Trail**, not debug logs. Debug logs capture Apex/flow transactions for a specific user's DML — they have no visibility into setup changes. Setup Audit Trail retains the last **180 days (6 months)** of setup change history and is downloadable as a CSV from Setup → Security → View Setup Audit Trail. **Trap:** Checking Login History (which records login events, not config changes) or debug logs (which record transaction activity). **Verify:** Download the Setup Audit Trail CSV and filter for `Action` values related to password policies to find the specific change and the user who made it.

10. **Competent:** Two Lookup fields on the same object pointing to the same parent object (Contact, in this case) cannot share the same `relationshipName` — even if their field API names are different. By default, Salesforce generates a `relationshipName` from the field label, which can collide. The fix is to **assign unique, role-specific `relationshipName` suffixes** to each lookup (e.g., `Primary_Contacts` and `Secondary_Contacts`). Crucially, renaming the `relationshipName` on the first field to add a suffix is **safe**: SOQL parent traversal uses the **field API name** (e.g., `Primary_Contact__r.Name`), not the `relationshipName`. The `relationshipName` is used for child-to-parent traversal in SOQL from the *parent* side; changing it only requires updating any SOQL that queries from the parent side using the old child relationship name. **Trap:** Believing that renaming the `relationshipName` breaks all existing SOQL — it only affects queries written from the parent object's perspective using the child relationship name; field-API-name-based traversal (`Primary_Contact__r`) is unaffected. **Verify:** After deploying with distinct `relationshipName` values, confirm the deploy succeeds and run `SELECT Primary_Contact__r.Name, Secondary_Contact__r.Name FROM Project__c LIMIT 1` to confirm both traversals resolve correctly.

11. **Competent:** The role hierarchy is an all-or-nothing mechanism for upward visibility when "Grant Access Using Hierarchies" is enabled — **every ancestor in the hierarchy inherits access**, not just the immediate parent. There is no native configuration to grant access to one level up but not two levels up. The only way to prevent the Director from seeing the Account while the Manager can is to **remove the Director from the role hierarchy path above the Manager** — i.e., restructure the role hierarchy so the Director is not a direct ancestor of the Manager for this sharing path. If restructuring is not acceptable, the alternative is to turn off "Grant Access Using Hierarchies" for the object and use **owner-based or criteria-based sharing rules** to explicitly share records with the Manager's role only (not the Director's). **Trap:** Trying to create a sharing rule that "blocks" the Director — sharing is additive-only; no rule can remove access already granted via the role hierarchy. Also wrong: creating a sharing rule to share with Manager while leaving the full hierarchy in place — the Director still inherits the Manager's access. **Verify:** After reconfiguring, log in as the Director and confirm the Account is not visible. Log in as the Manager and confirm it is still visible. Use the record's Sharing button to inspect sharing reasons.

13. **Competent:** The junior admin is wrong. Data Loader supports up to approximately **5 million records when using the legacy Bulk API 1.0** mode, but supports up to approximately **150 million records when using Bulk API 2.0** mode. Switching to Bulk API 2.0 in Data Loader's settings allows the 120 million record load to be handled in a single run (subject to the org's daily API quota). No file-splitting is required — the configuration change is enabling Bulk API 2.0 in the Data Loader settings panel. **Trap:** Accepting the 5M figure as an absolute ceiling — it applies only to the legacy Bulk API 1.0 path. Bulk API 2.0 removes that constraint. **Verify:** Check the Data Loader release notes or official Salesforce documentation for the current per-job and per-day limits before the load, as these numbers are `[volatile — verify live]`.

14. **Competent:** The stakeholder is using the wrong word. What happened on December 31, 2025 was **end of support**, not retirement or shut-down. The 30 existing active Workflow Rules **continue to execute exactly as before** — they still fire, still send emails, still update fields. The only thing that changed is that Salesforce no longer provides customer support or bug fixes for Workflow Rules and Process Builder. Disabling the rules immediately would break the business processes they automate. The correct guidance: leave the existing rules running, plan a deliberate migration to Flow Builder on an appropriate timeline, and test thoroughly in a sandbox before deactivating any rule. **Trap:** Equating "end of support" with "turned off" — a common misreading. Also wrong: emergency disabling of automation without a tested replacement, which would immediately break Opportunity close workflows. **Verify:** Confirm the rules are still active via Setup → Workflow Rules and perform a test transaction to confirm the rules fire as expected.

12. **Competent:** Two likely causes: (a) **Data Loader bypasses duplicate rules by default** — duplicate rules run on individual record saves through the UI/API, but Data Loader's bulk API calls (Bulk API v1/v2) do not invoke duplicate rule evaluation. The rule fires only in synchronous API or Bulk API v2 with header-level enforcement. (b) The **Matching Rule's matching fields** may not align with the imported data — if the external data uses a different email format or the name field is populated differently than existing contacts, the matching algorithm scores below the threshold and treats the records as unique. Before re-running: verify that the import uses **REST API / standard API** (not Bulk API) if duplicate rule enforcement is required, OR enable the `DUPLICATE_RULE_HEADER` on the Bulk API call; confirm the Matching Rule's criteria fields are populated correctly in the import file; and run a test import of 10–20 rows in a sandbox and check `DuplicateRecordSet` and `DuplicateRecordItem` objects for captured duplicates. **Trap:** Assuming that because both a Matching Rule and a Duplicate Rule are active, all API imports are automatically de-duplicated — the Bulk API path routinely bypasses them without explicit configuration. **Verify:** After reconfiguring the import to use standard API or enabling the duplicate header, run a small test batch and `soql_query SELECT COUNT() FROM DuplicateRecordSet` to confirm duplicates are being captured rather than silently inserted.
