1. **Competent:** Move `SearchResults` to module top level and pass the callback as a prop. Defining a component inside another component's render body creates a new function reference on every render — React treats it as a different component type, unmounts the previous instance, and resets all its state, causing flicker and lost focus. **Trap:** leaving the definition inline but wrapping it with `useCallback` or `useMemo` (these memoize a value, not a stable component identity — React still sees a new type each render). **Verify:** confirm in React DevTools that `SearchResults` no longer unmounts on parent re-renders; check that any input state inside `SearchResults` survives parent updates.

2. **Competent:** Add a cleanup function that sets an `ignore` flag to prevent a stale `fetch` response from overwriting newer state: `let ignore = false; fetchUser(userId).then(data => { if (!ignore) setUser(data); }); return () => { ignore = true; }`. Without it, a slow response for a previous `userId` arrives after the new fetch begins and overwrites the current result — a classic race condition. **Trap:** adding a `userId` check inside the `.then()` callback using a closure over the current prop value — the closure captures the initial value and does not see updates. **Verify:** simulate slow responses (e.g., with a delayed mock); confirm rapid userId changes never show stale data.

3. **Competent:** Pass a `key` prop equal to the controlling `tab` value to the component from the parent — `<TabContent key={tab} />` — so React unmounts and remounts it fresh when the tab changes, resetting its internal state cleanly. Alternatively, if the component should not own the selection at all, make it fully controlled by accepting `selectedTab` as a prop and removing local state entirely. The `useEffect` approach is a bug: it causes an extra render cycle (state is set after the render that mirrored the prop), and more critically, the component will still briefly render with the stale state before the effect runs. **Trap:** keeping the `useEffect` but adding a guard condition (e.g., `if (selectedTab !== tab) setSelectedTab(tab)`) — this still fires a double render and is an antipattern. **Verify:** confirm the component never renders with the old tab value when the prop changes; check for any brief flash of the stale view.

4. **Competent:** Check whether the sort comparator is not actually the cause — look at what other props or context values the memoized child consumes. The most likely culprit is a prop that is an object or array literal created inline in the parent's JSX (e.g., `style={{ ... }}` or `columns={[...]}`) — these are new references on every parent render, and `React.memo` does a shallow comparison, so any new-reference prop busts the memo. 6. **Competent:** `useDeferredValue` (plus `React.memo` on the expensive list component) is the correct solution. react.dev is explicit: "Transitions can't be used to control text inputs." Wrapping the controlled input's `setFilter` in `startTransition` will not work correctly — React cannot defer a controlled input update because the input must stay in sync with its state. The right pattern is: keep the input fully controlled (`value={filter}` / `onChange` → `setFilter`), derive a deferred copy (`const deferredFilter = useDeferredValue(filter)`), and pass `deferredFilter` to a `memo`-wrapped list component. React immediately commits the urgent input update and runs an interruptible background re-render for the list. An alternative is a two-state split: one state drives the controlled input, a separate state (updated via `startTransition`) drives the list. **Tempting-but-wrong:** reaching for `useTransition` / `startTransition` directly on the controlled-input state update — this violates the react.dev constraint and will not keep the input responsive. **Verify:** after applying `useDeferredValue` + `memo`, type rapidly in the filter input; confirm the input updates at 60 fps and the list renders with a brief visual lag (stale value) during rapid typing. Use React DevTools Profiler to confirm the list render is interruptible. Sources: react.dev/reference/react/useTransition, react.dev/reference/react/useDeferredValue.

9. **Competent:** Replace `key={index}` with a stable, unique identifier for each task — typically a database ID or a UUID generated when the task is created (e.g., `key={task.id}`). When array index is used as key on a reorderable list, React matches DOM nodes by position, not identity — after a reorder, the task at index 0 is still at index 0 in the DOM even though it is now a different task, so React reuses its DOM node (including its checked state and any uncontrolled input state) for the wrong task. **Trap:** using `key={task.title}` as the stable identifier — titles may not be unique, and if they change, React unmounts and remounts unnecessarily. **Verify:** after a drag-and-drop reorder, confirm each task's checkbox state, focused state, and any input text follows the correct task item; no state leaks to an adjacent item.

12. **Competent:** Do not approve without flagging the pattern's concrete problems beyond race conditions: (a) no caching — every mount triggers a new network request even if the data was fetched moments ago elsewhere in the app; (b) no deduplication — if this component mounts multiple times simultaneously (e.g., in a list), each instance fires a separate fetch for the same data; (c) no background refresh — stale data persists until the component unmounts and remounts; (d) no built-in loading/error state normalization — the component must manage its own `isLoading`, `error`, and `data` state manually, increasing surface area for bugs. 14. **Competent:** The expensive `<CommentList>` must be wrapped in `React.memo`. `useDeferredValue` defers the *value* passed to the child, but if the child component is not memoized, React re-renders it synchronously on every parent render regardless of whether the deferred value changed. With `memo`, React skips the list's re-render when `deferredComments` has not changed (i.e., during the urgent input re-render), then runs the interruptible background re-render once the deferred value updates. **Trap:** assuming `useDeferredValue` alone is sufficient — the docs explicitly state that `useDeferredValue` only helps when the consuming component is memoized. **Verify:** after adding `React.memo`, use React DevTools Profiler to confirm `CommentList` no longer appears in the flame chart on every keystroke; confirm it re-renders only after typing pauses. Source: react.dev/reference/react/useDeferredValue.

15. **Competent:** Use the `<Activity>` component (React 19.2). Wrap the sidebar: `<Activity mode={isOpen ? 'visible' : 'hidden'}><Sidebar /></Activity>`. `<Activity mode="hidden">` keeps the subtree mounted and its state preserved, but hides it from the user and deprioritizes updates to it. The key behavioral difference from CSS `display:none` on the wrapper is that `<Activity>` is integrated with React's scheduler and the browser's rendering pipeline — hidden subtrees do not block the main thread during transitions, and React can progressively pre-render the hidden content in the background. CSS `display:none` keeps the DOM nodes present but React still reconciles and commits to them synchronously on every render, providing no scheduling benefit. **Trap:** using `{isOpen && <Sidebar />}` (conditional rendering) — this unmounts the sidebar on close and resets all state, losing scroll position and form inputs. **Verify:** toggle the sidebar open and closed; confirm scroll position and form input values persist across toggles. Source: react.dev/blog/2025/10/01/react-19-2.

16. **Competent:** In React 19+, function components accept `ref` directly as a named prop — no `forwardRef` wrapper needed. The component signature becomes `function MyInput({ ref, value, onChange })` (or destructured in a props object). The parent passes `<MyInput ref={inputRef} />` as before. `forwardRef` is no longer required for new components; existing `forwardRef` usage continues to work but is considered legacy and can be removed incrementally. **Trap:** continuing to write `React.forwardRef((props, ref) => …)` for new React 19 components — not a bug, but unnecessarily verbose and signals unfamiliarity with 19's API. A separate trap: accepting `ref` in the props object but forgetting to forward it to the underlying DOM element (e.g., `<input ref={ref} …/>`) — the ref will be set to the function component, not the DOM node, which is almost never what the caller wants. **Verify:** in the parent, confirm `inputRef.current` is the underlying DOM element after render; verify this works without `forwardRef` in the import. Source: react.dev/blog/2025/10/01/react-19-2.

17. **Competent:** The teammate is **incorrect** about `use()`. Unlike `useState`, `useEffect`, and all other hooks, `use()` can be called conditionally — inside `if` blocks, loops, or early returns — because it is not subject to the Rules of Hooks. react.dev explicitly documents this exception: "Unlike all other React Hooks, `use` can be called within loops and conditional statements." The Rules of Hooks (call order must be stable across renders) exist because React tracks hook state by call order; `use()` is implemented differently and does not rely on call-order tracking for its semantics. The developer's conditional call is valid. **Trap:** conflating `use()` with all other hooks and applying the same call-order rule — this is a common error when `use()` is first introduced to a team. **Verify:** confirm that `use(UserContext)` inside an `if` block compiles without an `eslint-plugin-react-hooks` error (the linter is aware of the `use()` exemption in v6+). Source: react.dev/reference/react/use.

18. **Competent:** The `<Activity>` component advantages over CSS `visibility: hidden` (or `display: none`): (1) React deprioritizes and does not synchronously commit updates to hidden `<Activity>` subtrees, so they do not block the main thread during transitions; (2) React can progressively pre-render hidden panels in the background, making switching instant without a mount cost; (3) state is fully preserved (scroll position, form inputs, focus). The CSS approach keeps DOM nodes present and React still reconciles them synchronously on every render, so there is no performance benefit from the scheduler's perspective — the hidden panels still consume render time. **Trade-off:** `<Activity>` is a React 19.2 feature — verify the project is on React 19.2+ before using it; in older projects, the CSS approach is the only option for state preservation without unmounting. **Trap:** assuming `React.memo` on the panel components is equivalent — `memo` prevents re-renders from parent prop changes but does not integrate with the scheduler to defer or background-render hidden content. **Verify:** open React DevTools Performance Tracks; confirm hidden `<Activity>` panels are rendered in a low-priority background lane, not on the critical render path. Source: react.dev/blog/2025/10/01/react-19-2.
