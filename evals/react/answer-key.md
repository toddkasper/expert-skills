1. **Competent:** Move `SearchResults` to module top level and pass the callback as a prop. Defining a component inside another component's render body creates a new function reference on every render — React treats it as a different component type, unmounts the previous instance, and resets all its state, causing flicker and lost focus. **Trap:** leaving the definition inline but wrapping it with `useCallback` or `useMemo` (these memoize a value, not a stable component identity — React still sees a new type each render). **Verify:** confirm in React DevTools that `SearchResults` no longer unmounts on parent re-renders; check that any input state inside `SearchResults` survives parent updates.

2. **Competent:** Add a cleanup function that sets an `ignore` flag to prevent a stale `fetch` response from overwriting newer state: `let ignore = false; fetchUser(userId).then(data => { if (!ignore) setUser(data); }); return () => { ignore = true; }`. Without it, a slow response for a previous `userId` arrives after the new fetch begins and overwrites the current result — a classic race condition. **Trap:** adding a `userId` check inside the `.then()` callback using a closure over the current prop value — the closure captures the initial value and does not see updates. **Verify:** simulate slow responses (e.g., with a delayed mock); confirm rapid userId changes never show stale data.

3. **Competent:** Pass a `key` prop equal to the controlling `tab` value to the component from the parent — `<TabContent key={tab} />` — so React unmounts and remounts it fresh when the tab changes, resetting its internal state cleanly. Alternatively, if the component should not own the selection at all, make it fully controlled by accepting `selectedTab` as a prop and removing local state entirely. The `useEffect` approach is a bug: it causes an extra render cycle (state is set after the render that mirrored the prop), and more critically, the component will still briefly render with the stale state before the effect runs. **Trap:** keeping the `useEffect` but adding a guard condition (e.g., `if (selectedTab !== tab) setSelectedTab(tab)`) — this still fires a double render and is an antipattern. **Verify:** confirm the component never renders with the old tab value when the prop changes; check for any brief flash of the stale view.

4. **Competent:** Check whether the sort comparator is not actually the cause — look at what other props or context values the memoized child consumes. The most likely culprit is a prop that is an object or array literal created inline in the parent's JSX (e.g., `style={{ ... }}` or `columns={[...]}`) — these are new references on every parent render, and `React.memo` does a shallow comparison, so any new-reference prop busts the memo. Move object/array literals to a constant outside the component or wrap them with `useMemo`. **Trap:** adding more `useCallback` or `useMemo` throughout without profiling to identify the specific prop causing the comparison failure. **Verify:** use React DevTools Profiler "why did this render?" to identify which prop triggered the re-render; confirm the memoized child's render count drops after the fix.

5. **Competent:** Move the `config` object inside the effect so it is created fresh each time the effect runs (currently only on mount, since deps is `[]`), or restructure to pass only the primitive values from `config` that the handler actually needs, listing those primitives in the deps array. If the effect truly must run only once, use a `useRef` to hold the latest `config` and read `ref.current` inside the handler — refs are mutable and not part of React's data flow, so the handler always reads the current value without the effect needing to re-run. **Trap:** adding `config` to the dependency array without stabilizing the object (e.g., moving its definition outside the component or wrapping it with `useMemo`) — this causes the effect to teardown and re-subscribe on every parent render, defeating the `[]` intent and potentially causing infinite loops. **Verify:** after a resize, confirm the handler uses the current `config` value; confirm the event listener is attached only once in the DevTools event listeners panel.

6. **Competent:** `useTransition` is more appropriate here. The developer controls the state update (`setFilter(query)`), so they can wrap it in `startTransition` to tell React it is non-urgent, keeping the input update urgent and immediately responsive while the expensive list render is deferred. `useDeferredValue` is appropriate when you receive a value you do not control (e.g., from a prop or context) — it defers the child's render with a stale value, but the expensive render still happens synchronously once the deferred value updates; it does not give you the same control over when work is interrupted. **Trap:** choosing `useDeferredValue` on the filter value and considering the problem solved — the input may still lag because the expensive render is not marked as interruptible in the same direct way, and it requires passing the deferred value into the child component to be effective. **Verify:** after wrapping the state update in `startTransition`, use React DevTools Profiler to confirm the expensive render is interruptible; the input field should remain responsive at 60 fps during rapid typing.

7. **Competent:** Store the trigger element reference in a `useRef` that is populated at the time the modal opens (not at unmount), and restore focus in a stable close handler that fires before unmount — for example, call `triggerRef.current?.focus()` inside the button click handler or in the modal's `onClose` callback that runs while both elements are still in the DOM. Do not rely on `useEffect` cleanup for focus restoration because the cleanup fires after the component unmounts, at which point the trigger may have been removed. **Trap:** wrapping the focus call in a `setTimeout` inside the cleanup to "wait for the DOM to settle" — this is fragile, timing-dependent, and does not address the root cause. **Verify:** open and close the modal with keyboard only; confirm focus returns to the trigger button; use a screen reader to confirm the announcement context is preserved.

8. **Competent:** Add a `<label htmlFor="email-input">` with descriptive text, and set `id="email-input"` on the `<input>` — or wrap the input inside the label element. Note the JSX attribute is `htmlFor`, not `for`. Placeholder text is not an accessible label: it disappears as soon as the user begins typing, has typically poor contrast, and is not reliably announced by all screen reader / browser combinations as a persistent field name. **Trap:** adding `aria-label="Enter your email"` to the input instead of a visible label — while this satisfies the programmatic name requirement and is better than nothing, it is a lower-quality fix because sighted users who rely on visible labels (including cognitive accessibility users) get no persistent label either. **Verify:** run the form through an automated accessibility checker (e.g., axe-core) and confirm no "missing label" violation; manually test with a screen reader to confirm the field name is announced when focused.

9. **Competent:** Replace `key={index}` with a stable, unique identifier for each task — typically a database ID or a UUID generated when the task is created (e.g., `key={task.id}`). When array index is used as key on a reorderable list, React matches DOM nodes by position, not identity — after a reorder, the task at index 0 is still at index 0 in the DOM even though it is now a different task, so React reuses its DOM node (including its checked state and any uncontrolled input state) for the wrong task. **Trap:** using `key={task.title}` as the stable identifier — titles may not be unique, and if they change, React unmounts and remounts unnecessarily. **Verify:** after a drag-and-drop reorder, confirm each task's checkbox state, focused state, and any input text follows the correct task item; no state leaks to an adjacent item.

10. **Competent:** No — custom hooks do not share state. Each component that calls `useWindowSize` gets an independent hook invocation with its own state and its own `resize` event listener, meaning three components means three listeners. To share a single listener and a single state across all consumers, lift the window-size state to a React context (with a single provider that registers one listener) and have all consumers read from that context via `useContext`. **Trap:** assuming the hook's behavior is similar to a singleton or module-level cache — hooks encapsulate logic and per-call state, not shared global state. **Verify:** open the browser DevTools Event Listeners panel on the `window` object; with three components using the naive custom hook, confirm there are three `resize` listeners; with the context-based solution, confirm there is one.

12. **Competent:** Do not approve without flagging the pattern's concrete problems beyond race conditions: (a) no caching — every mount triggers a new network request even if the data was fetched moments ago elsewhere in the app; (b) no deduplication — if this component mounts multiple times simultaneously (e.g., in a list), each instance fires a separate fetch for the same data; (c) no background refresh — stale data persists until the component unmounts and remounts; (d) no built-in loading/error state normalization — the component must manage its own `isLoading`, `error`, and `data` state manually, increasing surface area for bugs.
