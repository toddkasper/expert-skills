1. **Competent:** The trigger issues one SOQL query and one DML statement *per invocation*, but Trigger.new can contain up to 200 records per batch chunk, and 5,000 records means the trigger body executes approximately 25 times. However the real danger is that the trigger queries and updates accounts one-at-a-time within its loop even within a single invocation: if the trigger code issues a `SELECT` inside a `for (Opportunity o : Trigger.new)` loop and an `update` per Account record, each of the 200 records in a single batch chunk costs one SOQL query and one DML statement, hitting the 100-SOQL limit at chunk size 101 and the 150-DML limit at chunk size 151. The fix is to bulkify: collect all Account Ids from `Trigger.new` into a `Set<Id>` before any loop, issue one `SELECT ... WHERE Id IN :acctIds` query into a `Map<Id, Account>`, compute the date updates in-memory per Opportunity, then issue one `update accts.values()` after the loop. **Trap:** Believing "one SELECT and one update" means it is safe — the developer is counting statement types in isolation, not multiplications by `Trigger.new.size()`. A trigger that reads correctly in single-record test can silently have SOQL/DML inside a loop that only surfaces in bulk. **Verify:** Write a test that inserts 200 Opportunities in a single DML call and assert that `Limits.getQueries()` inside or after the trigger execution is 1 (not 200).

3. **Competent:** The tech lead is **partially correct but imprecise — and the safer alternative is `inherited sharing`.** The precise rule is version-dependent: for classes compiled at **API v66 or lower**, a class with no sharing keyword defaults to `without sharing` — it ignores the running user's record-level sharing entirely. For classes compiled at **API v67+ (Summer '26 and later)**, the default flips to `with sharing`, so the no-keyword class now enforces sharing by default. In *neither* case does omitting the keyword mean "inherit the caller's sharing mode." `inherited sharing` is the explicit Apex keyword that adopts the calling context's sharing mode — use it on reusable helper/service classes so they automatically enforce sharing when called from a `with sharing` context. Simply omitting the keyword is never equivalent to `inherited sharing` regardless of API version. **Trap (v66 and below):** Believing an unkeyed class inherits the caller's sharing by default — it does not; it defaults to `without sharing`, a privilege escalation risk. **Trap (v67+):** Bumping an existing no-keyword class to v67 and not realizing its sharing mode silently changed from `without sharing` to `with sharing`, which can break logic that relied on bypassing record visibility. **Verify:** Write a test where a restricted user calls the class indirectly through a `with sharing` entry class; assert that the helper class does not return records the user cannot normally see. With no keyword at v66 it will; with `inherited sharing` it will not.

4. **Competent:** The specific condition is **Queueable chaining from within an asynchronous context.** When a Queueable's `execute()` method calls `System.enqueueJob()` to chain another instance, the platform allows exactly **one** chained enqueue per asynchronous execution. The pattern works in sandbox synchronous developer testing but fails at runtime when the chain depth exceeds this limit or when the system detects a circular enqueue that would run indefinitely. Additionally, in a synchronous (non-async) context such as a test, `System.enqueueJob()` is allowed up to 50 times per transaction, but each asynchronous execute context allows only 1 child enqueue. If the pagination loop tries to enqueue more than one child in a single `execute()` invocation, it will throw `LimitException: Too many queueable jobs added to the queue: 2`. **Trap:** Believing the pattern is safe because it worked in the sandbox — sandbox sync testing does not replicate the async-context enqueue limit. The reviewer is right that this will fail in production when the Queueable runs asynchronously and attempts multiple child enqueues in one execute call. **Verify:** Run the Queueable asynchronously (not in a test's `startTest/stopTest` block) in a sandbox against a data set that requires more than one child page, and observe whether a second `enqueueJob` call inside the same `execute()` throws.

7. **Competent:** The issue is that **`Test.stopTest()` guarantees synchronous execution of `@future` methods but does NOT guarantee synchronous execution of Queueable jobs in all runtimes** — actually, in current Salesforce implementations `stopTest()` does flush Queueable jobs enqueued during `startTest()/stopTest()`, but the common failure is that the developer is asserting *immediately* on database state without understanding that `stopTest()` must be called *before* the assertions, not after. If `stopTest()` is not called before the assertions, async work has not yet run. The correct pattern is: (1) call `Test.startTest()`, (2) perform the DML that fires the trigger, (3) call `Test.stopTest()` — this forces the enqueued Queueable to execute synchronously, (4) then assert on results. If `stopTest()` is called after the assertions, the Queueable never ran before the asserts. **Trap:** Believing `stopTest()` is a no-op or that enqueued jobs run "in the background" during the test — they don't; they are held until `stopTest()` flushes them. Or wrapping only part of the operation and asserting before `stopTest()`. **Verify:** Move all assertions *after* `Test.stopTest()`, not before it; confirm the assert values reflect the Queueable's work.

8. **Competent:** The enforced ceiling on the Batch scope parameter is **2,000 records per chunk** (`Database.executeBatch(job, 2000)` is the maximum). If you pass a value greater than 2,000, the platform throws `System.AsyncException: Batch Apex scope must be between 1 and 2000` at the point of the `executeBatch` call — the job does not start. In the scenario, `Database.executeBatch(new MyBatch(), 2000)` is exactly at the limit and will succeed, processing in two chunks of 1,500 + 1,500 (or similar split). The developer's assumption about finishing in exactly two chunks is also imprecise — the actual chunk distribution depends on the QueryLocator result set size, not a floor division by scope. **Trap:** Believing you can pass a larger scope to finish faster — the 2,000 limit is a hard platform ceiling and the call throws immediately, not at execution time. Or confusing batch chunk size with the total rows limit (50,000 rows per SOQL call, but a QueryLocator-backed batch can process millions over many chunks, each chunk getting its own fresh governor limits). **Verify:** In a sandbox, call `Database.executeBatch(new MyBatch(), 2001)` in Execute Anonymous and confirm the `AsyncException` is thrown immediately; then confirm `Database.executeBatch(new MyBatch(), 2000)` submits the job successfully.

9. **Competent:** At deploy time Salesforce **auto-generates a `relationshipName`** for the second Lookup field when none is specified — typically the field's API name with `__r` suffix applied. 10. **Competent:** **Visibility scope (`public` vs `private`) does NOT affect View State.** All instance variables that Salesforce must serialize to maintain page state between postbacks are included in the View State regardless of Java/Apex access modifier — `public`, `private`, and `protected` fields are all serialized unless explicitly excluded. The correct fix is to mark properties that do not need to survive a round-trip as **`transient`**: `public transient List<WrapperClass> rows { get; set; }`. Transient fields are excluded from the View State serialization entirely and are re-computed or re-fetched on each postback. The 170 KB View State limit is the ceiling; large collections must either be transient (re-fetched on action) or paginated. **Trap:** Changing access modifiers (`public` → `private`) to "hide" the property from View State — Salesforce serializes by instance membership, not by access level. The trap is a reasonable-sounding Java mental model that is wrong in the Visualforce runtime. **Verify:** Use the **View State Inspector** (enable in Setup → User Interface → Developer Preview; then available as a tab in the page footer during development) to see each variable's contribution to the 170 KB budget before and after adding `transient`.

12. **Competent:** The developer is **wrong** — the LIKE operator provides no sanitization. A user input containing a single quote, comment sequence (`--`), or additional SQL-like fragment can still close the string literal and alter the query structure. For example, the input `%' OR Name LIKE '%` would change the WHERE clause semantics entirely. The correct fix is to use **`String.escapeSingleQuotes(userInput)`** to escape any embedded single quotes before concatenation, which prevents the injected input from closing the string literal. The ideal fix is to use a **bind variable** instead: rewrite the query as `'SELECT Id, Name FROM Account WHERE Name LIKE :searchPattern'` where `String searchPattern = '%' + userInput + '%'` is a bound Apex variable — bind variables are never evaluated as SOQL syntax regardless of content. **Trap:** Believing that LIKE's wildcard behavior somehow neutralizes injected content — LIKE is a SOQL filter operator, not a sanitization layer. The single-quote injection path is identical whether the comparison operator is `=`, `LIKE`, `!=`, or any other. **Verify:** In Execute Anonymous on a sandbox (never production), pass a malicious test string with an embedded single quote through the method and confirm it throws a `QueryException` rather than executing the malformed query; then confirm the bind-variable version handles the same input correctly.
