# Answer key — salesforce-administrator (held-out set, 2026-06-07)

> _Held-out eval content — original, not exam material (no real exam questions; see POLICY.md). Do not paste into a skill body._

PASS = competent move identified AND trap avoided. Partial = right instinct, misses the rule/trap.

1. **Competent:** When a manager loses upward visibility through the role hierarchy with no apparent security change, suspect a **change to the manager's role assignment** — the user may have been moved to a different role (or removed from one), silently losing their position above the subordinates. Role hierarchy grants upward visibility; if the user is no longer the role ancestor of the reps, they lose access. Check **Setup Audit Trail** (180-day log) for role-reassignment events, then verify the role hierarchy in Setup → Roles. **Trap:** Assuming OWD changed and resetting it to Public (which would expose everyone's records). Or checking sharing rules — sharing rules only open access above the OWD floor and cannot explain a manager losing hierarchy-based access. **Verify:** `soql_query("SELECT Id, Name, UserRoleId FROM User WHERE Id = '<manager_id>'")` to confirm current role, then inspect the role hierarchy to confirm the expected parent-child relationship with subordinate users.

2. **Competent:** The root cause is that **flows process records in batches of 200**, so a nightly job over 10,000+ records will iterate many batches. If the flow's Update Records call is correctly placed after the loop but still touches too many rows, the issue is that **each batch of 200 runs as a separate transaction** — but within a transaction, all DML against child records can still accumulate. The fix is to ensure the flow does not also perform DML on related records inside the loop, and to audit the total DML row count across all elements (not just the Update). If the flow must touch many related records per Account, consider a **scheduled Apex batch job** (chunked at a configurable size) instead. **Trap:** Adding another Update element to "split" the load — the governor-limit counter is per-transaction, not per-element, so splitting elements in the same transaction changes nothing. **Verify:** Run the flow in a sandbox against a representative record set (500+ records) and inspect the Apex debug log for cumulative DML row counts.

3. **Competent:** **Freeze the user** (Setup → Users → select user → Freeze). Freezing instantly blocks all login without deactivating the account, which means the user's license remains consumed, their record ownership is unchanged, and they still appear in the system — exactly what is required during an active investigation. Deactivation is the correct action once the investigation concludes (it frees the license), but deactivation can be blocked if the user owns open approval process steps or is referenced in automation, making freeze the safe immediate action. **Trap:** Deactivating immediately — deactivation frees the license and can be rejected if the user is stuck in open approvals, creating a delay when immediate lockout is critical. Or changing the user's password (they can still log in and are not truly locked out). **Verify:** Confirm the user cannot log in by checking their session status; confirm their record ownership is intact via `soql_query("SELECT OwnerId FROM Opportunity WHERE OwnerId = '<user_id>' LIMIT 1")`.

4. **Competent:** **Login hours and login IP ranges are profile-exclusive settings** — they cannot be configured on a permission set. There is no mechanism to place time-based login restrictions on a permission set. The correct approach is to **split the users into a separate profile** that has the login-hour restriction, or to use a separate role/profile combination. If splitting profiles is not viable, consider a third-party identity provider (SSO) that enforces time-based access at the IdP layer. **Trap:** Adding the login hours to the permission set and assuming a future Salesforce update will honor them, or editing session settings thinking that applies per-user. **Verify:** Confirm the setting is absent from the permission set UI (the Login Hours section does not appear on permission sets — only on profiles). After creating the profile with restrictions, test a login outside the allowed hours.

5. **Competent:** The analyst likely created a **tabular report** — tabular reports produce flat lists with no row groupings and **cannot be used as the source for most dashboard components** (which require grouped data to aggregate). They should convert it to a **summary report** grouped by Account, which enables the grouping that dashboard charts require. Matrix reports also work but are more complex. **Trap:** Assuming the issue is a folder-permission problem (the greyed-out option is about report type, not access). Or trying to use the tabular report with a "table" dashboard component — while a tabular report can back a limited dashboard table, it cannot back charts or gauges. **Verify:** Edit the report, change type to Summary, add a Group By on Account Name, save, then confirm the dashboard "Add Component" wizard accepts it.

6. **Competent:** Two likely reasons the Alert-mode Duplicate Rule does not show a warning: (1) **the Duplicate Rule is inactive** — rules must be explicitly activated after creation; check the Active checkbox on the rule. (2) **the Matching Rule the Duplicate Rule references is inactive or not associated** — if the Matching Rule is deactivated or not saved/activated, no matching occurs and the Duplicate Rule has nothing to act on. A third possibility is that the rule's conditions (e.g. "Apply rule on: All records") are scoped incorrectly and exclude the user's profile. **Trap:** Assuming Alert mode is functioning but silently failing — the most common real cause is simply an inactive rule. Or concluding the Duplicate Rule needs to be set to Block. **Verify:** Setup → Duplicate Rules → confirm Status is Active; Setup → Matching Rules → confirm the referenced rule is Active. Then create a test duplicate record as the affected user and confirm the alert appears.

7. **Competent:** A standard report type cannot express "Contacts WITHOUT Opportunity Contact Roles" — standard reports can filter on fields that exist, but they cannot natively return records that have **no related records** in a joined object. The mechanism for this is a **Cross-Filter** (available in standard and custom report types): add a Cross-Filter to the report with "Contacts without Opportunity Contact Roles." Cross-filters answer "show me records with no matching related record" without needing a custom report type. **Trap:** Trying to filter on an empty Opportunity Contact Role field — if the join produces no child rows, the Contact row is simply absent from a joined report, so a standard filter on a null field won't work. Or building a Custom Report Type unnecessarily (cross-filters are the correct tool for the "without" pattern). **Verify:** Build the report with a cross-filter "Contacts without Opportunity Contact Roles," run it, and spot-check a few returned Contacts against their related lists to confirm they have no OCRs.

8. **Competent:** The most likely gap is that the Flow-backed email action is **not linked to any topic** in Agent Builder. In Agentforce, actions must be explicitly added to a topic — an action that exists in the action library but is not associated with an active topic is unreachable by the agent, even if the Flow itself works. The fix is to open Agent Builder, navigate to the relevant topic, and add the email action under that topic's Actions section. **Trap:** Debugging the Flow itself or checking FLS on the email fields — the Flow works (confirmed in testing) and the issue is the agent's topic/action configuration, not the Flow. Or assuming the agent automatically discovers all active Flows. **Verify:** In Agent Builder, open the topic the user is conversing on → Actions tab → confirm the email action appears in the list. Then re-test with a conversation prompt that clearly maps to that topic.

9. **Competent:** The "You cannot deploy to a required field" error is triggered when a `<fieldPermissions>` block in the XML references a field that Salesforce considers **required** — but the definition of "required" here includes **fields that are required by a validation rule or record type**, not just the field-level Required checkbox. It can also be triggered if the field was recently changed to required in the org after the metadata was pulled, creating a mismatch between the XML and the current org state. The fix is to re-pull the field metadata from the org, check the current `required` attribute, remove the `<fieldPermissions>` entry for any field that is now required, and redeploy. **Trap:** Assuming the field metadata XML is authoritative and the org is wrong — the live org state always wins; metadata XML is a snapshot and can drift. **Verify:** `describe_object('Opportunity')` and inspect the `nillable` and `updateable` flags on `Contract_Value__c` in the live org; a required field will show `nillable: false`. If the field was changed to required after the pull, redeploy with the updated metadata.

10. **Competent:** Standard Salesforce reports **cannot calculate elapsed business hours** natively — they can filter on date fields and show time stamps, but they have no built-in function to compute the duration of a Case's open state against configured business hours. Business-hours-aware time calculations (as used in Entitlements and Milestones) are not exposed as report formula functions. To satisfy this requirement, you would need either: (a) an **Entitlement with a Milestone** that tracks the 48-business-hour SLA and populates a date/time field on the Case that can then be reported, or (b) a custom formula field on Case that approximates elapsed hours (without business-hours awareness), or (c) a custom Apex/Flow solution that stamps a "business hours elapsed" field at closure. **Trap:** Assuming a custom report formula can call business-hours logic — SFDC report formulas do not have access to the BusinessHours Apex class or similar runtime logic. **Verify:** Check whether an Entitlement/Milestone is already configured on the org; if so, query `SELECT Id, MilestoneTypeId, CompletionDate, IsCompleted FROM CaseMilestone` to see if the milestone data is already being captured.

11. **Competent:** Configure **On-Demand Email-to-Case**. The architectural difference is that On-Demand Email-to-Case routes inbound email **through Salesforce's servers** (using a Salesforce-provided email address, e.g. `<org>@emailtosalesforce.com`) rather than requiring a direct SMTP connection from a customer-controlled mail server. Customers forward or redirect inbound mail to the Salesforce address; Salesforce receives it and creates the Case. Standard Email-to-Case requires an open inbound SMTP port on your own mail server so Salesforce's agent can poll it — hence the firewall requirement. On-Demand has no such requirement. **Trap:** Configuring standard Email-to-Case and attempting workarounds such as firewall exceptions — the admin should recognize On-Demand is the purpose-built solution for this constraint. **Verify:** Setup → Email-to-Case → confirm "Enable On-Demand Email-to-Case" is checked; create a routing address and confirm test emails sent to the Salesforce address create Cases.

12. **Competent:** Check **whether the flow's active version has changed or a new version was deployed but not activated**. A common deployment-related cause is that a new inactive flow version was deployed to production (overwriting or alongside the active version), and the component on the page is now pointing to the correct flow API name but executing a version with a broken or changed element. Also check **record-triggered validation rules or required fields** on the object that may have been added in the same release and are now blocking the flow's DML. The first stop is **Setup → Flows → [Flow Name]** to confirm which version is active and review its Save element configuration. **Trap:** Assuming the flow itself is the bug and making changes to it in production — always debug in a sandbox first. Or clearing the component from the page and re-adding it (doesn't address an underlying version or validation conflict). **Verify:** In Setup → Flows, confirm the correct version is Active; check Setup Audit Trail for any validation rule or field changes deployed in the same release window; test the flow in isolation via the Flow debug tool in a sandbox with a real record ID.
