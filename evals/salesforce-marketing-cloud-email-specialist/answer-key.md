# Answer key — salesforce-marketing-cloud-email-specialist (held-out set, 2026-06-07)

PASS = competent move identified AND trap avoided. Partial = right instinct, misses the key rule or trap.

1. **Competent:** The team blasted full volume on a brand-new dedicated IP with no warm-up period. A new IP has no sending reputation, so ISPs throttle or defer high volumes from it. The correct approach is a **gradual IP warm-up over approximately 30 days**, starting with a small daily send (e.g., ~50–500 highly engaged subscribers on day one) and roughly doubling volume every few days, allowing ISPs to learn the IP's reputation before full volume is applied. **Trap:** Assuming that because the content and authentication (SPF/DKIM/DMARC) are correct, volume is irrelevant — IP reputation is built independently of content legitimacy. Some teams also try to split the send across two days to "warm up faster," which doesn't substitute for a multi-week ramp. **Verify:** Check the Email Performance by Domain report for deferral codes; a new-IP warm-up plan should show progressively improving delivery rates over 3–4 weeks. Monitor daily complaint and bounce rates against the volume curve.

2. **Competent:** Re-importing hard-bounced or unsubscribed addresses using **Add/Update** mode silently re-activates their SFMC subscriber record, resetting their status to Active or overwriting their Bounced/Unsubscribed status. This constitutes re-subscribing people who opted out — a CAN-SPAM violation — and re-mailing addresses that already hard-bounced, which harms sender reputation. The correct action is to never re-import suppressed addresses into a live subscriber population; a suppression file should only be loaded into a dedicated suppression list or held externally for exclusion logic. **Trap:** Believing that "updating records" on an existing file is a neutral administrative action — in SFMC, Add/Update on subscriber records overwrites status fields. Some admins also use Overwrite mode thinking it protects existing records; it truncates and reloads the entire DE, which could have other destructive consequences. **Verify:** After any import touching suppressed addresses, spot-check the All Subscribers list status for a sample of known hard-bounced addresses; if they show Active, the import caused re-subscription. Cross-reference the Bounce Summary report for those addresses.

3. **Competent:** Activities placed in the **same Automation Studio step run in parallel**. The SQL Query and the Send Email activity start simultaneously, so the Send Email targets the DE before the SQL has written the current audience into it — resulting in the send pulling stale data from the prior run. The fix is to move the SQL Query into **Step 1** and the Send Email into **Step 2**, forcing sequential execution so the audience DE is fully populated before the send starts. **Trap:** Assuming SFMC executes activities in left-to-right order within a step, or thinking a brief difference in start time within the same step provides enough lead time for the SQL to finish first. Adding a Wait activity inside the same step does not help because Wait is a step-level construct and does not sequence intra-step activities. **Verify:** In Automation Studio, open the automation and confirm the SQL Query and Send Email appear in separate numbered steps, SQL preceding Send. After a test run, check the Activity History — if both show the same start timestamp (to the minute), they ran concurrently and the fix is needed.

4. **Competent:** This send is **not compliant and must not proceed**. CASL (Canada's Anti-Spam Legislation) requires **express prior consent** from each recipient before sending a commercial electronic message. A purchased list does not carry individual express consent — the broker's blanket collection is not sufficient under CASL. CAN-SPAM compliance (physical address + unsubscribe) is necessary but not sufficient; CASL operates independently and carries penalties up to CAD $10 million per violation. Additionally, purchased lists typically contain spam traps and invalid addresses that damage sender reputation. The send must be stopped; the list must be discarded and replaced with a consent-based list (e.g. Web Collect / Smart Capture with a documented opt-in record). **Trap:** Assuming CAN-SPAM compliance covers the Canadian audience because it covers the U.S. portion of the same send. CAN-SPAM and CASL are separate regimes; satisfying one does not satisfy the other. Some teams also argue that a nonprofit's "relevant" communications are exempt — CASL has no blanket nonprofit exemption for fundraising solicitations. **Verify:** Review the consent documentation for each address: CASL requires you to be able to prove express consent with a timestamp, IP address, and the specific form or mechanism used. If that documentation does not exist for the list, the list cannot be used.

5. **Competent:** In Journey Builder, when a journey is **running (published)**, editing the email in Content Builder does not automatically push the updated content to in-flight or newly entering contacts. The journey version holds a reference to the email as it existed when the version was published. To deploy updated copy, the team must **publish a new journey version** that references the updated email, then allow new entrants to flow through the new version. Contacts who are already mid-journey stay on the old version until they exit. **Trap:** Assuming that saving and previewing the email in Content Builder is sufficient — Content Builder changes take effect in any journey only after a new journey version is published. Some teams stop the journey to "force" the update, which ejects all in-flight contacts and loses their progress. **Verify:** In Journey Builder, open the journey, click "Create Version," wire in the updated email, and publish the new version. Confirm that contacts entering after the new version goes live receive the updated headline by checking Journey History and performing a test entry.

6. **Competent:** The welcome-journey entrant DE should be applied as a **Suppression List** on the year-end appeal send. A suppression list excludes the matched addresses from the send regardless of their publication list opt-in status. The suppression is applied in the send configuration (Send Definition or Guided Send wizard) alongside the target audience DE. The correct pipeline: target the appeal DE (the opted-in donor audience) + suppress the journey entrant DE. **Trap:** Applying the welcome-journey entrant DE as the **send target** instead of as a suppression — if someone sets the entrant DE as the "To" audience, the send goes only to those contacts (the opposite of the intent), and depending on size this could mean sending to a tiny fraction of intended recipients or nobody if it is misapplied in the suppression slot. Confusing publication lists with suppression lists is the canonical red flag: "applying a suppression list as the send target sends to nobody; applying a publication list as a suppression silently blocks your intended audience." **Verify:** Before the live send, run a Test Send to a test subscriber who appears in the journey entrant DE and confirm they do not receive the appeal; confirm a subscriber who is NOT in the entrant DE does receive it.

7. **Competent:** The measurement problem is that **pixel-based open tracking is unreliable since Apple Mail Privacy Protection (MPP)**, which pre-fetches tracking pixels on behalf of the recipient. A contact recorded as "opened" in `_Open` may never have actually engaged — the pixel fire was triggered by Apple's proxy, not by the user. Building a re-engagement suppression list from opens inflates the "engaged" segment and suppresses people who are actually inactive. The competent fix is to **base re-engagement decisions on clicks (`_Click`)** and downstream conversion signals rather than opens, and to calculate **CTOR (Click-to-Open Rate)** as the primary engagement metric. Contacts with zero clicks in 90–180 days should be the re-engagement or suppression target. **Trap:** Trusting the `_Open` Data View as a reliable engagement proxy and simply extending the lookback window (e.g. to 120 days) instead of switching to a click-based signal. Some teams also apply MPP corrections by subtracting "machine opens," but this is imprecise and click-based segmentation is more reliable. **Verify:** Cross-reference the `_Open` cohort against the `_Click` Data View: a healthy list should show a click-to-open ratio; if the open count is large but clicks are near zero, MPP inflation is present. Switch the suppression query to `LEFT JOIN _Click` with a 90-day filter.

8. **Competent:** The missing guard is a **`RowCount()` check before the loop**. When `LookupOrderedRows()` returns zero rows, the loop body does not execute — no error is thrown, but no content renders either. The fix is to wrap the loop in an `IF RowCount(@rows) > 0 THEN ... ELSE` block that renders a graceful fallback message (e.g., "Your donation history is not yet available.") when no rows are returned. This prevents the blank section from reaching subscribers who have no data in the DE. **Trap:** Concluding that "no error = the code is correct" and assuming the blank is a CSS/HTML rendering issue rather than a zero-row AMPscript result. Some developers add an `IF Empty()` guard only on a single `Lookup()` call and overlook that `LookupOrderedRows()` requires a `RowCount()` pattern. **Verify:** In Subscriber Preview, select a subscriber with zero records in the donation DE and confirm the rendered email shows the fallback text rather than a blank section. Then select a subscriber with exactly one record and confirm the single-row table renders correctly.

10. **Competent:** The most likely cause is that the image block is stored in a **Content Builder folder in the parent business unit that is not shared with the child BU**. Blocks in a parent BU's private or unshared folder are not accessible to child BUs; the child BU's email editor cannot resolve the block reference and throws "content not found." The fix is to move the image block (and any other assets needed by the template) into a **shared Content Builder folder** that is configured to be visible to child BUs, then update the template's slot to reference the asset from the shared location. **Trap:** Assuming the issue is a template permission problem or that recreating the image block inside the child BU is the canonical fix — recreating the block in each BU creates duplicate assets that fall out of sync; using a shared folder is the scalable, correct solution. **Verify:** In the parent BU, navigate to Content Builder, locate the image block, check its folder, and confirm the folder has sharing enabled for the relevant child BUs. 11. **Competent:** (a) **ESTO is appropriate** — it individually schedules each contact's journey email delivery at their predicted optimal time, which is exactly its design. For Engagement Scoring, verify the score field is populated on the entry DE records and used in a Decision Split or pre-journey filter.
