1. **Competent:** The root cause is SSE-S3 on the artifact bucket. Cross-account pipelines require a **customer-managed KMS key (CMK)** — SSE-S3 keys are owned by the tools account and cannot have cross-account grant policies, so the prod account's role cannot decrypt the artifacts. Fix: re-encrypt the artifact bucket with a CMK, add `kms:Decrypt` and `kms:GenerateDataKey` to the prod account's pipeline role, and grant cross-account access in the CMK key policy. **Trap:** assuming IAM role permissions alone are the problem and adding more S3 permissions, which will not resolve a KMS decryption failure. **Verify:** `aws kms describe-key --key-id <arn>` to confirm it is a CMK; `aws kms get-key-policy --key-id <arn> --policy-name default` to confirm the prod account principal is in the key policy.

2. **Competent:** CodeDeploy canary deployments shift to 100% automatically only after the bake period has elapsed **and** no configured rollback alarm has fired. If the deployment is stuck past the bake time, the most likely cause is that a CloudWatch alarm configured as a rollback trigger is in `ALARM` or `INSUFFICIENT_DATA` state — CodeDeploy treats `INSUFFICIENT_DATA` as a reason to pause or roll back, not as "healthy." Check the deployment's alarm configuration, resolve the alarm state, or — if the alarm is spurious — update the alarm's `treat_missing_data` setting. You cannot manually "advance" a CodeDeploy canary mid-deployment without stopping and redeploying. **Trap:** assuming the deployment is healthy and will eventually proceed on its own, or attempting to manually invoke the Lambda alias shift outside of CodeDeploy. **Verify:** `aws deploy get-deployment --deployment-id <id>` to see `deploymentOverview` and error reason; `aws cloudwatch describe-alarms --alarm-names <alarm>` to check alarm state.

3. **Competent:** `DBInstanceClass` is a **Some Interruption** update (not Replacement) — RDS will reboot the instance during the resize, causing a brief outage typically 1–5 minutes, not zero downtime. The junior engineer is incorrect. Before applying to production, create a change set with `aws cloudformation create-change-set` and inspect it for the `Requires Replacement: False` / interruption type column. For zero-downtime, the correct approach is to use a read replica promotion or failover on Multi-AZ, not a direct instance class change. **Trap:** confusing "no replacement" with "no interruption" — a `Some Interruption` update still causes a reboot and connection drop. **Verify:** Change set output shows `Replacement: False` and `RequiresRecreation: Never`; review RDS event logs after the update for the actual reboot duration.

4. **Competent:** The Lambda function returned HTTP 200 to your own invocation, but CloudFormation communicates with Custom Resources via a **pre-signed S3 URL** included in the event payload, not via the Lambda return value. The function must explicitly send a `PUT` request to that URL with a response body containing `Status: SUCCESS` or `Status: FAILED`. If the function exits without making that PUT, CloudFormation waits up to one hour for a response that never arrives. Fix the Lambda handler to always PUT to `event['ResponseURL']`. To unblock the stuck stack now, manually send a `SUCCESS` response to the pre-signed URL (if it is not yet expired) or wait for the 1-hour timeout and then delete/rollback the stack. **Trap:** re-invoking the Lambda manually or assuming the HTTP 200 return satisfies CloudFormation. **Verify:** CloudWatch Logs for the function should show the outbound PUT to the pre-signed URL; if absent, the response was never sent.

5. **Competent:** The immediate fix is to add an **Auto Scaling lifecycle hook** on the `autoscaling:EC2_INSTANCE_TERMINATING` event. The hook holds the instance in `Terminating:Wait` state while the ALB connection draining (deregistration delay, default 300 seconds) completes before termination. Without the hook, the ASG may terminate the instance before the ALB has drained active connections. Also confirm the ALB target group's **deregistration delay** is set to a value sufficient to drain sessions. **Trap:** increasing the ASG cooldown period — cooldown governs when the next scale action fires, not whether active sessions are drained from a terminating instance. **Verify:** `aws autoscaling describe-lifecycle-hooks --auto-scaling-group-name <asg>` to confirm the hook is present; monitor ALB access logs for 5xx errors during the next scale-in event.

6. **Competent:** A single-region trail only captures API events in that one region. The missing configuration is a **multi-region trail** (`--is-multi-region-trail`), which replicates all global and regional API events to one S3 bucket. The gap means any API-level changes in the other six regions — including IAM events in `us-east-1` that affect global services — may be incomplete. To remediate without disrupting the existing trail, update it with `aws cloudtrail update-trail --name <trail> --is-multi-region-trail` or create a new multi-region trail. Also enable **log file validation** (`--enable-log-file-validation`) if not already set. **Trap:** creating six separate single-region trails — technically functional but operationally fragile and more expensive than one multi-region trail. **Verify:** `aws cloudtrail get-trail --name <trail>` — confirm `IsMultiRegionTrail: true` and `LogFileValidationEnabled: true`; `aws cloudtrail get-trail-status --name <trail>` — confirm `IsLogging: true`.

7. **Competent:** The on-call engineer is wrong. `INSUFFICIENT_DATA` means the alarm has not received enough data points to evaluate its threshold — it is not equivalent to `OK`. For a new Lambda function, this state is normal immediately after creation while CloudWatch waits for metric data to accumulate, but after 20 minutes of confirmed invocations the metric should be reporting. Check whether the Lambda function's metrics are being published under the correct namespace and dimension (function name). Also check whether the alarm's evaluation period and datapoints-to-alarm configuration require more data points than have arrived. Setting `treat_missing_data: notBreaching` suppresses false alarms during cold starts but must be deliberate, not assumed. **Trap:** treating `INSUFFICIENT_DATA` as a green/healthy state and ignoring it — a misconfigured alarm in this state will never fire even during a real failure. **Verify:** `aws cloudwatch describe-alarms --alarm-names <name>` to inspect `StateValue`, `Namespace`, `Dimensions`, `Period`, and `TreatMissingData`; `aws cloudwatch get-metric-statistics` to confirm data points are arriving.

8. **Competent:** The most likely cause is that the unpatched instances are **not tagged with the Patch Group tag** matching the Patch Baseline assignment. Patch Manager uses the `Patch Group` tag (key is exactly `Patch Group`, case-sensitive) on instances to associate them with a baseline. Instances without the tag fall into the default baseline, which may not be what the Maintenance Window targets. Check: (1) instance tags for `Patch Group`; (2) the Maintenance Window's target — confirm it uses a tag-based or resource-group filter that actually resolves to those instances. **Trap:** blaming SSM Agent connectivity — the question states all instances show as "Online" in Fleet Manager, which already confirms agent reachability. **Verify:** `aws ssm describe-instance-patch-states --instance-ids <ids>` to see patch state; `aws ssm describe-maintenance-window-targets --window-id <id>` to confirm target filter; `aws ec2 describe-tags --filters Name=key,Values="Patch Group"` to audit instance tagging.

9. **Competent:** `SELF_MANAGED` StackSets require manually created IAM roles — `AWSCloudFormationStackSetAdministrationRole` in the management/admin account and `AWSCloudFormationStackSetExecutionRole` in **each** target account — with the correct trust policies between them. The engineer created the execution role in only one account; the other two have no execution role, causing IAM errors. The fix is to deploy `AWSCloudFormationStackSetExecutionRole` (with a trust to the admin account) in all target accounts. Alternatively, switch to `SERVICE_MANAGED` (Organizations-integrated) to eliminate the manual role setup entirely. **Trap:** trying to fix the problem by adding permissions to the admin account's role — the missing piece is the execution role in the target accounts, not the admin role. **Verify:** `aws iam get-role --role-name AWSCloudFormationStackSetExecutionRole` in each target account to confirm the role exists and its trust policy lists the correct admin account principal.

10. **Competent:** AWS Config auto-remediation has a **retry limit and a cooldown window** — if the remediation SSM Automation document fails (e.g., the document doesn't have the right permissions, or the bucket policy blocks the remediation role's `s3:PutBucketPublicAccessBlock` call), Config will retry up to the configured limit and then stop. After the retry window, no further automatic attempts are made. Check: (1) SSM Automation execution history for the remediation document — look for failed executions; (2) the IAM role attached to the remediation action — confirm it has `s3:PutBucketPublicAccessBlock`; (3) whether the bucket has a resource policy denying the remediation role. **Trap:** assuming the remediation "hasn't run yet" and just waiting longer — Config does not retry indefinitely; a failed remediation stays failed until you investigate and manually re-trigger. **Verify:** `aws config get-remediation-execution-statuses --config-rule-name s3-bucket-public-read-prohibited` to see execution status and failure reason; `aws ssm list-automation-executions` to review the document's run history.

11. **Competent:** Storing the plaintext secret in the buildspec is wrong for two reasons: (1) CodeBuild **logs the values of environment variables** in the build log, which is stored in CloudWatch Logs and optionally in S3 — anyone with access to those logs sees the secret in plaintext; (2) the buildspec in source control means the secret is committed to version history and visible to anyone with repo read access now or in the future. The correct pattern is to store the secret in AWS Secrets Manager (or Parameter Store SecureString), then reference its ARN in the buildspec's `env.secrets-manager` or `env.parameter-store` block — CodeBuild fetches and injects the value at runtime without it appearing in logs. **Trap:** believing "private repo" is sufficient protection — the exposure surface includes build logs, pipeline console output, and anyone cloning the repo. **Verify:** `aws codebuild batch-get-projects --names <project>` — confirm the environment variables reference parameter/secrets ARNs, not literal values; `aws logs filter-log-events --log-group-name /aws/codebuild/<project>` — confirm no secret values appear in build log output.

12. **Competent:** The most likely missing configuration is a **Route 53 health check on the primary record**. Failover routing requires an active health check associated with the primary record — without it, Route 53 has no signal that the primary endpoint is unhealthy and will never switch to the secondary. Create a Route 53 health check targeting the primary ALB's endpoint (HTTP/HTTPS), associate it with the primary DNS record, and set the failover record type to `PRIMARY`. **Trap:** assuming the DNS failover is triggered by Route 53 detecting ALB health checks internally — Route 53 does not inspect ALB health check results; it only uses health checks you explicitly configure and associate with the record. **Verify:** `aws route53 get-health-check-status --health-check-id <id>` to confirm health check state; `aws route53 list-resource-record-sets --hosted-zone-id <zone>` to confirm the primary record has `HealthCheckId` set and `Failover: PRIMARY`; dig or `nslookup` from an external host to confirm DNS resolves to the secondary during the simulated failure.
