1. **Competent:** Replace `<ScrollView>` + `.map()` with `<FlatList>`. `ScrollView` renders all 500 items into memory immediately at mount, causing the 3–4 s load time and layout jank regardless of memoization. `FlatList` virtualizes the list — it only renders the visible window plus a small buffer, making initial render instant and scroll smooth. `keyExtractor` returning a unique stable string is also required. **Trap:** Adding `React.memo` to the item component — memoization cannot help when `ScrollView` forces all 500 items to mount simultaneously; it addresses re-render cost, not initial mount cost. **Verify:** Replace with `FlatList`; initial load time drops to well under 500 ms and scrolling achieves 60 FPS in a release build profile.

2. **Competent:** EAS Update delivers JS updates only to builds whose **runtime version exactly matches** the published update. Users still crashing likely have an older binary whose runtime version differs from the update's target runtime version — the update is never offered to them. Check `eas.json` and `app.json` for `runtimeVersion`; confirm the affected builds are on a matching runtime version. Users on an incompatible binary require a new store submission with the fix compiled into native code (or at minimum a binary that shares the correct runtime version). **Trap:** Assuming all users on the `production` channel automatically receive the update — channels filter by branch, but runtime version is an additional hard gate. **Verify:** In the EAS dashboard, confirm which runtime versions received the update and which builds are excluded; cross-reference with crash reporter device counts.

3. **Competent:** The missing step is running `eas build` (or `npx expo prebuild`) to regenerate the native binary with the new `expo-camera` library compiled in. In Expo Go, all Expo SDK modules are pre-bundled in the Expo Go client; in a standalone production build, each SDK module must be compiled into the native binary. Without a new build, the native camera module is absent and the app will either silently fail to open the camera or crash at runtime. **Trap:** Assuming that publishing an EAS Update (`eas update`) after adding `expo-camera` is sufficient — OTA updates cannot deliver new native modules; a new binary is required. **Verify:** Run `eas build --profile production`, install the resulting binary, and confirm `Camera.requestCameraPermissionsAsync()` succeeds without crashing.

4. **Competent:** The root cause is the synchronous 200 ms computation executing on the JS thread during the navigation transition, which starves the JS thread and causes frame drops. Switching to `createStackNavigator` would make things worse, not better — the JS-based stack renderer is more sensitive to JS thread load than the native stack, which delegates transitions to the OS. The correct fix is to defer the expensive computation with `InteractionManager.runAfterInteractions(() => { /* sort here */ })` so it runs after the transition animation completes, or to move the sort to a memoized selector that runs before navigation. **Trap:** Switching to `createStackNavigator` — the JS stack runs on the JS thread and will drop frames under any JS load; `createNativeStackNavigator` transitions run on the UI thread and are unaffected by JS work. **Verify:** Defer computation with `InteractionManager.runAfterInteractions`; transition completes at full frame rate; sort result appears after the transition without visible jank.

5. **Competent:** Animating `width` on an `<Image>` forces iOS to re-crop from the original source image on every animation frame, which is a GPU-intensive operation that causes frame drops even in release builds. The fix is to animate `transform: [{ scale }]` instead of `width`/`height` — scale transforms are composited on the GPU without re-sampling the image. `useNativeDriver: true` is already correct and necessary, but it cannot compensate for the underlying re-crop cost caused by a dimension change. **Trap:** Removing `useNativeDriver: true` to "debug" — that would make performance worse, pushing animation work back to the JS thread. Alternatively: assuming `useNativeDriver: true` alone is sufficient regardless of which property is animated. **Verify:** Replace `width` animation with `transform: [{ scale }]`; re-profile in a release build and confirm the animation sustains 60 FPS.

6. **Competent:** The Android keystore is permanently and irrecoverably tied to the Google Play listing. Without the original keystore, it is **impossible** to publish an update to the existing Play Store app — Google Play will reject any APK or AAB signed with a different key. The only viable path forward is to publish a new app under a new package name (`applicationId`) and request that existing users migrate. Data migration from the old app (if any) must be planned carefully. **Trap:** Believing EAS can regenerate or replace the keystore — EAS stores the keystore it generated, but if the original keystore was managed outside EAS and is lost, EAS cannot recover it. Also: attempting to contact Google to "reset" the signing key — Google does not allow this without the original key. **Verify:** Confirm with `eas credentials` whether EAS ever held this keystore; if not, attempt a Play Console upload of a new signed build and observe the rejection. The new package-name path is the only production-viable resolution.

8. **Competent:** SDK 55 makes the New Architecture mandatory — it cannot be disabled. 9. **Competent:** Add `getItemLayout` to the `FlatList`. Its signature is: `getItemLayout={(data, index) => ({ length: 120, offset: 120 * index, index })}`. This tells RN the pixel size and position of every item without measuring, enabling instant `scrollToIndex` for arbitrary indices. If item heights are variable (not the case here but worth stating), `getItemLayout` cannot be provided accurately — the alternative is to use `scrollToIndex` only for indices within the rendered window, or switch to a library like FlashList that handles variable heights with its own measurement strategy. **Trap:** Calling `scrollToIndex` without `getItemLayout` and expecting it to work for large indices — RN must measure every intervening item to compute the scroll position, which fails for off-screen items that have never been rendered. **Verify:** Add `getItemLayout` with the fixed 120 dp height formula; `scrollToIndex({ index: 3000 })` resolves without warning and the list snaps to the correct position.

10. **Competent:** Use `behavior="padding"` on iOS and `behavior="height"` on Android (or test both and hard-code via `Platform.select`). The two platforms have fundamentally different window-resize behaviors when the soft keyboard appears: iOS shrinks the visible window height and `"padding"` adds bottom padding to push content up; Android by default resizes the entire window (`adjustResize`) so `"height"` mode correctly responds to the available area shrinking. Using a single value for both platforms produces incorrect results on at least one. **Trap:** Using `behavior="padding"` for both platforms and concluding that `KeyboardAvoidingView` is broken on Android — the behavior is correct for iOS but mismatched for Android's resize model. **Verify:** Test the input field visibility on a physical Android device (or emulator) with both behaviors; confirm the input remains above the keyboard after applying `Platform.select({ ios: 'padding', android: 'height' })`.

11. **Competent:** Replace `AsyncStorage` with `expo-secure-store`. `AsyncStorage` stores data as **plaintext** in the app's sandboxed storage: on iOS, unencrypted in the app's `Documents` directory (readable if the device is jailbroken or backed up without encryption); on Android, unencrypted in the app's internal storage (readable on rooted devices). `expo-secure-store` uses iOS Keychain and Android Keystore — both hardware-backed encrypted storage — making tokens inaccessible even on compromised devices. `expo-secure-store` has a per-item size limit (~2 KB), so very large tokens may need to be split or stored differently. **Trap:** Adding encryption on top of `AsyncStorage` (e.g. encrypting with a key also stored in `AsyncStorage`) — this provides no meaningful security because both the data and the key are in the same unprotected plaintext store. **Verify:** Store a token with `expo-secure-store`, back up the device, inspect the backup — the token should not be readable; contrast with an `AsyncStorage` backup where the value is visible as plaintext.

12. **Competent:** In React Native, every string rendered to the screen must be wrapped in a `<Text>` component. Rendering a bare JavaScript string directly inside a `<View>` is a **runtime error on Android** — it throws `"Invariant Violation: Text strings must be rendered within a <Text> component"` and causes a hard crash. The fix is `<View>{isLoading && <Text>Loading...</Text>}</View>`. On iOS, the same code may silently render nothing or display incorrectly in development mode but does not necessarily hard-crash — this is a known platform inconsistency, not correct iOS behavior. **Trap:** Concluding the code is fine because it renders on iOS — iOS behavior in this case is coincidentally non-crashing (a renderer quirk, not a guarantee), and the code is incorrect on both platforms per the React Native specification. **Verify:** Run the component on a physical Android device or Android emulator; the crash is immediate and shows `Text strings must be rendered within a <Text> component` in the stack trace.
