6. **Competent:** A `for` retry loop inside a `@future` method is technically allowed (no platform rule prohibits looping on callouts within one transaction up to the 100-callout limit) but is architecturally wrong because **`@future` methods have no state persistence, no monitoring surface, and cannot be chained or scheduled for later retry**. If all three attempts fail (e.g., the external service is down for 10 minutes), the failure is unrecoverable within that transaction and the error is swallowed unless explicitly logged. The correct pattern is to replace the `@future` method with a **Queueable** that implements `Database.AllowsCallouts`: on a 5xx response, the `execute()` method can enqueue a new Queueable child job (with an attempt counter passed via the constructor) to retry after a delay — up to the maximum depth. This makes retries visible in `AsyncApexJob`, monitorable, and properly sequenced. **Trap:** Adding a `System.schedule()` call inside the `@future` method to schedule a future retry — `System.schedule` is not callable from a `@future` context (it's a setup object DML violation). Also wrong: calling another `@future` from within the `@future` method — chaining `@future` calls is not supported (the platform throws `CalloutException`). **Verify:** Implement the Queueable retry with a mock that returns 500 on the first two calls and 200 on the third. Wrap in `Test.startTest()/stopTest()`, assert that a success log record exists after three attempts, and confirm `AsyncApexJob` shows the expected number of enqueued jobs.

10. **Competent:** The reviewer is **partially correct but uses the wrong keyword**. When a `without sharing` class explicitly calls a method on a helper class, the helper's sharing mode is determined by the helper's own declaration — so a `without sharing` helper ignores the caller's `with sharing` context and can access records the running user cannot see. However, the fix is not to make the logging helper `with sharing` (which would break it when called from `without sharing` system-context callers that legitimately need to log unrestricted records). The correct keyword is **`inherited sharing`**: a class declared `inherited sharing` adopts the sharing context of its caller. When called from a `with sharing` class, it enforces sharing; when called from a `without sharing` class, it runs without sharing. This makes the helper safe to reuse in both contexts. **Trap:** Declaring the logging helper `with sharing` unconditionally — this prevents the helper from being used in system-context callers (e.g., integration handlers that legitimately run `without sharing` and need to log records outside the current user's visibility). Or leaving it `without sharing` and arguing that logging is a "system operation" that should always bypass sharing — this is acceptable for some helpers but in a `with sharing` caller context it silently circumvents record access enforcement, which is what the reviewer is correctly flagging. **Verify:** Write two test scenarios: one that calls the logging helper from a `with sharing` context where the running user lacks access to a record, and assert the helper respects the restriction; one that calls it from a `without sharing` context and assert it can access the same record. `inherited sharing` passes both.
