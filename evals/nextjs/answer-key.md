# Answer key — nextjs (grading rubric)

> _Held-out eval content — original, not exam material (no real exam questions; see POLICY.md). Do not paste into a skill body._

PASS = competent move identified AND trap avoided. Partial = right instinct, misses the rule/trap. FAIL = wrong action or falls for the trap.

---

1. **Competent:** `"use client"` converts the entire module — including its imports — into the client bundle. An `async` component with a direct `db.query(...)` call in a Client Component ships DB credentials/ORM logic to the browser and fails at runtime because ORMs are not browser-safe. The correct structure: keep `page.tsx` as a Server Component (no directive), fetch data there, and pass a serializable result as props to a thin Client Component for any interactive pieces. **Trap:** "it works in dev" is not a signal; Next.js dev mode is more permissive than the production build, and the ORM call in a Client Component may not throw until the client bundle actually runs in a browser context. **Verify:** run `next build`; inspect the client bundle with `@next/bundle-analyzer` to confirm no DB client code is present.

2. **Competent:** The correct fix is to create a thin wrapper Client Component (e.g. `ThemeBannerClient.tsx`) that contains `"use client"` and the `localStorage` read, and render it from the Server Component parent. The Server Component itself stays a Server Component. Adding `"use client"` to the existing component file is the trap — if `ThemeBanner` is large or imports other Server-only modules, promoting the entire file to a Client Component widens the bundle unnecessarily and may pull in server-only code. **Trap:** adding `"use client"` to the existing file and treating the problem as solved. **Verify:** confirm the wrapper is the smallest possible boundary; run `next build` and check that `server-only` imports in the surrounding tree still resolve correctly.

3. **Competent:** Import `CartButton` inside `ProductList` but never call it directly — pass it as `children` (or another prop) from a parent that assembles the tree. Server Components can render Client Components when they are passed *into* the Server Component's JSX as children/props; the Client Component is not *imported* inside the Server Component's module graph, so no boundary violation occurs. The pattern that silently fails: importing `CartButton` directly at the top of `ProductList.tsx` — Next.js will either error or silently convert the Server Component to a client component depending on version. **Trap:** `import CartButton from './CartButton'` directly inside the Server Component file. **Verify:** check that `ProductList` has no `"use client"` directive and `next build` does not warn about a Client Component import from a Server Component.

4. **Competent:** `route.ts` and `page.tsx` cannot coexist in the same route segment — Next.js throws a build conflict. Move the Route Handler to a different segment, e.g. `app/api/dashboard/route.ts` (or any path not shared with the page segment). The dashboard page at `app/dashboard/page.tsx` is unaffected. **Trap:** trying to merge the handler logic into the page, or renaming just the file without moving the folder. **Verify:** `next build` completes without conflict errors; hit both the dashboard URL and the API endpoint and confirm each responds correctly.

5. **Competent:** Every Server Action is independently reachable as a POST endpoint via its action ID — an attacker does not navigate through the UI to reach it. They can POST directly to the action endpoint bypassing the layout entirely, so the page-level `auth()` redirect provides zero protection for the action. The action must call `auth()` (or the DAL equivalent) at its own top, and additionally verify that the authenticated user is the owner of the resource being mutated (authz check, not just authn). **Trap:** reasoning that "users can't reach the form without being logged in, so the action is safe." **Verify:** use a tool like `curl` to POST directly to the Server Action endpoint ID without a session cookie; confirm the action returns a 401/403, not a successful mutation.

6. **Competent:** `revalidateTag` uses stale-while-revalidate semantics — it marks the cache entry as stale so the *next* background fetch can refresh it, but the immediately-following response may still serve the old value. For a read-your-own-writes scenario (the user must see their change right away), call `updateTag('posts')` inside the Server Action instead. `updateTag` synchronously removes the cache entry, so the next render for any user — including the submitter — forces a fresh fetch. **Trap:** using `revalidateTag` and assuming "revalidate" means "immediately purge." **Verify:** submit the form and confirm the count updates on the very next page load without a hard refresh or delay.

7. **Competent:** In v16, `middleware.ts` was renamed to `proxy.ts` and the exported function name changes from `middleware` to `proxy` (i.e., `export function proxy(...)`). Simply renaming the file but keeping `export { middleware }` leaves the proxy silently inert — Next.js looks for the `proxy` export, finds nothing, and skips execution without an error. Run `npx @next/codemod@canary middleware-to-proxy .` to rename the file and update the export name automatically. **Trap:** only renaming the file and not updating the export name (or vice versa). **Verify:** add a temporary `console.log` or response header inside the proxy handler; make a request and confirm the header is present in the response.

8. **Competent:** Wrap `fetchProductPrices()` in `React.cache()`. This memoizes the function's return value within a single request — all three Server Components calling it pay the fetch cost once. The cache scope is per-request only; it does not persist across requests (that would require `use cache` + `cacheTag`). Do not use a module-level variable as a cache — that persists across requests and users on the same server instance, which can leak data. **Trap:** reaching for `use cache` (which is correct for cross-request persistence but wrong if prices are user-specific or change each request) or using a module-level singleton. **Verify:** instrument the external API call with a counter or log; confirm it fires once per page render regardless of how many components call the function.

9. **Competent:** `cookies()` is a runtime API — any component that calls it synchronously inside a `layout.tsx` opts the entire layout subtree out of the static shell, blocking streaming for every segment below it. The minimal fix: extract the cookie-reading logic into a separate async component wrapped in `<Suspense>`. The layout shell (including the Suspense fallback) ships immediately; the session-reading component streams in at request time without blocking the rest of the page. **Trap:** wrapping the entire `layout.tsx` in `"use client"` or moving cookie logic to the page (which solves one segment but misses the architectural rule that layouts must not block streaming). **Verify:** use browser DevTools Network tab to confirm the initial HTML response arrives before the session component resolves, and that `loading.tsx` fallbacks appear immediately.

10. **Competent:** `updateTag` is only valid inside Server Actions — it is not callable from Route Handlers. In a Route Handler, `updateTag` is silently a no-op (it does not throw). The correct API for a Route Handler that needs to invalidate a cache tag is `revalidateTag('orders')`, which uses stale-while-revalidate semantics and is supported in both Server Actions and Route Handlers. **Trap:** assuming `updateTag` works anywhere you can call server-side code. **Verify:** trigger a webhook; confirm that after `revalidateTag` the next request fetches fresh order data (check via a log or cache-miss counter in the fetch layer).

11. **Competent:** Props passed from Server Components to Client Components must be serializable. JavaScript `Date` objects are not JSON-serializable — Next.js's serialization layer (which uses React's server-to-client transfer protocol) rejects them. The fix is to convert the `Date` to an ISO 8601 string before passing it as a prop (`createdAt.toISOString()`), then parse it back in the Client Component if needed. **Trap:** "it works in dev" — local Node.js environments may tolerate Date objects in certain code paths, masking the error until the production serialization boundary is enforced. **Verify:** run `next build && next start`; confirm the production server renders the page without a serialization error, and that the Client Component receives the value as a string.

12. **Competent:** Proxy (formerly Middleware) can be bypassed by an attacker who crafts a request with a manipulated internal routing header (CVE-2025-29927). The proxy is a UX convenience layer — it redirects browsers — but it is not the security enforcement point. An attacker posting directly to a Server Action or Route Handler with a forged header can skip the proxy redirect entirely. Every Server Action and sensitive Route Handler must independently verify authentication (and authorization) inside its own body, ideally via a Data Access Layer module marked `import 'server-only'`. **Trap:** concluding that 100% redirect coverage of browser traffic means the app is secure. **Verify:** use `curl` to POST directly to a sensitive Server Action endpoint with the internal bypass header set; confirm the action independently rejects the unauthenticated request.

13. **Competent:** The `experimental.useCache` flag (v15) and the `cacheComponents` flag (v16) are distinct configurations — setting one does not carry over to the other. When migrating to v16, `experimental.useCache: true` in `next.config.ts` is ignored; the v16 runtime looks for `cacheComponents: true`. The fix is to add `cacheComponents: true` explicitly in `next.config.ts` and remove the old experimental flags, following the Version 16 upgrade guide. Regarding `experimental.ppr`: in v15 this was a separate opt-in that enabled Partial Prerendering independently of `experimental.useCache`. In v16, `cacheComponents: true` is a unified flag that enables both `use cache` semantics AND PPR together as a single configuration — the two separate v15 flags are merged into one. A v15 app with `experimental.ppr` but without `experimental.useCache` had PPR but not the `use cache` directive model; `cacheComponents` requires and activates both. **Trap:** assuming any v15 `experimental.*` caching/PPR flag automatically migrates to `cacheComponents: true` in v16. **Verify:** after adding `cacheComponents: true`, run `next build` and confirm `use cache` directives generate cache entries (check with `NEXT_PRIVATE_DEBUG_CACHE=1 next dev`).
