1. **Competent:** The root cause is **missing Field-Level Security on `Integration_Key__c`**. Deploying a custom field via SFDX (`field-meta.xml`) creates the field but grants FLS to no one — including the integration user's profile or permission set. A SOQL query against a field for which the running user has no FLS read access returns zero rows silently (no error, no exception); the field is simply not readable from that user's perspective. The fix is to deploy (or manually assign) a Permission Set that includes `<fieldPermissions>` with `readable: true` for `Opportunity.Integration_Key__c`, then assign that permset to the integration user. **Trap:** Assuming the zero-row result means the matching records don't exist, and re-loading data or re-running the migration. Or checking OWD and sharing on the Opportunity object — those control record visibility, but the query returns zero rows (not a permission error) because the field itself is invisible, not the records. **Verify:** Run `SELECT Id, SObjectType, Field, PermissionsRead FROM FieldPermissions WHERE Field = 'Opportunity.Integration_Key__c'` via Tooling API. Confirm no row exists for the integration user's profile or permset. After deploying the permset, re-run the query and confirm the expected row count.

2. **Competent:** The error is **"Duplicate relationship name"**. The `relationshipName` on a Lookup field must be unique per parent object — two Lookup fields on Contact that both target Account cannot share the relationship name `PrimaryAccounts`. The first field deploys successfully; the second fails because a relationship named `PrimaryAccounts` already exists on the Account→Contact child relationship list. The fix is to assign distinct, role-specific `relationshipName` values — for example `PrimaryAccounts` and `SecondaryAccounts` (or more descriptive suffixes). Note that SOQL parent traversal uses the **field API name** (e.g. `Primary_Account__r.Name`), not the `relationshipName`, so renaming the relationship does not break existing queries. **Trap:** Thinking you can work around the conflict by reusing the same `relationshipName` across different child objects — the uniqueness constraint is per parent object, not globally, so the conflict is specifically because both fields point at Account. Also wrong: assuming the first field is broken and deleting it. **Verify:** After correcting the `relationshipName` values in the field metadata XML, re-run `sf project deploy start` and confirm both fields appear in Setup → Contact → Fields & Relationships with their distinct relationship names.

3. **Competent:** The migration used **insert instead of upsert**, creating new Opportunity records on the second run instead of updating the existing ones. The correct mechanism is to **mark a field as External ID + Unique** (e.g. `Source_CRM_ID__c`) on the Opportunity object, populate it with the stable identifier from the external CRM in the initial load, and use **upsert** (via Data Loader or Bulk API) on every subsequent run, keying on that External ID. Upsert creates the record if no matching External ID exists and updates it if a match is found — making the operation safe to re-run as many times as needed without duplicating records. **Trap:** Trying to prevent duplicates by configuring a Duplicate Rule on Opportunity. Duplicate Rules operate on user-facing saves and work on matching field values — they are not designed to make a programmatic insert idempotent, and they do not prevent Data Loader inserts from creating duplicates if the Block action is not set (and even then, the root fix is upsert). **Verify:** After marking the field External ID + Unique and switching to upsert, run the migration twice on a sandbox with a controlled set of records and confirm via SOQL `SELECT COUNT() FROM Opportunity WHERE Source_CRM_ID__c != null` that the count does not increase on the second run.

4. **Competent:** Salesforce Files access is governed by **Content Library membership and Sharing Settings on the ContentDocument** — "All Internal Users" having Viewer access to the Library grants visibility to the library itself, but **individual ContentDocuments linked to records may require a ContentDocumentLink** giving the user access to that specific file. The rep may not have a share on the specific document if it was uploaded to a library and not linked to a record the rep has access to. The correct fix is to verify whether a `ContentDocumentLink` exists for each affected rep (or a public group they belong to), and create one with `ShareType = V` (Viewer) if missing — or share the document from the file detail page directly with the relevant reps or a public group. **Trap:** Escalating the library permission from Viewer to Collaborator or Author thinking the reps need a higher library role — the library role governs what a user can do to the library itself (add/edit files), not whether they can open a specific existing file. Also wrong: granting the reps a more permissive profile. **Verify:** Query `SELECT Id, LinkedEntityId, ShareType FROM ContentDocumentLink WHERE ContentDocumentId = '<docId>'` and confirm the affected users or their groups appear. If not, insert the missing links and retest.

5. **Competent:** Forecast **adjustments are one level deep** — a manager can adjust a subordinate's forecast number, but that adjustment delta does not automatically propagate upward into the manager's own manager's view of the manager's forecast. The VP's manager sees the VP's "own" forecast number (the VP's rolled-up total before any adjustments the VP made to subordinate amounts), not the VP's adjusted view. To surface the adjusted number to the skip-level, the VP must either (a) make her own adjustment at her level to reflect her confidence or (b) understand that adjustments are peer-to-subordinate only and train the skip-level manager to drill into the VP's layer. **Trap:** Concluding the Forecast Category mappings are misconfigured or that there is a sharing/visibility issue preventing the skip-level manager from seeing the VP's numbers. The numbers are visible — the semantic behavior of adjustments (not cascading upward) is the issue. **Verify:** In the Forecasting UI, log in as the VP's manager's manager and drill into the VP's row. Confirm the unadjusted rollup is visible. Then check the VP's direct manager's forecast to see the adjustment delta applied there.

6. **Competent:** **No — Master-Detail is the wrong relationship type here.** Master-Detail cascades delete: if the parent Opportunity is deleted (e.g. the deal is lost and cleaned up), the child `Deal_Price_Book_Snapshot__c` record is automatically deleted too, destroying the intended historical record. For an audit or snapshot record that must persist even if the parent is removed or changed, use a **Lookup relationship**. A Lookup makes the relationship optional and non-cascading — the snapshot record survives independently. The same logic applies if the Price Book record itself were referenced: using Master-Detail from snapshot to Price Book would mean deleting the Price Book deletes the snapshot. **Trap:** Arguing that Master-Detail is correct because it enforces referential integrity — while that's true, the cost is cascade-deletion, which directly violates the requirement to "permanently record" the snapshot. Confirm the snapshot record still exists (Lookup) vs. is deleted (Master-Detail).

7. **Competent:** Adding FLS edit permissions will not fix the error. The failure is caused by a **restricted picklist value rejection** — the integration is sending a value that is not in the field's allowed-value list for the restricted picklist `Order_Status__c`. A restricted picklist enforces its allowed values at the API layer regardless of FLS; a user or integration with full edit FLS still cannot write an unlisted value. The correct fix is either (a) add the missing value to the `Order_Status__c` picklist's allowed list (if it is a valid business status), or (b) map/translate the source system value to the corresponding Salesforce-allowed value in the integration layer before the write. **Trap:** Granting the integration user System Administrator profile or the "Modify All Data" permission — these permissions do not bypass restricted picklist enforcement. Or debugging as a CRUD issue (the integration user can Create/Edit the object; the problem is the value, not the access). **Verify:** Use `describe_object('Sales_Order__c')` or the Tooling API to inspect `Order_Status__c` and confirm `restrictedPicklist: true` and the `picklistValues` list. 8. **Competent:** The Quick Action layout cache has not been invalidated. Deploying additional fields to an existing Quick Action's `quickActionLayoutItems` via SFDX updates the stored metadata, but Salesforce's org-level runtime cache for Quick Actions — which drives the Lightning component rendering — **does not flush on a field-list-only change**. The new fields are silently absent with no error. The fix is to **edit a non-field-list metadata property** on the same Quick Action (e.g. change `<description>` by adding a space or updating `<label>`) and redeploy; Salesforce treats that as a structural change to the action and flushes the cache, causing the full `quickActionLayoutItems` to be re-read. **Trap:** Assuming the deploy failed silently and re-running it identically — a clean re-deploy of unchanged non-field metadata does not bust the cache. Also wrong: removing and re-adding the Quick Action component from the Lightning App Builder page (this controls which action appears on the page, but the cached action layout still serves the stale field list). **Verify:** After the cache-busting redeploy, open a Lightning Opportunity record in a sandbox, invoke the Quick Action, and confirm all three new fields are visible. Also confirm via Setup → Quick Actions → [action name] → Fields that the layout matches the deployed XML.

9. **Competent:** The native Salesforce mechanism for trending point-in-time data is a **Reporting Snapshot**. A Reporting Snapshot runs a summary report on a schedule (e.g. weekly) and writes the aggregated results as rows into a custom object, accumulating a historical record over time. To track total open Opportunity amount week-over-week, you configure a summary report of open Opportunities (grouped appropriately), map its columns to fields on a custom snapshot target object, and schedule it to run weekly. The limiting factor is that **a Reporting Snapshot captures data at the moment it runs** — it cannot backfill historical data before the snapshot was configured. Six months of historical data requires that the snapshot was already running for six months; if it was not, those weeks are unrecoverable from native snapshots alone. **Trap:** Using report subscriptions or scheduled reports — these deliver the current-state report via email on a schedule but do not write data to an object and therefore cannot be queried to trend over time. Or suggesting Einstein Analytics / CRM Analytics as the "simple" fix when a Reporting Snapshot is the native no-code mechanism. **Verify:** After configuring the snapshot, confirm the target object accumulates a new row set each week by checking the custom object's record count before and after the scheduled run. Validate the rollup field matches the report total.

10. **Competent:** **Do not recommend activating Einstein Opportunity Scoring now.** Einstein Opportunity Scoring is a **predictive AI feature that requires a substantial volume of historical closed/converted records to train its model**. The minimum threshold Salesforce publishes is roughly 200 closed Opportunities with a mix of won and lost outcomes; 22 closed deals in eight months is far below this. Activating the feature with insufficient history will produce unreliable or random scores that could mislead the team. The correct recommendation is to continue building the Opportunity record base, revisit scoring eligibility after reaching the minimum history threshold, and in the interim use simpler pipeline hygiene tools (stage validation rules, required fields, standard reports) to prioritize deals. **Trap:** Activating Einstein Scoring anyway and telling leadership the scores will "improve over time" — this is true in principle, but early-stage scores with no training signal are noise, not signal, and can undermine trust in both the feature and the CRM. Also wrong: recommending Sales Engagement cadences as a substitute for scoring — cadences are a workflow tool, not a predictive scoring mechanism. **Verify:** Check the Einstein Lead/Opportunity Scoring setup page in the org; Salesforce displays a data readiness indicator and will flag if the minimum history threshold has not been met before the model can train.

11. **Competent:** The correct declarative mechanism is a **criteria-based Sharing Rule** on Account. Set the rule to share Accounts where `Industry = Technology` with the "Partner Success" public group (or role), granting Read Only access. Criteria-based sharing rules are designed exactly for this pattern: granting lateral access to a set of records defined by field values, independent of ownership and role hierarchy. The common mistake to avoid is setting the **Account OWD to Public Read-Only** thinking that is the simplest path — OWD applies universally to all users and all records, which would expose every Account (not just Technology industry ones) to every internal user, violating the principle of most-restrictive baseline. **Trap:** Using Manual Sharing to open individual Accounts one by one — this is operationally unsustainable for a large record set and does not auto-apply to future Accounts that match the Industry criteria. Or using a permission set to grant object-level Read access and assuming that alone gives record access (object CRUD grants the ability to see the object, but with Private OWD, record access still requires a sharing mechanism). **Verify:** After saving and activating the criteria-based Sharing Rule, log in as a Partner Success team member and confirm they can view a Technology-industry Account owned by a different rep but cannot view a non-Technology Account.

12. **Competent:** Once an External Client App (ECA) is created in the Salesforce UI, API access must be authorized by navigating to the **ECA detail page → Policies → Edit → App Policies → Select Permission Sets** and selecting the permission set(s) whose members should be able to use the app. This is the working authorization path for ECAs. The wrong approach that developers commonly attempt first is editing the "Assigned Connected Apps" section on a permission set's detail page — this page applies to classic Connected Apps, not External Client Apps, and has no effect on ECA authorization. Clicking through that UI path gives no error but also grants no access; the ECA remains unauthorized for all users. **Trap:** Assuming the ECA is authorized because no error appeared when editing the permission set's "Assigned Connected Apps" section — the absence of an error is misleading; the configuration is simply inert for ECAs. Or trying to deploy authorization via a `connectedApp-meta.xml` change — the ECA authorization policy is not exposed via metadata API and must be set in the UI. **Verify:** After setting App Policies on the ECA, confirm via SOQL: `SELECT Id, SetupEntityId, SetupEntityType FROM SetupEntityAccess WHERE SetupEntityType = 'ConnectedApplication'` joined against `PermissionSetAssignment` for the affected user, or attempt an OAuth flow as a user in the authorized permission set and confirm a token is issued without an "app not authorized" error.
