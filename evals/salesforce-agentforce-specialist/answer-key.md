# Answer key — salesforce-agentforce-specialist (grading rubric)

PASS = competent move identified AND trap avoided. Partial = right instinct, misses the key rule or the trap.

---

1. **Competent:** The template was deployed but not activated in the target org. A change-set deploy of a Prompt Builder template lands it in Draft status in the destination org; activation does not travel with the deployment. Open Prompt Builder in the target org, locate the template, and activate it before the agent action can invoke it. **Trap:** Assuming the deployment error would surface, or re-deploying to force activation — redeploy does nothing for activation status. **Verify:** Open Prompt Builder in the target org, confirm the template status reads "Active" (not "Draft"), then re-run the agent action.

2. **Competent:** The static template body text (the paragraph of example diagnostic language) is the problem. Static text in a Prompt Builder template is never processed by the Trust Layer's masking engine — it is sent to the LLM verbatim, exactly as written. Only dynamic merge fields resolved at run time pass through masking rules. The fix is to remove all sensitive medical language from the static template body; if context framing is needed, use neutral, non-PII language in the static portion and keep all patient-specific data in dynamic merge fields. **Trap:** Believing the Trust Layer masks everything in the prompt, including static text — it does not. **Verify:** Review the template body for any hard-coded sensitive terms; run the Trust Layer audit trail on a test invocation and confirm no sensitive static content appears in the outbound prompt log.

3. **Competent:** The architectural problem is a single over-broad topic ("General Support — Handle all customer inquiries") that gives the planner no reliable boundary to classify against or reason about when to stop. Without explicit "when NOT to use this topic" instructions and a scoped description, the planner either misroutes to this topic or, when it does route correctly, the model interprets "handle all inquiries" as license to generate an answer for anything. The fix is to split the topic into narrow, scoped topics with clear descriptions and explicit instructions that state what the topic covers and what it does NOT cover, plus configure a graceful escalation or "I can't help with that" response path for out-of-scope utterances. **Trap:** Thinking the model's own alignment will prevent fabrication when topics are vague — the planner and the topic instructions are the guardrail, not the model's default behavior. **Verify:** Author adversarial test utterances outside the intended scope in the Agentforce Testing Center, confirm the agent declines or escalates rather than fabricates, and check topic-routing accuracy scores.

4. **Competent:** The target field `Product_Blurb__c` is almost certainly defined as 255 characters in the org, not 300. Field Generation output silently truncates to the target field's maximum length at write time — no error is raised. The preview pane generates output bounded only by the model's response, not the field definition, so 300 characters pass preview but truncate on actual record save. The developer must `describe` the object to read the actual `length` property of `Product_Blurb__c`, then update the prompt's max-output instruction to match (e.g., "Write a product description of no more than 255 characters"), and then re-activate the template. **Trap:** Trusting the preview pane as definitive — the preview does not simulate field-write truncation. **Verify:** `describe` the object, read `length` on `Product_Blurb__c`, confirm the prompt's output constraint is at or below that value, activate, and save a real record to confirm the blurb is not cut off.

5. **Competent:** Without Data Cloud enabled, the Data Library, vector search, and RAG retriever features do not exist in the org — they cannot be configured regardless of effort. Knowledge articles (if Knowledge is enabled) are the available unstructured grounding mechanism, but they require content to be authored or imported into the Knowledge base, not pulled from SharePoint directly. Viable options for two weeks: (a) scope Knowledge article import/authoring as the grounding source if Knowledge is enabled and the team can migrate the content; (b) use merge fields from structured CRM data for a reduced feature set; (c) scope Data Cloud provisioning as a formal prerequisite with its own budget and timeline. It is premature to commit to full RAG/PDF grounding on a two-week timeline when Data Cloud is not enabled and no provisioning request is in flight. **Trap:** Beginning Data Library or vector search configuration in Setup, or promising the full PDF-grounded experience within two weeks. **Verify:** Check the object list for Data Cloud DMO/DLO objects; if absent, document Data Cloud as an unmet prerequisite and reset delivery expectations before committing a date.

6. **Competent:** A trustworthiness score of 0.91 is a strong quality signal on the 15 authored utterances, but Testing Center scores are graded quality indicators to iterate on — not a binary production gate. The gap the reviewer sees: topic-routing accuracy and action-execution accuracy have not been validated across adversarial or edge-case utterances, only the 15 happy-path ones the QA engineer wrote. The additional step is to expand the test set with negative/out-of-scope utterances, boundary conditions, and utterances that could plausibly trigger the wrong topic, and to re-evaluate. A deployment decision should be based on a representative test set, not just the authored positives. **Trap:** Treating a high trustworthiness score on a small curated set as equivalent to production readiness — the score only covers the cases tested. **Verify:** Add adversarial and out-of-scope test utterances to the Testing Center evaluation, re-run, and confirm topic and action accuracy across the broader set before recommending a production release.

7. **Competent:** The shared agent user's permission set grants access to both the customer-facing objects and the internal HR policy content (Knowledge articles or custom object records). Because the agent runs with exactly the permissions of its assigned user, the Service Agent executing as that user can read and surface HR content that the user is authorized to see. The fix is to give each agent its own dedicated least-privilege agent user with a permission set scoped to only the objects and records that agent's job requires — the Service Agent's user should have no access to internal HR content. **Trap:** Thinking the agent's topic configuration alone scopes what data can surface — data access is governed by the agent user's profile and permission sets, not by topic instructions. **Verify:** Describe and query the objects accessible to the shared agent user; create separate agent users, scope each to minimum required FLS and sharing, then re-run the Service Agent and confirm it can no longer retrieve HR content.

8. **Competent:** The "[Field Unavailable]" values indicate that some merge fields cannot be resolved for certain records — either because the lookup chain is null for that record (the related object or record doesn't exist), or because the agent/template user lacks FLS to read those fields on those records. The review checklist item is: verify that every merge field in the chain has (a) a non-null lookup path on the records being tested, and (b) FLS granted to the template's running user for every field in every related object traversed. Additionally, verify the Flow data provider's SOQL runs within governor limits and returns results for the template user's access level. Do not approve a template with unresolved merge fields — they indicate either a data gap or a permissions gap that will produce unreliable output in production. **Trap:** Approving the template because "[Field Unavailable]" looks like a minor display issue rather than a signal of a null relationship or FLS gap. **Verify:** `describe` each related object for FLS on the template user's permission set; identify which records produce nulls and confirm whether the lookup is populated; adjust the template's merge-field design or data model accordingly.

9. **Competent:** A Flow action is the correct choice. The decision rule: prefer Flow over Apex when both can accomplish the task; use Apex only when Flow cannot express the required logic or the transaction requires something declarative tools can't do. A single SOQL lookup and summary is well within Flow's capability (Get Records element). Building it as a Flow action is declarative, testable without a deployment, and easier to govern and audit. Apex is not warranted here. **Trap:** Defaulting to Apex for agent actions because it feels more powerful or flexible — this bypasses the least-complexity rule and introduces code that requires test coverage and deployment overhead when a Flow would suffice. **Verify:** Build the Flow action, dry-run the Get Records query in Flow Builder against the intended agent user's visibility, confirm records are returned, then wire the action to the agent topic.

10. **Competent:** The Flow-backed action's underlying Flow was not deployed — it was omitted from the change set. When the agent action executes, it tries to invoke a Flow that either does not exist in production or is a stale prior version, causing immediate failure. This is a frequent and critical cutover miss: the agent Bot, topics, and prompt templates deploy as metadata, but the backing Flows (and any referenced Apex) must also be deployed separately, and the agent user plus permission sets must exist in the target org. Before the agent is production-ready, deploy the missing Flow, confirm the agent user exists with the correct permission set assignments, and run a post-deployment smoke test. **Trap:** Assuming the agent metadata deployment is sufficient and the Flow will resolve automatically, or that the Flow already exists in production from a prior release without verifying it. **Verify:** Query Flows in the target org by API name to confirm the correct version is active; then trigger the action and confirm it executes without error.

11. **Competent:** The third-party model's context window, while larger than the default Einstein model, has different token limits and behavior around how it handles or truncates content near those limits. Grounded prompts that include many related child records may be hitting the model's effective input limit; the model silently drops the tail of the grounded content — the same behavior as exceeding a merge-list retrieval limit. The key verification is to check the token count of a grounded prompt for a representative high-volume record against the configured model's documented context window, and to test whether the number of retrieved child records is being bounded by the retrieval/merge limit rather than the query itself. Additionally, confirm that no prompt instruction was written assuming the default Einstein context window size. **Trap:** Assuming the larger context window means the problem is gone and looking elsewhere, or attributing partial output to a bug in the template rather than a model-side limit being hit. **Verify:** Review the model's documented token limits in Model Builder; count tokens in a representative grounded prompt for a high-volume record; reduce the related-record merge set to a bounded list and re-test to confirm full output is returned.

12. **Competent:** SMS has strict character limits (typically 160 characters per segment for standard SMS, though platform behavior varies) that are far below the 400-character average of the web widget templates. Copying web widget response templates to SMS without modification will produce multi-segment or truncated messages, degrading the user experience or exceeding channel limits. The correct design approach is to maintain per-channel response template variants: shorter, more direct templates for SMS (and any other constrained channel) and fuller templates for the web widget. Size each variant to the most constrained format of its target channel. **Trap:** Assuming that because the underlying agent logic is the same, the same response templates work across all channels. **Verify:** Review the SMS channel's character and media limits in the channel configuration; audit all response templates for the SMS deployment and confirm each fits within a single SMS segment or an explicitly tested multi-segment budget; test on an actual SMS handset before go-live.
