# Answer key ‚Äî nodejs (grading rubric)

PASS = competent move identified AND trap avoided. Partial = right instinct, misses the rule/trap.

1. **Competent:** The check that the field is "present and non-empty" only validates existence ‚Äî it never validates *type*. MongoDB `$where` and operator injection attacks pass objects (e.g. `{ "$gt": "" }`), not strings. The fix is to assert the value is a primitive string before use: `if (typeof req.body.userId !== 'string') return res.status(400).send()`. Use Fastify's schema validation or a library like Zod to enforce `type: 'string'` at the request boundary. **Trap:** assuming that "non-empty" string presence check defeats injection, or thinking the problem is SQL injection (it's NoSQL operator injection). **Verify:** send `{ "userId": { "$gt": "" } }` ‚Äî a fixed endpoint returns 400; a broken one executes the operator.

2. **Competent:** `fs.readFileSync` is a blocking syscall that halts the Node.js event loop for every concurrent client for its entire duration ‚Äî even if it's fast at 50 req/s, it becomes catastrophic under higher concurrency or on a slow filesystem (NFS, network mount). Replace with `fs.promises.readFile` (or `fs.readFile` with a callback) and `await` the result, keeping the loop free during the I/O wait. **Trap:** arguing "it works in load tests, so it's fine" ‚Äî that ignores the single-threaded event loop model where one slow sync call blocks all peers. **Verify:** run a concurrent load test while injecting disk latency (e.g. `tc netem` or a slow NFS mount); async version maintains throughput, sync version serializes all requests.

3. **Competent:** Every HTTP request calls `.on('data', handler)` on the shared EventEmitter, registering a *new* listener that closes over that request's `req.id`. These listeners accumulate because they are never removed after the request completes ‚Äî this is a classic EventEmitter listener leak. Fix: use `.once('data', handler)` if only one event is needed, or store the handler reference and call `.off('data', handler)` (or `removeListener`) when the request is done. **Trap:** increasing `setMaxListeners()` to silence the warning, which papers over the leak rather than fixing it. **Verify:** log `emitter.listenerCount('data')` before and after requests ‚Äî a fixed implementation stays bounded; the broken one grows linearly with request count.

4. **Competent:** `===` for secret comparison is vulnerable to a timing attack ‚Äî JavaScript string comparison short-circuits at the first differing byte, leaking information about how many leading characters match via response time differences. Replace with `crypto.timingSafeEqual(Buffer.from(incoming), Buffer.from(expected))` which always compares every byte in constant time. **Trap:** adding length checks (`if (a.length !== b.length)`) thinking that alone defeats timing attacks ‚Äî it doesn't when lengths are equal, and even the length comparison itself can leak data. **Verify:** use a statistical timing tool (e.g. `bleichenbacher`-style measurement) against both endpoints; verify both take constant time regardless of how many characters match.

5. **Competent:** No ‚Äî `process.nextTick()` callbacks drain the *nextTick queue* between every event loop phase, **before** I/O callbacks. Stacking `nextTick` in a loop effectively starves I/O: network reads, disk writes, and timer callbacks are blocked until the entire nextTick chain finishes ‚Äî the opposite of the intended effect. For CPU-bound work in a Lambda, either restructure as true async I/O where possible, use `setImmediate()` to yield after each chunk (it defers to the *check* phase, after I/O), or better yet, decompose the work into smaller Lambda invocations or use `worker_threads`. **Trap:** believing `nextTick` yields control to I/O between calls. **Verify:** instrument with `node:trace_events` and observe that I/O callbacks don't fire until the nextTick chain is exhausted.

6. **Competent:** Without an `exports` field, Node.js ESM consumers hit CJS-only resolution through the `main` field; on older Node versions this fails outright, and even on newer ones CJS packages imported as ESM surface as a single default export with no named exports. The library must add an `exports` map in `package.json` with `"import"` and `"require"` conditions pointing to ESM (`.mjs`) and CJS (`.cjs`) build artifacts respectively, and ship both build outputs. **Trap:** assuming adding `"type":"module"` to the library's own `package.json` fixes it ‚Äî that breaks existing CJS consumers who use `require()`. **Verify:** create a test consumer with `"type":"module"` that `import`s the package and a CJS consumer that `require()`s it; both must resolve and expose named exports correctly.

7. **Competent:** `npm install` resolves dependencies at install time and *may* update `package-lock.json` if the lock is out of sync with the manifest ‚Äî this means different CI runs or different developers can get different dependency trees, making builds non-deterministic. Replace `npm install` with `npm ci`, which reads `package-lock.json` exactly, errors if the lock diverges from `package.json`, and never updates the lock. **Trap:** thinking the risk is only theoretical since "it's worked so far" ‚Äî transitive dependency minor/patch bumps are common and can introduce breaking changes or vulnerabilities silently. **Verify:** deliberately skew `package-lock.json` by one patch version on a dep and run both commands ‚Äî `npm ci` errors; `npm install` silently resolves and proceeds.

8. **Competent:** An `async` function inside the Promise constructor creates a hidden inner Promise whose rejections are *not* connected to the outer `reject` callback ‚Äî any `throw` or rejected `await` inside the async executor silently becomes an unhandled rejection that bypasses the outer `.catch()`. The correct pattern is to avoid the Promise constructor entirely and use `util.promisify` at the callback boundary, or ‚Äî if a Promise constructor is truly necessary ‚Äî use a plain (non-async) executor and call `resolve`/`reject` explicitly from the callback. **Trap:** believing the `try/catch` inside the async executor passes errors to `reject` ‚Äî it does not, because the async function returns its own Promise that the constructor ignores after the synchronous portion. **Verify:** have the async executor throw after an `await`; observe that the outer `.catch()` never fires and Node raises an unhandled rejection warning instead.

9. **Competent:** `.pipe()` does not implement full backpressure management ‚Äî when `writable.write()` returns `false` (client download buffer full), the Readable stream is not automatically paused, so disk reads continue flooding the internal stream buffers, growing unboundedly and exhausting heap. Replace `.pipe()` with `stream.pipeline(fs.createReadStream(path), res)` from `node:stream/promises`: `pipeline()` monitors the writable's `drain` event and pauses the readable automatically when the consumer is slow. **Trap:** trying to fix the leak by increasing `highWaterMark` ‚Äî that raises the ceiling but doesn't stop the unbounded growth; proper backpressure handling is the only real fix. **Verify:** throttle the HTTP client to 1 KB/s and stream a multi-GB file; measure RSS over time ‚Äî `pipeline()` stays flat, `.pipe()` climbs.

10. **Competent:** Two distinct concerns: (1) **DoS via giant allocation** ‚Äî `Number(req.query.size)` can be `Infinity`, `NaN`, or an astronomically large integer supplied by the attacker, causing a multi-gigabyte allocation or a V8 crash; range-check strictly (`if (size < 1 || size > MAX || !Number.isInteger(size)) reject`) before any allocation. (2) **Information disclosure** ‚Äî `Buffer.allocUnsafe` does not zero-fill; it contains whatever was previously in that memory region. If the Buffer is not *immediately and completely* overwritten before any bytes are used or returned, stale heap contents (which could include secrets, tokens, or prior request data) may leak. **Trap:** fixing only the size issue and leaving `allocUnsafe`, or replacing with `Buffer.alloc` (safe default) while leaving the unvalidated `Number(...)` call. **Verify:** (1) send `?size=99999999999` and observe allocation failure/rejection; (2) inspect Buffer bytes before overwrite in a test ‚Äî `allocUnsafe` will contain non-zero garbage.

11. **Competent:** `Array.prototype.forEach` ignores the Promise returned by each async callback ‚Äî it fires all the async functions and returns immediately without waiting for any to settle. "All done" is logged before any `db.save` has completed, and any rejections become unhandled. The correct pattern is `await Promise.all(items.map(async (item) => db.save(item)))` (parallel) or a `for...of` loop with `await` inside (sequential). **Trap:** using `items.forEach(async ...)` and adding a `try/catch` around the `forEach` call ‚Äî the catch block never sees the inner rejections. **Verify:** make `db.save` reject on the second item; with `forEach` the rejection is unhandled and "all done" still logs; with `Promise.all` the `await` throws and the error is caught.

12. **Competent:** After an `uncaughtException` the process is in an *unknown, potentially corrupt state* ‚Äî heap may be inconsistent, open file descriptors or sockets may be in a partial state, database transactions may be half-committed. Continuing normally risks data corruption and undefined behavior. The handler should log the error for observability (the one valid use), then **call `process.exit(1)`** so a process supervisor (systemd, PM2, Kubernetes) can restart from a clean state. **Trap:** arguing "it keeps the service available" ‚Äî uptime achieved by running a corrupted process is worse than a clean restart with a brief outage. **Verify:** induce a `null` dereference in a request handler; with `exit()` in the handler, the supervisor restarts and the next request gets a clean process; without it, subsequent requests may behave unpredictably or silently corrupt state.
