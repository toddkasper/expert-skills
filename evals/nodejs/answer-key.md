1. **Competent:** The check that the field is "present and non-empty" only validates existence — it never validates *type*. MongoDB `$where` and operator injection attacks pass objects (e.g. `{ "$gt": "" }`), not strings. The fix is to assert the value is a primitive string before use: `if (typeof req.body.userId !== 'string') return res.status(400).send()`. Use Fastify's schema validation or a library like Zod to enforce `type: 'string'` at the request boundary. **Trap:** assuming that "non-empty" string presence check defeats injection, or thinking the problem is SQL injection (it's NoSQL operator injection). **Verify:** send `{ "userId": { "$gt": "" } }` — a fixed endpoint returns 400; a broken one executes the operator.

2. **Competent:** `fs.readFileSync` is a blocking syscall that halts the Node.js event loop for every concurrent client for its entire duration — even if it's fast at 50 req/s, it becomes catastrophic under higher concurrency or on a slow filesystem (NFS, network mount). Replace with `fs.promises.readFile` (or `fs.readFile` with a callback) and `await` the result, keeping the loop free during the I/O wait. **Trap:** arguing "it works in load tests, so it's fine" — that ignores the single-threaded event loop model where one slow sync call blocks all peers. **Verify:** run a concurrent load test while injecting disk latency (e.g. `tc netem` or a slow NFS mount); async version maintains throughput, sync version serializes all requests.

3. **Competent:** Every HTTP request calls `.on('data', handler)` on the shared EventEmitter, registering a *new* listener that closes over that request's `req.id`. These listeners accumulate because they are never removed after the request completes — this is a classic EventEmitter listener leak. Fix: use `.once('data', handler)` if only one event is needed, or store the handler reference and call `.off('data', handler)` (or `removeListener`) when the request is done. **Trap:** increasing `setMaxListeners()` to silence the warning, which papers over the leak rather than fixing it. **Verify:** log `emitter.listenerCount('data')` before and after requests — a fixed implementation stays bounded; the broken one grows linearly with request count.

4. **Competent:** `===` for secret comparison is vulnerable to a timing attack — JavaScript string comparison short-circuits at the first differing byte, leaking information about how many leading characters match via response time differences. Replace with `crypto.timingSafeEqual(Buffer.from(incoming), Buffer.from(expected))` which always compares every byte in constant time. **Trap:** adding length checks (`if (a.length !== b.length)`) thinking that alone defeats timing attacks — it doesn't when lengths are equal, and even the length comparison itself can leak data. **Verify:** use a statistical timing tool (e.g. `bleichenbacher`-style measurement) against both endpoints; verify both take constant time regardless of how many characters match.

5. **Competent:** No — `process.nextTick()` callbacks drain the *nextTick queue* between every event loop phase, **before** I/O callbacks. Stacking `nextTick` in a loop effectively starves I/O: network reads, disk writes, and timer callbacks are blocked until the entire nextTick chain finishes — the opposite of the intended effect. For CPU-bound work in a Lambda, either restructure as true async I/O where possible, use `setImmediate()` to yield after each chunk (it defers to the *check* phase, after I/O), or better yet, decompose the work into smaller Lambda invocations or use `worker_threads`. **Trap:** believing `nextTick` yields control to I/O between calls. **Verify:** instrument with `node:trace_events` and observe that I/O callbacks don't fire until the nextTick chain is exhausted.

6. **Competent:** Without an `exports` field, Node.js ESM consumers hit CJS-only resolution through the `main` field; on older Node versions this fails outright, and even on newer ones CJS packages imported as ESM surface as a single default export with no named exports. The library must add an `exports` map in `package.json` with `"import"` and `"require"` conditions pointing to ESM (`.mjs`) and CJS (`.cjs`) build artifacts respectively, and ship both build outputs. **Trap:** assuming adding `"type":"module"` to the library's own `package.json` fixes it — that breaks existing CJS consumers who use `require()`. **Verify:** create a test consumer with `"type":"module"` that `import`s the package and a CJS consumer that `require()`s it; both must resolve and expose named exports correctly.

7. **Competent:** `npm install` resolves dependencies at install time and *may* update `package-lock.json` if the lock is out of sync with the manifest — this means different CI runs or different developers can get different dependency trees, making builds non-deterministic. Replace `npm install` with `npm ci`, which reads `package-lock.json` exactly, errors if the lock diverges from `package.json`, and never updates the lock. **Trap:** thinking the risk is only theoretical since "it's worked so far" — transitive dependency minor/patch bumps are common and can introduce breaking changes or vulnerabilities silently. **Verify:** deliberately skew `package-lock.json` by one patch version on a dep and run both commands — `npm ci` errors; `npm install` silently resolves and proceeds.

8. **Competent:** An `async` function inside the Promise constructor creates a hidden inner Promise whose rejections are *not* connected to the outer `reject` callback — any `throw` or rejected `await` inside the async executor silently becomes an unhandled rejection that bypasses the outer `.catch()`. The correct pattern is to avoid the Promise constructor entirely and use `util.promisify` at the callback boundary, or — if a Promise constructor is truly necessary — use a plain (non-async) executor and call `resolve`/`reject` explicitly from the callback. **Trap:** believing the `try/catch` inside the async executor passes errors to `reject` — it does not, because the async function returns its own Promise that the constructor ignores after the synchronous portion. **Verify:** have the async executor throw after an `await`; observe that the outer `.catch()` never fires and Node raises an unhandled rejection warning instead.

9. **Competent:** `.pipe()` does not implement full backpressure management — when `writable.write()` returns `false` (client download buffer full), the Readable stream is not automatically paused, so disk reads continue flooding the internal stream buffers, growing unboundedly and exhausting heap. Replace `.pipe()` with `stream.pipeline(fs.createReadStream(path), res)` from `node:stream/promises`: `pipeline()` monitors the writable's `drain` event and pauses the readable automatically when the consumer is slow. **Trap:** trying to fix the leak by increasing `highWaterMark` — that raises the ceiling but doesn't stop the unbounded growth; proper backpressure handling is the only real fix. **Verify:** throttle the HTTP client to 1 KB/s and stream a multi-GB file; measure RSS over time — `pipeline()` stays flat, `.pipe()` climbs.

10. **Competent:** Two distinct concerns: (1) **DoS via giant allocation** — `Number(req.query.size)` can be `Infinity`, `NaN`, or an astronomically large integer supplied by the attacker, causing a multi-gigabyte allocation or a V8 crash; range-check strictly (`if (size < 1 || size > MAX || !Number.isInteger(size)) reject`) before any allocation. (2) **Information disclosure** — `Buffer.allocUnsafe` does not zero-fill; it contains whatever was previously in that memory region. If the Buffer is not *immediately and completely* overwritten before any bytes are used or returned, stale heap contents (which could include secrets, tokens, or prior request data) may leak. **Trap:** fixing only the size issue and leaving `allocUnsafe`, or replacing with `Buffer.alloc` (safe default) while leaving the unvalidated `Number(...)` call. **Verify:** (1) send `?size=99999999999` and observe allocation failure/rejection; (2) inspect Buffer bytes before overwrite in a test — `allocUnsafe` will contain non-zero garbage.

11. **Competent:** `Array.prototype.forEach` ignores the Promise returned by each async callback — it fires all the async functions and returns immediately without waiting for any to settle. "All done" is logged before any `db.save` has completed, and any rejections become unhandled. The correct pattern is `await Promise.all(items.map(async (item) => db.save(item)))` (parallel) or a `for...of` loop with `await` inside (sequential). **Trap:** using `items.forEach(async ...)` and adding a `try/catch` around the `forEach` call — the catch block never sees the inner rejections. **Verify:** make `db.save` reject on the second item; with `forEach` the rejection is unhandled and "all done" still logs; with `Promise.all` the `await` throws and the error is caught.

12. **Competent:** After an `uncaughtException` the process is in an *unknown, potentially corrupt state* — heap may be inconsistent, open file descriptors or sockets may be in a partial state, database transactions may be half-committed. Continuing normally risks data corruption and undefined behavior. The handler should log the error for observability (the one valid use), then **call `process.exit(1)`** so a process supervisor (systemd, PM2, Kubernetes) can restart from a clean state. **Trap:** arguing "it keeps the service available" — uptime achieved by running a corrupted process is worse than a clean restart with a brief outage. **Verify:** induce a `null` dereference in a request handler; with `exit()` in the handler, the supervisor restarts and the next request gets a clean process; without it, subsequent requests may behave unpredictably or silently corrupt state.

13. **Competent:** `require(esm)` was added experimentally in Node 20.17.0 (behind `--experimental-require-module`) and was unflagged in Node 20.19.0+ and Node 22.12.0+ — it is not available in Node 20.15.0 at all. The Node 22.12.0 developer sees it work because their version is past the unflagged threshold. The Node 20.15.0 developer has two options: (a) upgrade to Node 20.19.0 or later, after which `require('esm-pkg')` works without flags; or (b) switch to dynamic `import()` — `const mod = await import('esm-pkg')` — which works on any Node version that supports ESM (12+). Dynamic `import()` is the safest option when the Node version cannot be controlled. **Trap:** assuming `require(esm)` is universally stable in Node 22 — it is only unflagged in 22.12.0+, not all 22.x releases; and it was not marked fully stable (non-experimental) until Node 25.4.0. **Verify:** run `node -e "require('esm-pkg')"` on Node 20.15.0 (throws ERR_REQUIRE_ESM), Node 20.19.0 (succeeds), and Node 22.12.0 (succeeds); confirm `process.features.require_module` is `true` on supported versions.

14. **Competent:** The default `highWaterMark` for byte-mode streams is 64 KiB (65 536 bytes) — not 16 KiB; that was the old default before nodejs/node#53494. The 65 536-byte buffer the profiler reports is the *correct* default ceiling, not a bug. Lowering to 16 384 bytes would make backpressure trigger more frequently, which forces the producer to pause earlier and reduces peak buffering per stream — at the cost of more context switches between read/drain cycles and lower throughput on fast producers. Before tuning, confirm the stalls are caused by a slow consumer, not by missing `await pipeline(...)` (which handles drain automatically). Lowering `highWaterMark` is only warranted if memory pressure is the measured bottleneck and throughput can be sacrificed. **Trap:** treating 65 536 bytes as an anomaly to fix — it is the documented default. **Verify:** benchmark with `autocannon` or a synthetic pipeline at both `highWaterMark` values; measure RSS and ops/sec to confirm the trade-off before shipping any change.

15. **Competent:** Kubernetes sends SIGTERM and then waits up to `terminationGracePeriodSeconds` (default 30 s) before sending SIGKILL. A process with no SIGTERM handler exits immediately (default OS behavior), dropping any in-flight requests. The fix is to stop accepting new connections and drain the server before exiting: `process.on('SIGTERM', () => { server.close(() => process.exit(0)); })`. `server.close()` prevents new connections while letting existing ones finish; calling `process.exit(0)` in the callback ensures the process exits cleanly once all requests are drained, within the grace period. **Trap:** calling `server.close()` without the `process.exit()` callback — `close()` does not end the process; the pod will hang until SIGKILL after the grace period, achieving nothing. **Verify:** deploy behind a load balancer, trigger a rolling restart, and check access logs for 5xx or connection-reset errors — a correct shutdown produces zero dropped requests during the grace period.

16. **Competent:** Two separate bugs: (1) **Type bug** — `process.env.TIMEOUT` is always a string; `setTimeout` expects a number. Passing a string makes `setTimeout(fn, "5000")` — JavaScript coerces it to a number via `ToNumber`, which *does* work in most engines, but if the env var is absent, `process.env.TIMEOUT` is `undefined` and `Number(undefined)` is `NaN`; `setTimeout(fn, NaN)` fires immediately (treated as 0) rather than not at all, but a missing var explains the CI hang if the env var is not set there. (2) **Missing default / missing presence check** — in CI `TIMEOUT` is not set, making `process.env.TIMEOUT` `undefined`; `Number(undefined)` → `NaN` → timer fires immediately but the callback logic that depends on a real delay may not work correctly, or the hang is caused by an uncaught rejection triggered by the immediate firing. Fix: `const delay = Number(process.env.TIMEOUT ?? '5000'); if (!Number.isFinite(delay) || delay <= 0) throw new Error('TIMEOUT must be a positive number'); setTimeout(fn, delay)`. **Trap:** only casting to `Number` without validating the result — `Number('')` is `0`, `Number('abc')` is `NaN`, both accepted silently. **Verify:** run the CLI with `TIMEOUT` unset and observe immediate invocation; set `TIMEOUT=abc` and observe `NaN` coercion; fix and confirm both cases throw or use the default.
