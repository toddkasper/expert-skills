1. **Competent:** Recommend **Customer Community** (or Experience Cloud for Nonprofits if on NPSP). Donors are customers — authenticated individuals who need to see only their own records — not channel partners. Partner Community is designed for resellers/channel users who work Leads, Opportunities, and Campaigns and require a full role hierarchy; it would be over-licensed and unnecessarily expensive for a donor portal. The program director's reasoning conflates "capable" with "correct": Partner Community adds features the use case doesn't need and mismatches the user model (donors have a Contact + Account relationship, not a business-partner relationship). **Trap:** Recommending Partner Community because it appears more feature-rich, or recommending Customer Community Plus (which adds cost for reports/dashboards donors don't need). **Verify:** Confirm the license choice against the decision table — identify whether donors need peer visibility, reports, or deal-object access. If they do not, base Customer Community (or the NPSP-specific variant) is the correct and cheapest authenticated tier.

4. **Competent:** The guest user profile is **one shared profile across all unauthenticated visitors** — there is no per-visitor session scoping for record ownership or visibility. If the Screen Flow creates an Application record and the guest profile has Read access on Application, any guest who can access a record URL or whose flow session is not tightly scoped could potentially retrieve another applicant's data. The correct control is: (a) enforce `with sharing` in any Apex invoked by the flow so sharing rules are respected, and (b) scope all record retrieval by an **explicit unique key** (e.g. a UUID generated at submission time and passed back via URL parameter), never by session identity alone. Guest-profile object Read should be granted only to a locked-down subset of fields, with no PII exposed. **Trap:** Assuming the guest session provides per-visitor isolation — it does not. Granting the guest profile broader Read to "make the flow work" is the pattern that creates exposure. Relying on OWD to protect records without recognizing that guest users have no record ownership and OWD "Private" still allows Apex `without sharing` to bypass it. **Verify:** In a sandbox, submit the form twice with different data as two separate browser sessions (one private window, one regular). Attempt to access the first submission's record URL from the second session. Confirm the record is not readable without the scoped key.

5. **Competent:** The issue is that the SAML flow is configured as **SP-initiated** only, but the partners are launching from the IdP (IdP-initiated flow). For IdP-initiated SSO, Salesforce must have an **IdP-initiated SSO URL** configured, and the partner's IdP app launchpad must point to that URL rather than to the Experience Cloud login page. In Setup → Single Sign-On Settings → [SAML Config], confirm the "IdP-Initiated Login URL" is populated and shared with the IdP team. If SP-initiated SSO works (e.g. the partner can log in by visiting the portal login page and clicking SSO), the metadata is correct — the gap is specifically the IdP-initiated flow configuration. **Verify:** Copy the "IdP-Initiated Login URL" from the SAML SSO configuration in Setup. 6. **Competent:** The most likely missed step is that **external user profiles must be explicitly added as site members** in Digital Experiences Administration, and this site-membership assignment is **org-specific and does not travel in a change set**. 7. **Competent:** Check the **CSP (Content Security Policy) Trusted Sites** configuration in Experience Builder. Any external script origin (the chat vendor's domain) must be added as a CSP Trusted Site (also called a Trusted URL in newer orgs) before the browser will allow it to execute. The widget fails silently in production — no user-visible error — because the browser blocks the script at the network layer in response to a CSP violation, and Experience Cloud does not surface CSP errors in the UI. The developer should open the browser developer console, look for a `Content-Security-Policy` violation error naming the blocked domain, then add that origin in Experience Builder → Settings → Security → CSP Trusted Sites. **Trap:** Assuming the deployment itself is corrupt or the component has a code bug, and starting a code investigation. Or checking Locker Service isolation instead of CSP — Locker would show JavaScript errors, not silence. **Verify:** Open the production portal in a browser with the developer console open. Confirm a CSP violation error appears naming the chat vendor's domain. Add the domain to CSP Trusted Sites in Experience Builder, republish the site, and confirm the console is clean and the widget loads.

8. **Competent:** Build Your Own (LWR) is the better choice given the performance requirement. LWR sites publish UI as **statically rendered pages cached at the CDN edge**, which is architecturally faster than Aura's dynamic component-data fetching on every page load — making it the Salesforce-strategic choice for performance-sensitive, public-facing portals. The personalization requirement (content variants by investment tier) is addressable with LWR Audience targeting. The trade-off the architect must weigh is **LWR component availability**: if any required OOTB feature (e.g. Reputation, certain CMS components) is Aura-only, choosing LWR forces a custom build or requirement change. The architect should verify the full component and feature list against the LWR limitations matrix before committing. **Trap:** Accepting the Aura recommendation because "more OOTB components" sounds safer. Aura's dynamic rendering directly conflicts with the sub-two-second mobile load requirement. **Verify:** Document every required component and feature; check each against the official "Considerations for Experience Cloud Sites Built on LWR" article. If all required features are LWR-compatible, commit to LWR. If any are Aura-only, surface the conflict to stakeholders before the template decision is locked.

11. **Competent:** The most likely cause is that Salesforce's **CDN is serving cached static assets** for the site, and the published changes have not propagated to the CDN edge nodes yet — or the CDN cache has not been purged after the publish. Salesforce CDN edge-caches static assets (CSS, images, JS) for public and LWR sites to optimize performance, but this means that after publishing, stale cached versions can persist at edge nodes for minutes to hours. The admin should navigate to **Experience Builder → Publish → and look for a "Purge CDN Cache" option** (or in Setup → Digital Experiences → [site] → Settings, depending on org version), trigger a cache purge, then ask users to hard-refresh. **Trap:** Assuming the publish did not complete and republishing, which does not force a CDN purge. Or concluding users need to clear browser cache — if the CDN is serving stale assets, browser cache-clearing will simply download the same stale CDN asset. **Verify:** Use a CDN-debugging browser extension or `curl -I <asset URL>` to check the `Cache-Control` and `X-Cache` headers. If `X-Cache: HIT` is returned with an old `ETag`, the CDN is caching. After purging, repeat the check and confirm a cache miss, then confirm users see the updated branding.
